Knowledge Bases & Retriever Tools

Before an agent can decide what to retrieve, it needs something to retrieve from — and a way to reach it. Here we build two separate vector stores and turn each into a tool the LLM can call by name.

The Plan for This Section

An Agentic RAG system is only as good as the knowledge it can reach. In this sub-chapter we build the ingredients the agent will use later: two independent FAISS vector stores (one on the LangGraph docs, one on the LangChain docs), each exposed to the LLM as a retriever tool. By the end, the agent will be able to look at a question and pick the right store — not because we wrote routing logic, but because we wrote good tool descriptions.

Where this fits

This is steps 0–2 of the build. The tools list we produce here is the exact list we'll bind to the LLM and hand to the ToolNode in 13.2. Nothing here is agent-specific yet — it's ordinary ingestion, plus one new function: create_retriever_tool.

Step 0 — Setup

We use two model providers. Groq (serving Llama 3.3 70B) is the agent's brain — fast and free-tier friendly — and OpenAI supplies the embedding model that turns document chunks into vectors. Both keys live in a .env file and are loaded with python-dotenv:

import os
from dotenv import load_dotenv
load_dotenv()

os.environ["GROQ_API_KEY"] = os.getenv("GROQ_API_KEY")
os.environ["OPENAI_API_KEY"] = os.getenv("OPENAI_API_KEY")

Mixing providers like this is completely normal and a real strength of the LangChain stack: the embedding model and the chat model are independent choices. You could swap OpenAI embeddings for a local HuggingFace model, or Groq for OpenAI's gpt-4o, without touching anything else in the pipeline.

Step 1 — Build the First Knowledge Base (LangGraph docs)

The ingestion recipe is the same one from earlier chapters: load → split → embed → store. We scrape three LangGraph documentation pages with WebBaseLoader, passing a browser-like User-Agent header so the request isn't refused.

from langchain_community.document_loaders import WebBaseLoader
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
from langchain_text_splitters import RecursiveCharacterTextSplitter

# LangGraph docs
loader = WebBaseLoader(
    web_paths=[
        "https://docs.langchain.com/oss/python/langgraph/overview",
        "https://docs.langchain.com/oss/python/langgraph/workflows-agents",
        "https://docs.langchain.com/oss/python/langgraph/graph-api",
    ],
    requests_kwargs={"headers": {"User-Agent": "Mozilla/5.0"}},
)
docs = loader.load()

The result, docs, is a list of LangChain Document objects — one per URL — each with page_content (the scraped text) and metadata (the source URL, title, language). Those raw pages are far too long to embed as single vectors, so we split them into overlapping chunks:

# Split into ~1000-char chunks with a little overlap
text_splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=100)
doc_splits = text_splitter.split_documents(docs)

# Add all these chunks to a FAISS vector DB
vectorstore = FAISS.from_documents(documents=doc_splits, embedding=OpenAIEmbeddings())
retriever = vectorstore.as_retriever()

Three parameters do the real work here. They're worth understanding rather than copying:

ParameterValueWhy
chunk_size1000Characters per chunk. Big enough to hold a coherent idea, small enough to embed precisely and to fit several chunks in the prompt later.
chunk_overlap100Each chunk repeats the last 100 characters of the previous one, so a sentence split across a boundary still appears whole in at least one chunk.
embeddingOpenAIEmbeddings()The function that maps each chunk to a vector. The same model must embed the query at search time, or the vectors won't be comparable.

RecursiveCharacterTextSplitter is "recursive" because it tries to split on the largest natural boundary first (paragraphs), then sentences, then words — only falling back to a hard character cut when nothing else fits. That keeps chunks readable instead of slicing mid-word.

A quick sanity check

Calling .as_retriever() turns the store into a retriever — an object with one job: given a query string, return the most similar chunks. Always test it before going further:

retriever.invoke("what is langgraph")
# [Document(page_content='...LangGraph ecosystem... pair LangGraph with: LangSmith...'),
#  Document(page_content='...low-level orchestration framework and runtime...'),
#  ... ]

Good — the chunks coming back are clearly about LangGraph. The store works. Now we make it reachable by an LLM.

Turning a Retriever into a Tool

This is the move that makes RAG agentic. A plain retriever is something you call in code. Wrapping it with create_retriever_tool turns it into something the LLM can call — a first-class tool with a name and a description, exactly like the @tool functions from Chapter 12.

from langchain_core.tools.retriever import create_retriever_tool

retriever_tool = create_retriever_tool(
    retriever,
    "retriever_vector_db_blog",                     # the tool's name
    "Search and run information about Langgraph",    # the model reads this to decide WHEN to use it
)

The function takes exactly three arguments, and each one matters:

ArgumentRole
retrieverThe thing that actually runs when the tool is called. Its search results become the tool's output.
"retriever_vector_db_blog"The tool's name — how the model refers to it in a tool call, and how ToolNode looks it up later.
"Search and run information about Langgraph"The description — the only hint the model gets about when this tool is the right one.

What you get back is a StructuredTool — the same type LangChain produces for any tool — with the retriever wired in as its function:

retriever_tool
# StructuredTool(name='retriever_vector_db_blog',
#   description='Search and run information about Langgraph',
#   args_schema=<class 'langchain_core.tools.retriever.RetrieverInput'>,
#   func=<function create_retriever_tool.<locals>.func ...>)
The description IS the routing logic

The agent never sees your FAISS index, your chunk size, or your embeddings — it only sees each tool's name and description. "Search and run information about Langgraph" is the entire reason the model will pick this tool for a LangGraph question. With two stores, these one-line descriptions are the only thing the agent uses to route. A vague description ("search the docs") would leave the model guessing; a specific one routes reliably. Treat the description as a prompt.

Step 2 — A Second Knowledge Base (LangChain docs)

One retriever tool is useful; two is where the agent starts genuinely routing. We repeat the exact same recipe on a different source — the LangChain docs — to produce a completely separate FAISS store and a second tool. Note we reuse the same text_splitter object; it holds no state between calls.

# LangChain docs — a SEPARATE vector store
loader_langchain = WebBaseLoader(
    web_paths=[
        "https://docs.langchain.com/oss/python/langchain/quickstart",
        "https://docs.langchain.com/oss/python/langchain/knowledge-base",
        "https://docs.langchain.com/oss/python/langchain/agents",
    ],
    requests_kwargs={"headers": {"User-Agent": "Mozilla/5.0"}},
)
docs = loader_langchain.load()

doc_splits = text_splitter.split_documents(docs)
vectorstorelangchain = FAISS.from_documents(documents=doc_splits, embedding=OpenAIEmbeddings())
retrieverlangchain = vectorstorelangchain.as_retriever()

retriever_tool_langchain = create_retriever_tool(
    retrieverlangchain,
    "retriever_vector_langchain_blog",
    "Search and run information about Langchain",
)

# The agent will choose between these two
tools = [retriever_tool, retriever_tool_langchain]

That tools list is the deliverable of this whole sub-chapter. Two tools, two stores, two crisp descriptions:

Tool nameBacked byDescription the agent reads
retriever_vector_db_blogLangGraph docs store"Search and run information about Langgraph"
retriever_vector_langchain_blogLangChain docs store"Search and run information about Langchain"

Why two separate stores instead of one big one?

You could dump both doc sets into a single FAISS index and have one retriever. Keeping them separate buys you three things:

BenefitWhat it means
RoutingThe agent picks a source, not just chunks — it reasons about which body of knowledge fits the question.
IsolationA LangGraph question can't accidentally surface near-duplicate LangChain chunks that happen to embed similarly.
IndependenceYou can re-index, swap, or permission each store on its own — add a third tool tomorrow without touching the first two.

So when a question arrives, the agent chooses a tool purely from the descriptions — a LangGraph question fires retriever_vector_db_blog, a LangChain question fires retriever_vector_langchain_blog:

Key Takeaways

Remember These Three

1. Building a knowledge base is the familiar load → split → embed → store recipe. chunk_size/chunk_overlap control how text is cut; the embedding model must be the same at index time and query time.

2. create_retriever_tool(retriever, name, description) turns a retriever into a StructuredTool the LLM can call. It's what lets the model — not your code — decide to retrieve.

3. With several stores, the tool descriptions are the routing logic. The agent picks a knowledge base from those one-liners alone, so write them to clearly distinguish the sources.
