Supervisor Multi-Agent RAG

A central supervisor agent routes tasks to specialized workers — a research agent with RAG and Tavily, and a math agent with arithmetic tools — one at a time, never doing the work itself.

Overview

When a single agent has too many tools, routing gets unreliable. The supervisor pattern solves this with a divide-and-conquer approach: create expert agents for specific domains and let a supervisor LLM decide which one to invoke based on the current task.

The supervisor controls all communication flow and task delegation. It reads the user query, picks the right worker, waits for the result, and either assigns the next sub-task or returns the final answer. It never calls retrieval or math tools itself — it only routes.

The notes draw an analogy to a project manager: the PM doesn't write code, but delegates to Dev A, Dev B, or Lead A based on what's needed. In our build, the supervisor delegates to a research agent (RAG + web) and a math agent (add, multiply, divide).

Where this fits

Building block 2 of 3 in Chapter 16. Pattern 16.1 used peer handoffs between two agents. Here a third layer — the supervisor — sits above the workers and decides who acts next. Implemented with the prebuilt create_supervisor from the langgraph-supervisor package.

Why Not Just One Agent with All Tools?

Single agent (6+ tools)Supervisor + specialists
LLM must choose among retrieval, web search, add, multiply, divide in one prompt Research agent sees 2 tools; math agent sees 3 — smaller decision space
System prompt becomes a long list of conflicting instructions Each agent gets a tight, role-specific prompt
Hard to debug which tool was wrongly selected Supervisor's routing decision is a separate, inspectable step
Adding a new capability means re-tuning the whole agent Register a new worker agent; update the supervisor prompt

Building the Worker Agents

Research agent

The research agent gets Tavily web search and the internal retriever. Its prompt is strict: research only, no math, respond with results only: 

from langchain_tavily import TavilySearch
from langgraph.prebuilt import create_react_agent

web_search = TavilySearch(max_results=3)

research_agent = create_react_agent(
    model=llm,
    tools=[web_search, internal_tool],
    prompt=(
        "You are a research agent.\n\n"
        "INSTRUCTIONS:\n"
        "- Assist ONLY with research-related tasks, DO NOT do any math\n"
        "- After you're done with your tasks, respond to the supervisor directly\n"
        "- Respond ONLY with the results of your work, do NOT include ANY other text."
    ),
    name="research_agent",
)

Math agent

Three simple arithmetic tools — enough to demonstrate domain separation. In production you'd swap these for a calculator API or a code-execution sandbox: 

def add(a: float, b: float):
    """Add two numbers."""
    return a + b

def multiply(a: float, b: float):
    """Multiply two numbers."""
    return a * b

def divide(a: float, b: float):
    """Divide two numbers."""
    return a * b

math_agent = create_react_agent(
    model=llm,
    tools=[add, multiply, divide],
    prompt=(
        "You are a math agent.\n\n"
        "INSTRUCTIONS:\n"
        "- Assist ONLY with math-related tasks\n"
        "- After you're done with your tasks, respond to the supervisor directly\n"
        "- Respond ONLY with the results of your work, do NOT include ANY other text."
    ),
    name="math_agent",
)

Creating the Supervisor

The langgraph-supervisor package provides create_supervisor, which wires transfer tools (transfer_to_research_agent, transfer_to_math_agent) and handles the message-passing protocol automatically: 

from langgraph_supervisor import create_supervisor

supervisor = create_supervisor(
    model=llm,
    agents=[research_agent, math_agent],
    prompt=(
        "You are a supervisor managing two agents:\n"
        "- a research agent. Assign research-related tasks to this agent\n"
        "- a math agent. Assign math-related tasks to this agent\n"
        "Assign work to one agent at a time, do not call agents in parallel.\n"
        "Do not do any work yourself."
    ),
    add_handoff_back_messages=True,
    output_mode="full_history",
).compile()
ParameterValueWhat it does
agents [research_agent, math_agent] Compiled ReAct subgraphs the supervisor can delegate to
add_handoff_back_messages True Inserts a confirmation message when control returns from a worker
output_mode "full_history" Preserves every message from supervisor and workers in the final state

Under the hood, the supervisor LLM sees transfer_to_* tools — one per worker. When it calls transfer_to_research_agent, LangGraph invokes the research subgraph, collects the result, and returns control to the supervisor for the next routing decision.

Running a Multi-Step Query

response = supervisor.invoke({
    "messages": (
        "List all the transformer variants in production deployments from the retriever "
        "and then tell me what is 5 plus 10"
    )
})

Tracing the message history step by step:

StepWhoWhat happened
1 Supervisor Reads the query; calls transfer_to_research_agent
2 Research agent Calls InternalResearchNotes with query “transformer variants in production deployments”
3 Research agent Returns list: EfficientFormer, Reformer, ViT, Swin, Wav2Vec 2.0
4 Supervisor Calls transfer_to_math_agent for “5 plus 10”
5 Math agent Calls add(5, 10) → 15
6 Supervisor Synthesizes final answer combining research results and math result 
print(response["messages"][-1].content)

Supervisor vs Peer Network

Aspect16.1 Peer Network16.2 Supervisor
Routing Fixed pipeline: researcher → writer (with loop-back) Dynamic: supervisor picks any worker based on context
Agent count Best for 2–3 sequential stages Scales to N specialists with one coordinator
Skill mix Agents in the same pipeline (research → write) Unrelated skills (research + math + code + ...)
Implementation Manual Command(goto=...) edges create_supervisor handles transfer tools
Parallelism Sequential by design This build is sequential; supervisor prompt forbids parallel calls
Gotcha

The supervisor prompt says “Assign work to one agent at a time” for a reason. Parallel agent invocation requires a different graph topology (fan-out / fan-in nodes). Without it, the supervisor may try to call two workers simultaneously and only one will execute. Keep it sequential until you explicitly build parallel branches.

Key Takeaways

Remember These Three

1. The supervisor never executes tools — it only routes. All retrieval and math happen inside worker subgraphs with focused prompts.

2. create_supervisor auto-generates transfer tools (transfer_to_*) so you don't hand-wire delegation edges for every new agent.

3. Multi-step queries decompose naturally: the supervisor assigns sub-task 1 to research, waits, then assigns sub-task 2 to math, then synthesizes the final answer.
