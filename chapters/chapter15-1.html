Chain-of-Thought RAG

Before retrieving, reason. Chain-of-Thought RAG decomposes a complex question into intermediate steps, retrieves for each step, and only then synthesizes the final answer.

Overview

In every RAG pipeline you've built so far, retrieval happens once, against the raw user question. That works for simple lookups — “What is FAISS?” — but falls apart when the question is layered. “What additional experiments were run in the transformer evaluation, and why do they matter?” requires the system to understand what pieces the question contains before it knows what to search for.

Chain-of-Thought (CoT) in RAG fixes this by inserting a planning step before retrieval. The LLM first breaks the question into 2–3 reasoning sub-steps, then the retriever searches for each sub-step separately, pools the documents, and only then generates the final answer. Think of it as the model writing a scratchpad before opening the filing cabinet.

Where this fits

This is building block 1 of 5 in Chapter 15, from 3-COTRag.ipynb. The graph is a straight chain: planner → retriever → responder — the same spine as basic RAG, with a planner bolted onto the front. There are no loops yet; that comes in 15.2.

The CoT Loop

Step 1 — Reason: decompose the question into sub-steps.
Step 2 — Act: retrieve documents for each step.
Step 3 — Observe: combine the retrieved context.
Step 4 — Reflect: generate the final, reasoned answer.

Read the diagram top to bottom: the question never goes straight to the retriever. It passes through planner first, which produces a list of sub-steps stored in the graph state. The retriever then runs once per sub-step, and the responder reads all pooled documents to write one final answer to the original question.

What We're Building

By the end of this sub-chapter you'll have a working LangGraph with three nodes and three straight edges — no conditions, no loops. The knowledge base is research_notes.txt, an internal experiment log about transformer evaluation. We'll walk through the build in six steps: set up the vector store, initialise the LLM, define the state, write the three node functions, wire the graph, and run a test query.

Step 1 — Imports & the Vector Store

Start with the usual ingestion pipeline from earlier chapters: load → split → embed → store. The only new import here is StateGraph from LangGraph, which we'll use later to wire the nodes together.

Imports

import os
from typing import List
from pydantic import BaseModel
from langchain_classic.schema import Document
from langchain_openai import OpenAIEmbeddings
from langchain_community.document_loaders import TextLoader
from langchain_classic.vectorstores import FAISS
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from langgraph.graph import StateGraph, END

Build the retriever

TextLoader reads the raw file. RecursiveCharacterTextSplitter breaks it into 500-character chunks with 50 characters of overlap so sentences at chunk boundaries aren't lost. FAISS.from_documents embeds every chunk and stores the vectors. as_retriever() wraps the store in the standard search interface we'll call with retriever.invoke(query). 

docs = TextLoader("research_notes.txt", encoding="utf-8").load()
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)
embedding = OpenAIEmbeddings()
vectorstore = FAISS.from_documents(chunks, embedding)
retriever = vectorstore.as_retriever()

At this point you have a searchable index over the experiment log. Nothing agentic yet — just the same retriever you'd use in a plain RAG pipeline.

Step 2 — Initialise the LLM

The planner and responder nodes both need an LLM. We use init_chat_model with GPT-4o — the same pattern from Chapters 13–14. Load your API key from .env before calling the model. 

from langchain.chat_models import init_chat_model
from dotenv import load_dotenv

load_dotenv()
os.environ["OPENAI_API_KEY"] = os.getenv("OPENAI_API_KEY")
llm = init_chat_model("openai:gpt-4o")

Step 3 — The State

LangGraph passes a single state object from node to node. Each node reads what it needs, does its work, and returns an updated copy. We use a Pydantic BaseModel so fields are typed and validated, and model_copy(update={...}) gives us a clean way to update just the fields that changed. 

class RAGCoTState(BaseModel):
    question: str
    sub_steps: List[str] = []
    retrieved_docs: List[Document] = []
    answer: str = ""
FieldSet byPurpose
questionYou, at invoke timeThe original user question — never modified.
sub_stepsplan_stepsThe reasoning steps the planner extracted from the question.
retrieved_docsretrieve_per_stepAll documents fetched across every sub-step, pooled together.
answergenerate_answerThe final synthesized response.

Notice the flow: each field is written by exactly one node, in order. By the time generate_answer runs, sub_steps and retrieved_docs are already populated.

Step 4 — The Nodes

Three functions, three jobs. Each takes the current state and returns an updated state. We'll write them one at a time so you can see exactly what each step does before moving on.

Node 1 — plan_steps (Reason)

This is the CoT step. We ask the LLM to break the question into 2–3 reasoning steps. The response comes back as plain text (often a numbered list); we split on newlines and strip bullet markers to get a clean list of strings. That list goes into sub_steps. 

def plan_steps(state: RAGCoTState) -> RAGCoTState:
    prompt = f"Break the question into 2-3 reasoning steps: \n\n {state.question}"
    result = llm.invoke(prompt).content
    subqs = [line.strip("- ") for line in result.split("\n") if line.strip()]
    return state.model_copy(update={"sub_steps": subqs})

Why not retrieve against the raw question? Because a vague or multi-part question embeds poorly. “Additional experiments in transformer evaluation” as a search query might miss chunks about FlashAttention2 or LoRA tuning. Sub-steps like “identify gaps in standard evaluation” retrieve much more precisely.

Node 2 — retrieve_per_step (Act)

Loop over every sub-step and call the retriever once per step. All returned documents are appended to a single list — we don't deduplicate here, which is fine for a first pass. The pooled list lands in retrieved_docs. 

def retrieve_per_step(state: RAGCoTState) -> RAGCoTState:
    all_docs = []
    for sub in state.sub_steps:
        docs = retriever.invoke(sub)
        all_docs.extend(docs)
    return state.model_copy(update={"retrieved_docs": all_docs})

Node 3 — generate_answer (Observe + Reflect)

Concatenate every retrieved chunk into one context block, then ask the LLM to synthesize a final answer to the original question (not the sub-steps). The prompt explicitly tells the model to reason over the combined evidence. 

def generate_answer(state: RAGCoTState) -> RAGCoTState:
    context = "\n\n".join([doc.page_content for doc in state.retrieved_docs])
    prompt = f"""
You are answering a complex question using reasoning and retrieved documents.

Question: {state.question}

Relevant Information:
{context}

Now synthesize a well-reasoned final answer.
"""
    result = llm.invoke(prompt).content.strip()
    return state.model_copy(update={"answer": result})

Step 5 — Build & Compile the Graph

Now wire the three nodes into a LangGraph. The pattern is always the same: create a StateGraph, register each node by name, set the entry point, add edges between nodes, and call compile(). 

builder = StateGraph(RAGCoTState)
builder.add_node("planner", plan_steps)
builder.add_node("retriever", retrieve_per_step)
builder.add_node("responder", generate_answer)

builder.set_entry_point("planner")
builder.add_edge("planner", "retriever")
builder.add_edge("retriever", "responder")
builder.add_edge("responder", END)

graph = builder.compile()
EdgeWhat happens
planner → retrieverSub-steps are ready; retrieval begins.
retriever → responderDocuments are pooled; answer generation begins.
responder → ENDAnswer is written; graph exits.

This is still a straight line — no conditional edges, no loops. That simplicity is intentional: CoT RAG is the smallest upgrade from basic RAG, and you'll need this same three-node spine in every later sub-chapter.

Step 6 — Run It

Create an initial state with just the question, invoke the graph, and read back the final state. LangGraph returns the state as a dict, so access fields with bracket notation. 

query = "What are the additional experiments in Transformer evaluation?"
state = RAGCoTState(question=query)
final = graph.invoke(state)

print("\n🪜 Reasoning Steps:", final["sub_steps"])
print("\n✅ Final Answer:\n", final["answer"])

Watch the printed reasoning steps first — they tell you what the planner thought the question needed. Then read the final answer and check whether the retrieved context actually supported it. If the sub-steps look wrong, the retriever searched for the wrong things and the answer will drift.

Output — Reasoned Synthesis

Reasoning Steps: (1) understand the baseline evaluation metrics, (2) identify gaps the standard evaluation misses, (3) propose additional experiments.

Final Answer: The additional experiments in the transformer evaluation log include FlashAttention2 (integrated into LLaMA2, ~50% lower context latency), Chain-of-Thought prompting (+8% on logic tasks, +3% from reflective prompting), tool-augmented prompting (LangGraph + Wikipedia + SQL), a human evaluation protocol (annotators plus GPT-4 as a synthetic judge), retrieval experiments (hybrid dense + sparse, Weaviate vs FAISS + BM25), LoRA tuning (rank 8, ~60% less GPU memory), and a safety suite (Detoxify, zero-shot out-of-scope filter, red teaming).

The planner's sub-steps guided retrieval toward evaluation gaps and experiments, not just a generic search for “transformer evaluation.” That's the whole point of CoT RAG: better retrieval queries produce better answers, without changing the retriever itself.

Key Takeaways

Remember These Three

1. CoT RAG adds a planner node before retrieval so the retriever searches for sub-steps, not just the raw question.

2. The graph is still a straight chain — no loops yet. That makes it the simplest upgrade from basic RAG.

3. Pool all retrieved docs, then synthesize once. The reasoning steps guide what to retrieve; the responder still answers the original question.
