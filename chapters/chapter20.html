Chapter 20: Cache-Augmented Generation (CAG)

Why call the LLM twice for the same question? Cache the answer once and serve it back in 0.00 seconds — first with a simple key–value dictionary, then with a semantic FAISS cache wired into a LangGraph workflow.

CAG in One Paragraph

Every chapter so far has treated the LLM call as unavoidable: question in, retrieval, generation, answer out — every single time. Cache-Augmented Generation asks a simpler question first: have we answered this before? If yes, return the stored response instantly — no retrieval, no generation, no tokens billed. If no, run the normal pipeline and write the answer into the cache on the way out, so the next similar question is free.

In the notebook this cuts a 17-second, 500-word generation down to 0.00 seconds on the second ask. That's not an optimization — it's a different cost curve.

Overview

CAG is a retrieval-free path through your RAG system. Instead of querying external knowledge sources at inference time, it reuses work the model has already done — storing question–answer pairs and serving them back when the same (or a similar) question arrives. The broader CAG idea goes further still: preload relevant documents into the LLM's extended context window and precompute the model's key–value (KV) cache, so generation needs no retrieval step at all.

This chapter builds caching twice, exactly as the source notebook does, and the difference between the two versions is the whole lesson:

  • Simple CAG — a Python dictionary keyed on the exact query string. Ten lines, dramatic speedup, one fatal flaw: change a single word and the cache misses.
  • Advanced CAG — a semantic cache. Questions are embedded and stored in a FAISS index; a new question that lands within a distance threshold of a cached one gets the cached answer, even when the wording is different. Built as a six-node LangGraph with a conditional edge that branches between the cache path and the full RAG path.
The Core Idea

A cache is just a key–value store: the question is the key, the answer is the value. The entire design question of CAG is what counts as “the same key”. Match on the exact string and you get a dictionary — fast, trivial, brittle. Match on meaning (embed the question, search a vector index, accept anything within a distance threshold) and you get a semantic cache that survives rephrasing. Part 1 builds the first; Part 2 builds the second.

CAG vs RAG — What Actually Changes

CAG doesn't replace RAG — in the advanced build below, the full retrieve-and-generate pipeline is still there. CAG wraps a fast path around it. The comparison that matters is what happens per request:

Plain RAGRAG + CAG (this chapter)
Every request Embed → retrieve → generate Check cache first; only misses run the pipeline
Repeat question Full cost again (~17s in the demo) Served from cache (~0.00s)
Token cost Paid on every call Paid once per unique question
Answer consistency Same question can get different answers Cached questions always get the same answer
Freshness Always current Cached answers can go stale → add a TTL

Part 1 — Simple CAG: An Exact-Match Dictionary

The first version needs nothing but a Python dict. Set up the model the usual way — init_chat_model with gpt-4o-mini: 

import os
from dotenv import load_dotenv
load_dotenv()
os.environ["OPENAI_API_KEY"] = os.getenv("OPENAI_API_KEY")

from langchain.chat_models import init_chat_model

llm = init_chat_model("openai:gpt-4o-mini")

The cache itself is one line — an empty dictionary that will map query → response: 

### Cache variable
Model_Cache = {}

The cache_model Function

All the logic lives in one function. Look up the query in the dict: if it's there, cache hit — return the stored response and skip the LLM entirely. If not, cache miss — invoke the model, time it, and store the response under the query before returning it. Timing both paths is what makes the payoff visible. 

import time

def cache_model(query):
    start_time = time.time()
    if Model_Cache.get(query):
        print("**Cache Hit**")
        end_time = time.time()
        elapsed_time = end_time - start_time
        print(f"EXECUTION TIME: {elapsed_time:.2f} seconds")
        return Model_Cache.get(query)
    else:
        print("***CACHE MISS - EXECUTING MODEL***")
        start_time = time.time()
        response = llm.invoke(query)
        end_time = time.time()
        elapsed = end_time - start_time
        print(f"EXECUTION TIME: {elapsed:.2f} seconds")
        Model_Cache[query] = response
        return response

The Fatal Flaw

Now drop two words — “can you” — and ask again. Any human would say this is the same question. The dictionary disagrees: 

query = "give me 500 words on langgraph?"
response = cache_model(query)
print(response)
***CACHE MISS - EXECUTING MODEL***
EXECUTION TIME: 12.18 seconds
content='LangGraph is an innovative concept that sits at the intersection of
natural language processing (NLP) and graph-based representation of
information. ...'
Gotcha — Exact Match Means EXACT

The dict key is the raw string. “What is LangChain?” and “Explain about LangChain?” mean the same thing to a person — and are completely different keys to a dictionary. So are a stray space, a capital letter, or a trailing question mark. Real users never phrase a question the same way twice, which means an exact-match cache almost never hits in production. The fix: stop comparing strings and start comparing embeddings.

Part 2 — Advanced CAG: A Semantic Cache in LangGraph

The advanced build turns the cache into a second vector store. Every answered question is embedded and written into a FAISS index (QA_CACHE) with its answer stored in the metadata. When a new question arrives, we embed it and search the cache: if the nearest cached question is within an L2 distance threshold, that's a semantic hit — serve the stored answer. Otherwise fall through to normal RAG (retrieve from RAG_STORE, generate) and write the fresh Q&A pair into the cache for next time.

This is the graph from the notes — six nodes and one conditional edge, exactly as the compiled LangGraph renders it:

Imports & Configuration

Two vector stores share one embedding model: HuggingFace's all-MiniLM-L6-v2 (384 dimensions, free, local). The numbers that shape the cache's behavior are pulled into one config block so they're tunable in one place: 

from __future__ import annotations
from typing import TypedDict, List, Optional
import time

# ---- LangGraph / LangChain ----
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver

from langchain_core.documents import Document
from langchain_openai import ChatOpenAI
from langchain_huggingface import HuggingFaceEmbeddings

# ---- FAISS vector stores ----
import faiss
from langchain_community.vectorstores import FAISS
from langchain_community.docstore.in_memory import InMemoryDocstore
# ================= CONFIG =================
EMBED_MODEL = "sentence-transformers/all-MiniLM-L6-v2"  # 384-dim
VECTOR_DIM = 384

LLM_MODEL = "gpt-4o-mini"
LLM_TEMPERATURE = 0

RETRIEVE_TOP_K = 4
CACHE_TOP_K = 3

CACHE_DISTANCE_THRESHOLD = 0.45

# Optional TTL for cache entries (seconds). 0 = disabled.
CACHE_TTL_SEC = 0
SettingValueWhat it controls
CACHE_DISTANCE_THRESHOLD 0.45 The heart of the semantic cache. FAISS returns L2 distance (lower = more similar); only a nearest neighbor at or under this distance counts as a hit. Lower it and the cache gets stricter (fewer, safer hits); raise it and it gets looser (more hits, more risk of serving a wrong answer).
CACHE_TOP_K 3 How many cached neighbors to fetch per lookup — only the best one is gated against the threshold.
RETRIEVE_TOP_K 4 Chunks pulled from the knowledge base on the miss path (normal RAG retrieval).
CACHE_TTL_SEC 0 Expiry for cached answers. 0 disables it; set it in production so stale answers age out (every entry stores a ts timestamp for exactly this).
LLM_TEMPERATURE 0 Deterministic generation — sensible when answers are going to be reused verbatim.

The State

The graph's state carries the question through both paths. cache_hit is the flag the conditional edge will branch on; citations lets us see at a glance where an answer came from ([doc-i] markers on the RAG path, "(cache)" on the cache path). 

# ================= STATE ==================
class RAGState(TypedDict):
    question: str
    normalized_question: str
    context_docs: List[Document]
    answer: Optional[str]
    citations: List[str]
    cache_hit: bool

Two Vector Stores & the LLM

First the embedder, then the empty cache. You can't call FAISS.from_texts with nothing, so the cache is assembled by hand from a raw IndexFlatL2 (exact L2 search), an empty in-memory docstore, and an empty id map — a safe zero-entry start: 

# ============== GLOBALS ===================
from langchain_huggingface import HuggingFaceEmbeddings
EMBED = HuggingFaceEmbeddings(model_name=EMBED_MODEL)
# ----- QA CACHE (EMPTY, SAFE INIT) -----
qa_index = faiss.IndexFlatL2(VECTOR_DIM)  # distance; lower is better
QA_CACHE = FAISS(
    embedding_function=EMBED,
    index=qa_index,
    docstore=InMemoryDocstore({}),
    index_to_docstore_id={}
)

The knowledge base for the RAG path is a small demo store — four facts about LangGraph, RAG, and caching: 

# ----- RAG STORE (demo only) -----
RAG_STORE = FAISS.from_texts(
    texts=[
        "LangGraph lets you compose stateful LLM workflows as graphs.",
        "In LangGraph, nodes can be cached; node caching memoizes outputs keyed by inputs for a TTL.",
        "Retrieval-Augmented Generation (RAG) retrieves external context and injects it into prompts.",
        "Semantic caching reuses prior answers when new questions are semantically similar."
    ],
    embedding=EMBED,
)
LLM = ChatOpenAI(model=LLM_MODEL, temperature=LLM_TEMPERATURE)
Why two FAISS stores?

They hold different things and answer different questions. RAG_STORE holds knowledge chunks and answers “what documents are relevant to this question?” QA_CACHE holds previously asked questions (with their answers tucked into metadata) and answers “have I been asked something like this before?” Same technology, opposite roles — one feeds the LLM, the other replaces it.

The Six Nodes

Each node is a plain function over RAGState. The interesting one is semantic_cache_lookup — note its three guard rails: skip an empty question, skip an empty index (FAISS crashes if you search a zero-entry index), and optionally skip entries older than the TTL. Only then does the distance gate decide hit or miss. 

# ================ NODES ===================
def normalize_query(state: RAGState) -> RAGState:
    q = (state["question"] or "").strip()
    state["normalized_question"] = q.lower()
    return state

def semantic_cache_lookup(state: RAGState) -> RAGState:
    q = state["normalized_question"]
    state["cache_hit"] = False  # default

    if not q:
        return state

    # Guard: FAISS crashes if ntotal == 0 and you ask for k>0
    if getattr(QA_CACHE, "index", None) is None or QA_CACHE.index.ntotal == 0:
        return state

    # For FAISS L2 wrapper, this returns (Document, distance) with lower=better
    hits = QA_CACHE.similarity_search_with_score(q, k=CACHE_TOP_K)
    if not hits:
        return state

    best_doc, dist = hits[0]

    # Optional TTL
    if CACHE_TTL_SEC > 0:
        ts = best_doc.metadata.get("ts")
        if ts is None or (time.time() - float(ts)) > CACHE_TTL_SEC:
            return state

    # L2 distance gate (lower = more similar)
    if dist <= CACHE_DISTANCE_THRESHOLD:
        cached_answer = best_doc.metadata.get("answer")
        if cached_answer:
            state["answer"] = cached_answer
            state["citations"] = ["(cache)"]
            state["cache_hit"] = True

    return state

def respond_from_cache(state: RAGState) -> RAGState:
    return state

def retrieve(state: RAGState) -> RAGState:
    q = state["normalized_question"]
    docs = RAG_STORE.similarity_search(q, k=RETRIEVE_TOP_K)
    state["context_docs"] = docs
    return state

def generate(state: RAGState) -> RAGState:
    q = state["question"]
    docs = state.get("context_docs", [])
    ctx = "\n\n".join([f"[doc-{i}] {d.page_content}" for i, d in enumerate(docs, start=1)])

    system = (
        "You are a precise RAG assistant. Use the context when helpful. "
        "Cite with [doc-i] markers if you use a fact from the context."
    )
    user = f"Question: {q}\n\nContext:\n{ctx}\n\nWrite a concise answer with citations."

    resp = LLM.invoke([{"role": "system", "content": system},
                       {"role": "user", "content": user}])
    state["answer"] = resp.content
    state["citations"] = [f"[doc-{i}]" for i in range(1, len(docs) + 1)]
    return state

def cache_write(state: RAGState) -> RAGState:
    q = state["normalized_question"]
    a = state.get("answer")
    if not q or not a:
        return state

    QA_CACHE.add_texts(
        texts=[q],
        metadatas=[{
            "answer": a,
            "ts": time.time(),
        }]
    )
    return state
NodeJobDetail worth noticing
normalize_query Canonicalize the question strip() + lower() — so trivial formatting differences never reach the cache
semantic_cache_lookup Decide hit or miss Embeds the question, finds the nearest cached question, gates on dist <= 0.45; on a hit it copies the stored answer into state and sets cache_hit = True
respond_from_cache Serve the hit A pass-through — the lookup already wrote the answer; this node just gives the hit path a named stop before END
retrieve Normal RAG retrieval Top-4 chunks from RAG_STORE on the miss path
generate Grounded answer Labels each chunk [doc-i] and asks for citations — the citation markers survive into the cached answer
cache_write Remember for next time Adds the normalized question as the vector, with the answer and a timestamp in metadata — the write that makes the next similar question free

Wiring the Graph

The wiring mirrors the diagram exactly. One conditional edge does the branching: after semantic_cache_lookup, the _branch router reads cache_hit and sends the flow either to respond_from_cache (→ END) or into the RAG path retrieve → generate → cache_write (→ END). 

# ============== GRAPH WIRING ==============
graph = StateGraph(RAGState)

graph.add_node("normalize_query", normalize_query)
graph.add_node("semantic_cache_lookup", semantic_cache_lookup)
graph.add_node("respond_from_cache", respond_from_cache)
graph.add_node("retrieve", retrieve)
graph.add_node("generate", generate)
graph.add_node("cache_write", cache_write)

graph.set_entry_point("normalize_query")
graph.add_edge("normalize_query", "semantic_cache_lookup")

def _branch(state: RAGState) -> str:
    return "respond_from_cache" if state.get("cache_hit") else "retrieve"

graph.add_conditional_edges(
    "semantic_cache_lookup",
    _branch,
    {
        "respond_from_cache": "respond_from_cache",
        "retrieve": "retrieve"
    }
)

graph.add_edge("respond_from_cache", END)
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", "cache_write")
graph.add_edge("cache_write", END)

memory = MemorySaver()
app = graph.compile(checkpointer=memory)

Read the citations, they tell the story

['[doc-1]', '[doc-2]', ...] means the answer was generated — the full retrieve-and-generate path ran. ['(cache)'] means the answer was remembered — no retrieval, no generation. Four questions produced only two LLM calls. That ratio is the entire value proposition of CAG, and it improves the longer the system runs.

Exact-Match vs Semantic Cache

AspectSimple CAG (Model_Cache dict)Advanced CAG (FAISS QA_CACHE)
Key Exact query string Embedding of the normalized query
“Same question” means Byte-for-byte identical Within L2 distance 0.45 of a cached question
Survives rephrasing No — “give me” vs “can you give me” misses Yes — “What is X?” hits for “Explain about X?”
Lookup cost O(1) dict lookup, effectively free One embedding + one FAISS search — milliseconds, still far cheaper than the LLM
False-hit risk None (exact match can't be wrong) Real — threshold too loose serves wrong answers; tune it on your traffic
Staleness control None built in Per-entry ts timestamp + CACHE_TTL_SEC
Best for Demos, fixed FAQ lists, idempotent internal calls Real user traffic, where nobody phrases anything the same way twice
Production notes

Three knobs matter when you take this live. The threshold: 0.45 works for this demo's embedder and data — validate on your own query pairs, because it's a trade between hit rate and wrong-answer risk. The TTL: any cache serving facts that change needs CACHE_TTL_SEC > 0, or yesterday's answer outlives its truth. The store: both the in-RAM dict and the in-RAM FAISS index vanish on restart — production semantic caches persist (saved FAISS index, Redis semantic cache, or a vector DB), just like Chapter 19's checkpointer story.

Key Takeaways

Remember These Four

1. CAG is a fast path, not a replacement. The full RAG pipeline is still there — the cache just answers the questions it has already seen, turning a 17-second generation into a 0.00-second lookup and cutting token spend to once per unique question.

2. Exact-match caching breaks on real users. A dict keyed on the query string misses the moment anyone rephrases — “What is LangChain?” and “Explain about LangChain?” are different keys but the same question. That flaw is what motivates everything in Part 2.

3. A semantic cache is a vector store of past questions. Embed the normalized query, search QA_CACHE, and gate the nearest neighbor on an L2 distance threshold (0.45 here). Hit → serve the answer from metadata; miss → retrieve, generate, and cache_write the new pair.

4. One conditional edge makes it a graph. normalize_query → semantic_cache_lookup, then _branch on cache_hit: hit → respond_from_cache → END, miss → retrieve → generate → cache_write → END. The citations tell you which path ran — [doc-i] means generated, (cache) means remembered.
