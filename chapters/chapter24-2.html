RAG Evaluation

A RAG target must return its evidence. With answer and documents exposed, four judges can measure correctness, answer relevance, groundedness, and retrieval relevance — and catch a correct answer that retrieval never supported.

Overview

This sub-chapter builds a small RAG application over three Lilian Weng articles, creates three reference examples, and evaluates both retrieval and generation. The LangSmith loop from the chatbot half stays the same; the target and evaluators grow to inspect intermediate documents.

The Core Idea

Returning documents is not debugging noise. Without it, you cannot tell whether a fluent answer came from the corpus or from the model's prior knowledge.

Build the RAG System Under Test

The RAG benchmark uses three Lilian Weng posts: autonomous agents, prompt engineering, and adversarial attacks on LLMs. The documents are split into 250-token chunks with no overlap, embedded into an in-memory vector store, and exposed through a retriever. 

from langchain_community.document_loaders import WebBaseLoader
from langchain_core.vectorstores import InMemoryVectorStore
from langchain_openai import OpenAIEmbeddings
from langchain_text_splitters import RecursiveCharacterTextSplitter

urls = [
    "https://lilianweng.github.io/posts/2023-06-23-agent/",
    "https://lilianweng.github.io/posts/2023-03-15-prompt-engineering/",
    "https://lilianweng.github.io/posts/2023-10-25-adv-attack-llm/",
]

docs = [WebBaseLoader(url).load() for url in urls]
docs_list = [item for sublist in docs for item in sublist]

text_splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
    chunk_size=250,
    chunk_overlap=0,
)
doc_splits = text_splitter.split_documents(docs_list)

vectorstore = InMemoryVectorStore.from_documents(
    documents=doc_splits,
    embedding=OpenAIEmbeddings(),
)

retriever = vectorstore.as_retriever(k=6)

On current LangChain releases, the explicit form is commonly vectorstore.as_retriever(search_kwargs={"k": 6}). The chapter keeps the notebook's executed form above, but production code should pin the LangChain version and confirm the configured k appears in the trace.

The Target Must Return Its Evidence

from langchain.chat_models import init_chat_model
from langsmith import traceable

llm = init_chat_model("openai:gpt-4o-mini")

@traceable()
def rag_bot(question: str) -> dict:
    docs = retriever.invoke(question)
    docs_string = " ".join(doc.page_content for doc in docs)

    instructions = f"""You are a helpful assistant who is good at analyzing source information and answering questions.
Use the following source documents to answer the user's questions.
If you don't know the answer, just say that you don't know.
Use three sentences maximum and keep the answer concise.

Documents:
{docs_string}"""

    ai_msg = llm.invoke([
        {"role": "system", "content": instructions},
        {"role": "user", "content": question},
    ])

    return {
        "answer": ai_msg.content,
        "documents": docs,
    }

Returning documents is not incidental debugging data. Groundedness and retrieval relevance cannot be computed from the final answer alone. Evaluation-friendly application boundaries expose the intermediate artifacts needed to test each stage.

The Four Core RAG Metrics

MetricComparesNeeds reference?Failure means
CorrectnessAnswer ↔ reference answerYesThe final claim conflicts with or misses ground truth
Answer relevanceQuestion ↔ answerNoThe response does not directly help with the question
GroundednessRetrieved docs ↔ answerNoThe answer contains claims not justified by retrieved evidence
Retrieval relevanceQuestion ↔ retrieved docsNoThe retriever returned evidence unrelated to the question
Do Not Merge These Scores

A single “RAG quality” number hides the repair action. Low retrieval relevance points to embeddings, chunking, metadata filters, query rewriting, or k. Low groundedness points to the generation prompt, context ordering, citations, or model behavior. Low answer relevance points to instruction following. Low correctness may involve any stage — and needs row-level evidence to diagnose.

Create the RAG Evaluation Dataset

The three examples deliberately target different source articles so the benchmark exercises routing through the shared vector store rather than repeating one topic.

Build Structured LLM Judges

The RAG evaluators use TypedDict schemas with two fields: an explanation followed by a Boolean verdict. Putting the explanation first forces the judge to analyze the evidence before emitting its score, while strict JSON Schema makes the return value machine-readable.

Correctness: Answer vs Reference

from typing_extensions import Annotated, TypedDict
from langchain_openai import ChatOpenAI

class CorrectnessGrade(TypedDict):
    explanation: Annotated[str, ..., "Explain your reasoning for the score"]
    correct: Annotated[
        bool,
        ...,
        "True if the answer is correct, False otherwise.",
    ]

correctness_instructions = """You are a teacher grading a quiz.

You will be given a QUESTION, the GROUND TRUTH (correct) ANSWER,
and the STUDENT ANSWER.

Grade the student answer only on factual accuracy relative to the
ground truth. It must not contain conflicting statements. Extra
information is acceptable when it is factually accurate.

Explain your reasoning, then return the correctness verdict."""

grader_llm = ChatOpenAI(
    model="gpt-4o-mini",
    temperature=0,
).with_structured_output(
    CorrectnessGrade,
    method="json_schema",
    strict=True,
)

def correctness(inputs: dict, outputs: dict, reference_outputs: dict) -> bool:
    answers = f"""QUESTION: {inputs['question']}
GROUND TRUTH ANSWER: {reference_outputs['answer']}
STUDENT ANSWER: {outputs['answer']}"""

    grade = grader_llm.invoke([
        {"role": "system", "content": correctness_instructions},
        {"role": "user", "content": answers},
    ])
    return grade["correct"]

Answer Relevance: Answer vs Question

class RelevanceGrade(TypedDict):
    explanation: Annotated[str, ..., "Explain your reasoning for the score"]
    relevant: Annotated[bool, ..., "Whether the answer addresses the question"]

relevance_llm = ChatOpenAI(
    model="gpt-4o",
    temperature=0,
).with_structured_output(
    RelevanceGrade,
    method="json_schema",
    strict=True,
)

def relevance(inputs: dict, outputs: dict) -> bool:
    answer = (
        f"QUESTION: {inputs['question']}\n"
        f"STUDENT ANSWER: {outputs['answer']}"
    )
    grade = relevance_llm.invoke([
        {"role": "system", "content": relevance_instructions},
        {"role": "user", "content": answer},
    ])
    return grade["relevant"]

No reference is passed here. Relevance asks whether the response addresses the user's intent, not whether its claims are factually correct.

Groundedness: Answer vs Retrieved Documents

class GroundedGrade(TypedDict):
    explanation: Annotated[str, ..., "Explain your reasoning for the score"]
    grounded: Annotated[
        bool,
        ...,
        "Whether the answer is supported by the documents",
    ]

grounded_llm = ChatOpenAI(
    model="gpt-4o",
    temperature=0,
).with_structured_output(
    GroundedGrade,
    method="json_schema",
    strict=True,
)

def groundedness(inputs: dict, outputs: dict) -> bool:
    doc_string = "\n\n".join(
        doc.page_content for doc in outputs["documents"]
    )
    answer = f"FACTS: {doc_string}\nSTUDENT ANSWER: {outputs['answer']}"
    grade = grounded_llm.invoke([
        {"role": "system", "content": grounded_instructions},
        {"role": "user", "content": answer},
    ])
    return grade["grounded"]

Retrieval Relevance: Documents vs Question

class RetrievalRelevanceGrade(TypedDict):
    explanation: Annotated[str, ..., "Explain your reasoning for the score"]
    relevant: Annotated[
        bool,
        ...,
        "Whether the retrieved documents are relevant to the question",
    ]

retrieval_relevance_llm = ChatOpenAI(
    model="gpt-4o",
    temperature=0,
).with_structured_output(
    RetrievalRelevanceGrade,
    method="json_schema",
    strict=True,
)

def retrieval_relevance(inputs: dict, outputs: dict) -> bool:
    doc_string = "\n\n".join(
        doc.page_content for doc in outputs["documents"]
    )
    answer = f"FACTS: {doc_string}\nQUESTION: {inputs['question']}"
    grade = retrieval_relevance_llm.invoke([
        {"role": "system", "content": retrieval_relevance_instructions},
        {"role": "user", "content": answer},
    ])
    return grade["relevant"]
Evaluator signatureData made available
(inputs, outputs, reference_outputs)Correctness can inspect the curated answer
(inputs, outputs)Relevance sees the question and final answer
(inputs, outputs)Groundedness and retrieval relevance read outputs["documents"]

Run the RAG Experiment

def target(inputs: dict) -> dict:
    return rag_bot(inputs["question"])

experiment_results = client.evaluate(
    target,
    data=dataset_name,
    evaluators=[
        correctness,
        groundedness,
        relevance,
        retrieval_relevance,
    ],
    experiment_prefix="rag-doc-relevance",
    metadata={"version": "LCEL context, gpt-4-0125-preview"},
)

experiment_results.to_pandas()

Three application calls plus four model-based evaluators per example completed in about 40 seconds in the captured run. That multiplication matters: LLM judges add evaluation cost and latency. Run them offline, cache stable judge inputs where appropriate, and reserve expensive rubrics for the examples that need semantic review.

The Revealing Failure: Correct Answer, Broken RAG Path

This is the chapter's central result. If evaluation stopped at correctness and answer relevance, the benchmark would report a perfect system. The document-aware scores show a different reality: the ReAct answer passed because the model could produce a plausible correct response despite weak evidence. In a private knowledge base, that same behavior becomes an unsupported claim or hallucination.

Observed patternLikely interpretationInspect first
Correct ✓ · Grounded ✗Model knew the answer, but RAG did not support itRetrieved chunks and generation prompt
Retrieval relevant ✓ · Grounded ✗Useful evidence arrived, but generation ignored or contradicted itContext formatting, ordering, and instructions
Retrieval relevant ✗ · Correct ✗Retrieval failure likely propagated to the answerQuery, embeddings, filters, chunking, and k
Grounded ✓ · Correct ✗The answer faithfully repeats evidence that is incomplete or wrongSource quality and corpus freshness
Correct ✓ · Answer relevant ✗Facts may be right, but the response does not resolve user intentPrompt, verbosity, and question interpretation

Making LLM Judges Reliable

An LLM judge is itself a model-driven system. Treat its rubric, prompt, schema, and model version as evaluation code, not as an oracle. A stable benchmark combines judge discipline with deterministic measurements and periodic human calibration.

PracticeReason
Use explicit pairwise relationships“Question vs docs” is more testable than “overall RAG quality”
Define pass and fail criteria in the rubricReduces hidden assumptions and judge variability
Use structured output with a strict schemaEliminates brittle text parsing
Set temperature to zero and pin the judge modelImproves repeatability across runs
Keep the explanation with the scoreMakes disagreements auditable and helps debug the rubric
Build an adversarial calibration setTests partial answers, plausible hallucinations, and irrelevant but keyword-rich docs
Measure judge agreement with humansValidates that automated scores reflect the intended quality bar
Version evaluator prompts and modelsPrevents score drift from masquerading as application change
A Subtle Rubric Choice

The notebook's retrieval-relevance rubric passes a document set when it contains any keyword or semantic meaning related to the question, even if some content is unrelated. That is a permissive recall-oriented rule. A production system may also need a precision metric that scores each retrieved chunk separately, because one useful chunk should not hide five distracting ones.

Key Takeaways

Remember These Five

1. Evaluation is a repeatable experiment. Freeze examples, run a target over all of them, score with explicit evaluators, preserve traces and metadata, then compare named versions. A few manual prompts are a demo, not a benchmark.

2. RAG has four distinct relationships. Correctness compares answer to reference, answer relevance compares answer to question, groundedness compares answer to retrieved documents, and retrieval relevance compares documents to question. Each failure points to a different repair path.

3. Return intermediate artifacts. A RAG target should expose the final answer and retrieved documents; otherwise retrieval and grounding cannot be evaluated independently.

4. Correct does not mean grounded. The source experiment scored all three answers correct, yet one ReAct answer was neither grounded nor backed by relevant retrieval. The model likely answered from prior knowledge, revealing a broken RAG path hidden behind a good final response.

5. Treat judges as versioned software. Combine deterministic checks, structured LLM judges, and calibrated human review. Pin judge models, make rubrics explicit, test disagreement cases, and track judge changes separately from application changes.
