Chapter 24: RAG Evaluation — Measuring Retrieval, Grounding & Answer Quality

A fluent answer is not evidence that a RAG system works. Evaluation turns questions, reference answers, retrieved documents, and model responses into repeatable experiments that tell you exactly which stage succeeded — and which one only looked successful.

Evaluation in One Paragraph

An evaluation is a controlled experiment: freeze a dataset, run a version of the application over every example, score each run with explicit evaluators, and compare experiments. A chatbot can be judged from its question, answer, and reference answer. RAG needs more visibility: the answer may be correct while the retriever found weak evidence, or well-grounded while failing to answer the question. That is why this chapter measures four separate relationships instead of collapsing quality into one vague score.

Why Evaluation Must Be a Dataset, Not a Demo

A single hand-picked question can show that a system runs. It cannot show that the system is reliable. The handwritten notes frame evaluation as four deliberate activities: generate or curate data points, use an LLM as a judge where semantic interpretation is needed, define evaluation metrics, and compare multiple LLM or application versions. LangSmith provides the dataset and experiment layer that makes those activities repeatable.

ObjectWhat it containsWhy it is frozen
ExampleInput question and optional reference outputEvery version faces the same test case
DatasetA named collection of examplesDefines the benchmark population
TargetA function mapping example inputs to application outputsCreates one consistent invocation boundary
EvaluatorA scoring function with a declared data dependencyMakes “good” explicit and reviewable
ExperimentRuns, traces, scores, errors, metadata, and timingProvides a versioned comparison point
The Comparison Rule

Change one major variable per experiment whenever possible. If you switch the model, rewrite the prompt, change chunking, and replace the embedding model together, a better score cannot tell you which change helped. Evaluation is useful only when experiments remain interpretable.

The Core Idea

Chatbot evaluation asks whether the answer matches a reference. RAG evaluation asks four questions: Is the answer correct? Does it address the question? Is it grounded in the retrieved docs? Were those docs relevant? A single aggregate score hides which stage failed.

Key Takeaways

Remember These Four

1. Evaluation is a repeatable experiment — freeze examples, run a target, score with explicit evaluators, compare named versions.

2. Start with a chatbot benchmark to learn the LangSmith loop before adding retrieval artifacts.

3. RAG needs four relationships, not one score — correctness, answer relevance, groundedness, and retrieval relevance.

4. A correct answer can still hide a broken retrieval path; expose intermediate documents so you can catch it.
