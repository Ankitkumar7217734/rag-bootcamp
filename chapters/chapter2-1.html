Phase 2 & 3: Query Processing and Generation

How a user's question becomes a grounded, cited answer — the online half of the RAG pipeline.

The Online Pipeline

The ingestion phase (Chapter 2) built the searchable index offline. Now, every time a user sends a message, the online pipeline fires in real time. It has two phases: Query Processing (find the right context) and Generation (produce the answer).

Phase 2: Query Processing

Step 1 — Embed the User's Query

The user's question — say, "What is RAG?" — is passed through the same embedding model used during ingestion. This is critical: the query and the stored chunks must live in the same vector space for similarity search to work.

User query:  "What is RAG?"
Embedding:   [0.3, 0.4, 0.6, 0.2, ...]    ← now a vector

Step 2 — Similarity Search

The query vector is compared against every stored chunk vector in the database. The chunks with the highest similarity scores are retrieved as candidates.

There are two main distance metrics used for this comparison:

MetricWhat it MeasuresWhen to Use
Cosine SimilarityAngle between vectors (direction)Text search — the default choice. Length-insensitive, so short and long texts compare fairly.
Euclidean DistanceStraight-line gap between pointsWhen vector magnitudes matter (e.g., image embeddings, some numeric data).
Dot ProductMagnitude + directionWhen embeddings are already normalized — fast and equivalent to cosine then.

Step 3 — Retrieved Results (D1, D2, D3)

The top-k most similar chunks are returned from the vector database. A typical RAG system retrieves 3–10 chunks — enough context to answer well, but not so much that it overwhelms the LLM's context window.

Step 4 — Enriching with Metadata

Before passing to the LLM, retrieved chunks are enriched with their metadata (source, page, date). This is what enables source citations in the final answer.

{
  "original_query": "What is RAG?",
  "retrieved_context": [
    {
      "text": "RAG stands for Retrieval-Augmented Generation...",
      "source": "RAG_Bootcamp_Notes.pdf",
      "page": 1
    },
    {
      "text": "The vector database stores embeddings generated by...",
      "source": "RAG_Architecture_Guide.pdf",
      "page": 7
    }
  ]
}

Phase 3: Generation

Step 5 — Building the Augmented Prompt

The original query and retrieved context are combined into a single prompt for the LLM. A typical template looks like this:

Augmented Prompt Template

System: You are a helpful assistant. Answer the question using ONLY the provided context. If the answer is not in the context, say "I don't know." Cite your sources.

Context:
[D1]: "RAG stands for Retrieval-Augmented Generation. It is a technique that..."
[D2]: "The vector database enables semantic search across millions of documents..."
[D3]: "Embedding models convert text into numerical vectors that capture meaning..."

Question: What is RAG?

Step 6 — LLM Options

The augmented prompt is sent to a Large Language Model. The LLM's job is to synthesize the retrieved information into a natural, coherent answer — not to hallucinate from its training data. Common LLM providers for RAG pipelines:

ProviderNotable ModelsStrength
OpenAIGPT-4o, GPT-4 TurboIndustry standard, large context window
AnthropicClaude Sonnet, Claude HaikuLong context, low hallucination rate
GoogleGemini 1.5 Pro1M token context, multimodal
GroqLlama 3 (via Groq)Ultra-fast inference (open-source models)
Meta / OSSLlama 3, MistralSelf-hosted, no data leaves your infra

Step 7 — Summarized Output

The LLM produces the final answer, synthesizing only the retrieved context. A well-formed RAG response has three qualities:

  1. Grounded — every factual claim is traceable to a retrieved chunk
  2. Cited — the source and page are included so the user can verify
  3. Concise — the LLM summarizes; it doesn't dump the raw retrieved text
Example RAG Response

Question: "What is RAG?"

Answer: "RAG (Retrieval-Augmented Generation) is a technique that enhances AI language models by combining their generation capabilities with external knowledge retrieval. Instead of relying solely on training data, the model retrieves relevant documents at query time and uses them as context to generate accurate, up-to-date answers. [Source: RAG_Bootcamp_Notes.pdf, p.1]"

End-to-End: The Complete RAG Flow

Why This Architecture Works

ProblemHow RAG Solves It
Hallucination The LLM is instructed to answer only from retrieved context, not from its training weights.
Stale Knowledge The vector database can be updated continuously — new documents are ingested without retraining the LLM.
No Citations Metadata (source, page, date) is retrieved alongside the text and included in the prompt, enabling verifiable answers.
Context Limits Instead of stuffing entire documents into the prompt, only the top-k relevant chunks are passed — staying well within the LLM's context window.
Private Data Your documents stay in your own vector database. The LLM only sees what you choose to retrieve and pass as context.
Summary

Ingestion (offline): Documents → Chunks → Embeddings → Vector DB

Query Processing (online): User Query → Embed → Similarity Search → Retrieve Top-K Chunks

Generation (online): Original Query + Retrieved Chunks → Augmented Prompt → LLM → Cited Answer
