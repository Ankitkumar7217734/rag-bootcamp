Multi-Agent Network RAG

Two specialized ReAct agents — a researcher with RAG and Tavily, and a blog writer with no tools — hand work back and forth until one of them signals FINAL ANSWER.

Overview

The simplest multi-agent RAG pattern is a peer network: two agents, each a compiled ReAct subgraph, connected by explicit handoff edges. The researcher gathers facts from an internal vector store and Tavily web search. The blog writer turns those facts into a polished article. Neither agent knows the other's tools — they only see the shared message history.

This is the pattern shown in the notes as “Blog Generation → RAG”: input flows to Agent 1 (Researcher with retriever + Tavily), then to Agent 2 (Content Writer with ReAct and self-reflection), then to output. LangGraph implements the handoffs with Command(goto=...) — each node decides whether to pass control to the other agent or stop.

Where this fits

Building block 1 of 3 in Chapter 16, from 8-multiagent.ipynb. You already know create_react_agent from Chapter 14; here we wrap two of them as graph nodes and wire the edges ourselves.

The Handoff Protocol

Multi-agent networks need a way to say “I'm done, pass it on” or “I'm done, stop the graph.” This notebook uses a simple string convention: any agent whose last message contains FINAL ANSWER routes to END; otherwise control passes to the next agent in the pipeline. 

from langchain_core.messages import BaseMessage, HumanMessage
from langgraph.graph import MessagesState, END
from langgraph.types import Command
from typing import Literal

def get_next_node(last_message: BaseMessage, goto: str):
    if "FINAL ANSWER" in last_message.content:
        return END
    return goto

Each agent's output is re-wrapped as a HumanMessage with a name field (researcher or blog_generator). This is a LangGraph compatibility trick: some model providers reject an AI message as the last entry in the input list when the next agent invokes. Naming the message also makes the shared history readable — you can see which agent said what.

Shared system prompt

Both agents share a base prompt that tells them they are collaborating teammates. The suffix narrows each agent's scope: 

def make_system_prompt(suffix: str) -> str:
    return (
        "You are a helpful AI assistant, collaborating with other assistants."
        " Use the provided tools to progress towards answering the question."
        " If you are unable to fully answer, that's OK, another assistant with different tools "
        " will help where you left off. Execute what you can to make progress."
        " If you or any of the other assistants have the final answer or deliverable,"
        " prefix your response with FINAL ANSWER so the team knows to stop."
        f"\n{suffix}"
    )

Building the Two Agents

Researcher — RAG + web search

The researcher gets the internal retriever tool and Tavily. Its suffix restricts it to research only and tells it a content-writer colleague is waiting: 

from langchain_community.tools.tavily_search import TavilySearchResults
from langgraph.prebuilt import create_react_agent

tavily_tool = TavilySearchResults(max_results=5)

research_agent = create_react_agent(
    llm,
    tools=[internal_tool, tavily_tool],
    prompt=make_system_prompt(
        "You can only do research. Use the tools you are bound with."
        " You are working with a content writer colleague."
    ),
)

Blog writer — no tools, pure generation

The blog writer has an empty tool list. Its job is to read the researcher's findings from the message history and produce a detailed blog post. When satisfied, it prefixes FINAL ANSWER: 

blog_agent = create_react_agent(
    llm,
    tools=[],
    prompt=make_system_prompt(
        "You can only write a detailed blog. You are working with a researcher colleague."
    ),
)

Wrapping Agents as Graph Nodes

Each compiled ReAct agent becomes a node function that invokes the subgraph, checks the handoff condition, and returns a Command with the next destination: 

def research_node(state: MessagesState) -> Command[Literal["blog_generator", END]]:
    result = research_agent.invoke(state)
    goto = get_next_node(result["messages"][-1], "blog_generator")

    result["messages"][-1] = HumanMessage(
        content=result["messages"][-1].content, name="researcher"
    )
    return Command(
        update={"messages": result["messages"]},
        goto=goto,
    )

def blog_node(state: MessagesState) -> Command[Literal["researcher", END]]:
    result = blog_agent.invoke(state)
    goto = get_next_node(result["messages"][-1], "researcher")

    result["messages"][-1] = HumanMessage(
        content=result["messages"][-1].content, name="blog_generator"
    )
    return Command(
        update={"messages": result["messages"]},
        goto=goto,
    )

Notice the asymmetry: if the researcher finishes early (unlikely — it usually passes raw findings), it can END directly. If the blog writer needs more research, it routes back to researcher. The graph can loop until someone says FINAL ANSWER.

Compiling the Workflow

from langgraph.graph import StateGraph, START

workflow = StateGraph(MessagesState)
workflow.add_node("researcher", research_node)
workflow.add_node("blog_generator", blog_node)
workflow.add_edge(START, "researcher")

graph = workflow.compile()

The state is MessagesState — a list of messages with the add_messages reducer. Both agents read and append to the same history, so the blog writer sees every tool call and observation the researcher made.

Running it

response = graph.invoke({
    "messages": "Write a detailed blog on transformer variants in production deployments"
})

print(response["messages"][-1].content)

On the first run the researcher calls InternalResearchNotes, pulling chunks about EfficientFormer (92.4% Top-1 on TinyImageNet, 290 MB peak memory), Longformer (8192-token support logs, >1.2 s latency), Reformer (LSH bucket collisions, gradient clipping fixes), and LLaMA2 (75% win rate vs human with RAG). The blog writer then synthesizes a structured article with sections on lightweight variants, deployment challenges, and practical implementation strategies — prefixed with FINAL ANSWER.

Design Decisions

ChoiceWhyAlternative
Command(goto=...) handoffs Each node explicitly chooses the next step; no separate router LLM needed Supervisor pattern (16.2) when >2 agents or unrelated skills
FINAL ANSWER string check Simple, debuggable stop condition visible in message content Structured output with a done: bool field
Re-wrap as HumanMessage Avoids provider errors when chaining agent subgraphs Custom state schema with separate research_notes field
Researcher has tools; writer doesn't Clean separation of gather vs generate; writer can't accidentally re-retrieve Give writer a read-only document tool
When to Use This Pattern

Multi-agent networks work best for sequential pipelines with 2–3 stages where each stage has a clearly different skill. Blog generation (research → write), code review (analyze → fix), and report building (data → summarize → format) are natural fits. Once you need a central coordinator picking among many unrelated agents, move to the supervisor pattern in the next sub-chapter.

Key Takeaways

Remember These Three

1. A multi-agent network is just ReAct subgraphs wired as graph nodes — each agent keeps its own tool loop; the parent graph handles routing.

2. Command(goto=...) is the handoff primitive. Pair it with a visible stop signal (FINAL ANSWER) so agents know when to release control.

3. Specialization beats generalization: give the researcher retrieval tools and the writer none. Narrow prompts + narrow tools = fewer routing mistakes.
