Chapter 13: Agentic RAG — Retrieval as a Decision

Chapter 12 gave the LLM tools and a loop. Now we point that loop at retrieval itself: instead of blindly fetching documents on every query, the agent decides whether to retrieve, which knowledge base to search, whether the results are any good, and what to do when they aren't. This is Agentic RAG.

Overview

Every RAG pipeline you've built so far has been a straight line: take the question, embed it, retrieve the top-k chunks, stuff them into a prompt, generate. It never pauses to ask whether retrieval was even needed, whether it searched the right place, or whether the chunks it got back are relevant. If the retriever returns garbage, the LLM generates an answer from garbage.

Agentic RAG fixes this by putting an agent in charge of the retrieval process. The agent is an LLM with retriever tools bound to it (exactly the bind_tools pattern from Chapter 12), running inside a LangGraph loop. On each turn it makes real decisions — call a retriever or answer directly, grade what came back, and either generate the final answer or rewrite the question and try again.

The Core Idea

Traditional RAG always retrieves, once, from one place, and trusts the result. Agentic RAG turns retrieval into a series of decisions made by an LLM. Give the model a retriever tool and it can choose to use it — and a grading step lets it judge its own retrievals and loop back to fix a bad query before it ever reaches the answer.

The Simplest Form — A Two-Node Graph

Before adding any decisions, let's see the smallest possible Agentic RAG. Two nodes, no conditions, no loops. The retriever fetches; the responder answers. That's it.

This graph is compiled directly from 1-agenticrag.ipynb. LangGraph draws it as a vertical flow: __start__ → retriever → responder → __end__. Here it is in the site's design system:

The code that produces this graph uses a Pydantic BaseModel as the state (rather than the message-based TypedDict you'll see in the full version) — convenient because dot-access to state.question is cleaner than dict keys:

from typing import List
from pydantic import BaseModel
from langchain_classic.schema import Document
from langgraph.graph import StateGraph, END

# --- State ---
class RAGState(BaseModel):
    question: str
    retrieved_docs: List[Document] = []
    answer: str = ""

# --- Nodes ---
def retrieve_docs(state: RAGState) -> RAGState:
    docs = retriever.invoke(state.question)
    return RAGState(question=state.question, retrieved_docs=docs)

def generate_answer(state: RAGState) -> RAGState:
    context = "\n\n".join([doc.page_content for doc in state.retrieved_docs])
    prompt = (f"Answer the question based on the context.\n\n"
              f"Context:\n{context}\n\nQuestion: {state.question}")
    response = llm.invoke(prompt)
    return RAGState(question=state.question,
                    retrieved_docs=state.retrieved_docs,
                    answer=response.content)

# --- Graph ---
builder = StateGraph(RAGState)
builder.add_node("retriever", retrieve_docs)
builder.add_node("responder", generate_answer)
builder.set_entry_point("retriever")
builder.add_edge("retriever", "responder")
builder.add_edge("responder", END)
graph = builder.compile()

# --- Run ---
initial_state = RAGState(question="What is the concept of agent loop in autonomous agents?")
final_state = graph.invoke(initial_state)
print(final_state['answer'])
What makes this "Agentic" at all?

It's a fair question — there's no LLM decision, no conditional edge, no loop. The answer is that retrieval is now a node in a graph, not a hard-coded function call. That means it can be extended: add a conditional edge after retriever and you can decide whether the result is good enough before calling responder. The two-node graph is the minimal seed; the full graph below grows from it by adding those decisions.

The Full Graph — Adding Decisions

The production-grade version we build across the sub-chapters adds two conditional edges to that simple skeleton, turning the linear pipeline into a graph that routes and self-corrects. Follow the two diamonds — those are the decisions that make it "agentic": does the agent want to retrieve? and are the retrieved docs relevant?

The whole chapter is just code that builds this picture. Two nodes do work the moment they run (retrieve, generate); the two diamonds are conditional edges — small functions that decide where the flow goes next. The dashed amber edge is the loop that lets the system recover from a bad retrieval instead of answering from it.

Why Agentic? Traditional vs Agentic RAG

To see what we're buying, put the two side by side. Traditional RAG is a fixed assembly line. Agentic RAG is a worker who can look at the part coming down the belt and decide what to do with it.

QuestionTraditional RAGAgentic RAG
Should we retrieve at all?Always yesThe agent decides — it can answer simple questions directly
Which knowledge base?One, hard-codedThe agent picks the right tool from several stores
Are the chunks relevant?Never checkedA grading step scores them yes / no
What if retrieval is bad?Answer from bad context anywayRewrite the question and try again
Control flowLinear, one passA graph with branches and a loop
Cost / latencyLow — one LLM callHigher — extra calls to decide, grade, rewrite
When is Agentic RAG worth it?

The extra LLM calls (deciding, grading, rewriting) cost latency and tokens. Reach for Agentic RAG when queries are varied (some need retrieval, some don't), when you have multiple knowledge bases a single retriever can't cover, or when wrong answers are expensive and the self-check pays for itself. For a narrow FAQ bot over one tidy corpus, plain RAG is still the right call.

The Anatomy — Four Nodes, Two Decisions

Before any code, here's the full inventory of the graph. Notice that two of the six pieces aren't nodes at all — they're conditional edges, the functions that read the state and return the name of the next node. That's the single most important idea to hold onto: in LangGraph, a decision is just a function that returns a string.

PieceKindJobBuilt in
agentNodeThe LLM with retriever tools bound. Decides to call a retriever or answer directly.13.2
retrieveNode (ToolNode)Runs whichever retriever tool the agent requested and appends the chunks to the state.13.2
generateNodeWrites the final answer from the retrieved documents.13.2
rewriteNodeReformulates a weak question into a clearer one, then sends it back to the agent.13.2
tools_conditionConditional edgeAfter agent: tool call → retrieve, otherwise → END.13.2
grade_documentsConditional edgeAfter retrieve: relevant → generate, not relevant → rewrite.13.2
