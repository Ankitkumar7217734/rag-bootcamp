State Schemas — TypedDict, Dataclass & Pydantic

The state is the data every node reads and writes. LangGraph lets you describe its shape three different ways — and the one you pick decides how much safety you get.

Why the State Schema Matters

When you define a LangGraph, you give it a state schema: the structure and the types of the data your graph will carry. Every node communicates through this schema — it's the shared contract that lets one node's output become the next node's input. Get it right and the whole graph stays coherent; get it wrong and bad data silently flows from node to node.

LangGraph is deliberately flexible here. It offers three ways to define the schema, trading convenience against strictness:

#ApproachAccess styleValidation
1TypedDictstate["name"] (dict)Hints only — not enforced at runtime
2@dataclassstate.name (dot)Hints + defaults; stricter, still not runtime-validated
3Pydantic BaseModelstate.name (dot)Runtime validation — raises on bad types

To make the comparison fair, all three examples below build the exact same graph: a PlayGame node that flows, via a conditional edge, into either a cricket or a badminton node. Only the state definition changes.

Approach 1 — TypedDict (built-in Python)

As we saw in the parent chapter, TypedDict from Python's typing module lets you specify keys and their value types. But remember: these are only type hints. Static checkers like MyPy or your IDE can flag mismatches before you run, but they are not enforced at runtime — pass the wrong type and Python won't stop you.

from typing_extensions import TypedDict
from typing import Literal

class TypedDictState(TypedDict):
    name: str
    game: Literal["Cricket", "Badminton"]

Nodes read keys with dictionary syntax, state["name"]:

def play_game(state: TypedDictState):
    print("..Play Game node has been called.. ")
    return {"name": state["name"] + " want to play"}

def cricket(state: TypedDictState):
    print("..Cricket node has been called.. ")
    return {"game": "Cricket"}

def badminton(state: TypedDictState):
    print("..Badminton node has been called.. ")
    return {"game": "Badminton"}

The conditional edge returns the name of the next node:

import random

def decide_play(state: TypedDictState) -> Literal["cricket", "badminton"]:
    if random.random() < 0.5:
        return "cricket"
    else:
        return "badminton"

Wiring it up is the standard five-step recipe — add nodes, connect START, attach the conditional edge, route both leaves to END, compile:

from IPython.display import Image, display
from langgraph.graph import StateGraph, START, END

builder = StateGraph(TypedDictState)
builder.add_node("PlayGame", play_game)
builder.add_node("cricket", cricket)
builder.add_node("badminton", badminton)

builder.add_edge(START, "PlayGame")
builder.add_conditional_edges("PlayGame", decide_play)
builder.add_edge("cricket", END)
builder.add_edge("badminton", END)

graph = builder.compile()
display(Image(graph.get_graph(xray=True).draw_mermaid_png()))
graph.invoke({"name": "Ankit"})
..Play Game node has been called..
..Badminton node has been called..

{'name': 'Ankit want to play', 'game': 'Badminton'}

Approach 2 — Dataclass

Python's dataclasses provide a more structured, class-based alternative to TypedDict. They offer a concise syntax for classes whose main job is to hold data. The differences that matter for LangGraph:

vs TypedDictDataclass
Field accessDot notation — state.name instead of state["name"]
DefaultsSupported directly in the class definition (game: str = "")
MethodsYou can define methods on the state class
Type checkingStricter than TypedDict
 
from dataclasses import dataclass

@dataclass
class DataclassState:
    name: str
    game: str = ""  # default value supported in dataclass

The one change in the nodes is the access style — state.name, not state["name"]:

def play_game_dc(state: DataclassState):
    print("..Play Game node has been called.. ")
    # Dot notation access — state.name instead of state["name"]
    return {"name": state.name + " want to play"}

def cricket_dc(state: DataclassState):
    print("..Cricket node has been called.. ")
    return {"game": "Cricket"}

def badminton_dc(state: DataclassState):
    print("..Badminton node has been called.. ")
    return {"game": "Badminton"}

def decide_play_dc(state: DataclassState) -> Literal["cricket_dc", "badminton_dc"]:
    if random.random() < 0.5:
        return "cricket_dc"
    else:
        return "badminton_dc"

The graph is identical in shape; only the names differ:

dc_builder = StateGraph(DataclassState)
dc_builder.add_node("PlayGame", play_game_dc)
dc_builder.add_node("cricket_dc", cricket_dc)
dc_builder.add_node("badminton_dc", badminton_dc)

dc_builder.add_edge(START, "PlayGame")
dc_builder.add_conditional_edges("PlayGame", decide_play_dc)
dc_builder.add_edge("cricket_dc", END)
dc_builder.add_edge("badminton_dc", END)

dc_graph = dc_builder.compile()

Note the difference at invocation: a dataclass is instantiated, not passed as a dict:

dc_graph.invoke(DataclassState(name="Ankit"))
..Play Game node has been called..
..Badminton node has been called..

{'name': 'Ankit want to play', 'game': 'Badminton'}

Approach 3 — Pydantic BaseModel

Pydantic models are the strict option: they provide runtime validation. Unlike the first two approaches, a Pydantic model will raise an error if you pass the wrong type — the bad data never makes it into the graph.

When to reach for Pydantic

Use a Pydantic BaseModel when you need strict runtime type validation, want auto-coercion (e.g. the string "42" becomes the int 42), or you're building production pipelines where bad data could silently propagate and corrupt later steps. For quick prototypes, TypedDict is lighter.

from pydantic import BaseModel

class PydanticState(BaseModel):
    name: str
    game: Literal["Cricket", "Badminton"] = "Cricket"

Nodes use dot access, just like the dataclass:

def play_game_pd(state: PydanticState):
    print("..Play Game node has been called.. ")
    return {"name": state.name + " want to play"}

def cricket_pd(state: PydanticState):
    print("..Cricket node has been called.. ")
    return {"game": "Cricket"}

def badminton_pd(state: PydanticState):
    print("..Badminton node has been called.. ")
    return {"game": "Badminton"}

def decide_play_pd(state: PydanticState) -> Literal["cricket_pd", "badminton_pd"]:
    if random.random() < 0.5:
        return "cricket_pd"
    else:
        return "badminton_pd"
pd_builder = StateGraph(PydanticState)
pd_builder.add_node("PlayGame", play_game_pd)
pd_builder.add_node("cricket_pd", cricket_pd)
pd_builder.add_node("badminton_pd", badminton_pd)

pd_builder.add_edge(START, "PlayGame")
pd_builder.add_conditional_edges("PlayGame", decide_play_pd)
pd_builder.add_edge("cricket_pd", END)
pd_builder.add_edge("badminton_pd", END)

pd_graph = pd_builder.compile()

Invoke it with a model instance. This time the coin landed on cricket:

pd_graph.invoke(PydanticState(name="Ankit"))
..Play Game node has been called..
..Cricket node has been called..

{'name': 'Ankit want to play', 'game': 'Cricket'}

Seeing validation in action

The payoff of Pydantic is that an invalid game value (anything outside "Cricket" / "Badminton") is rejected before the graph ever runs — a ValidationError is raised at construction time, instead of a wrong value quietly flowing through nodes as it would with a TypedDict. That early failure is exactly what you want in production.

Choosing a Schema

Use…When…
TypedDictYou're prototyping, want the lightest syntax, and trust your inputs.
@dataclassYou like dot access and default values, and want a bit more structure without a dependency.
Pydantic BaseModelYou need runtime validation, type coercion, or you're shipping to production where bad data is costly.
Remember

The graph topology — nodes, edges, conditional routing — is identical across all three. The state schema only changes (a) how nodes read fields (state["x"] vs state.x), (b) how you invoke (dict vs class instance), and (c) how much validation you get. Start with TypedDict, reach for Pydantic when correctness matters.
