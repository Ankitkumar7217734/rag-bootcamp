Chapter 25: Graph Databases, Knowledge Graphs & Neo4j

Relational databases store facts in rows. Graph databases store the connections between facts as first-class data — making questions about paths, neighborhoods, influence, ownership, and multi-hop relationships natural to ask and fast to explore.

Graph Databases in One Paragraph

A graph database represents the world as nodes connected by named relationships, with labels and key-value properties adding meaning. A knowledge graph uses that structure as a semantic network of people, places, organizations, events, and concepts. Neo4j stores this model directly; Cypher describes the pattern you want to find. For RAG, the result is retrieval based on explicit connections and multi-hop paths, not only similarity between isolated chunks.

Overview

The supplied notes move through three layers. First comes the knowledge graph: a semantic network built from real-world entities and the relationships between them. Next comes the Neo4j property-graph database, which stores nodes, directed relationships, labels, and properties while supporting ACID transactions. Finally comes the application layer: Cypher retrieval, Graph RAG, and LangChain or LangGraph agents that reason over connected evidence.

Knowledge Graphs: Meaning as a Network

A knowledge graph is a semantic network: it represents real-world entities — such as people, teams, products, places, events, situations, and concepts — together with the meaning of their connections. The graph does not merely record that two values coexist. It names why they are connected: captain of, plays for, owns, located in, or depends on.

Natural-language factSubjectRelationshipObject
Rohit Sharma is captain of the Indian cricket teamRohit SharmaCAPTAIN_OFIndian Cricket Team
Virat Kohli plays for the Indian cricket teamVirat KohliPLAYS_FORIndian Cricket Team
Rohit Sharma and Virat Kohli are teammatesRohit SharmaTEAMMATE_OFVirat Kohli
The Semantic Difference

A search index may tell you that “Rohit Sharma” and “Indian cricket team” occur near one another. A knowledge graph records the explicit claim that Rohit Sharma CAPTAIN_OF that team. Similarity suggests association; a graph stores a typed relationship that can be traversed and checked.

The Four Pieces of a Property Graph

The notes first identify nodes, edges, and labels, then refine the Neo4j model into nodes, relationships, and properties. Together they form the practical property-graph vocabulary:

ElementPurposeCricket example
NodeA distinct entity: a person, place, object, event, or conceptRohit Sharma
LabelA category attached to a node; one node can have multiple labelsPerson, Player
RelationshipA named, directed connection between two nodesCAPTAIN_OF
PropertyA key-value detail stored on a node or relationshipname: Rohit Sharma; since: 2024

Relational Databases vs Graph Databases

Relational database management systems such as MySQL and SQL Server organize data into tables. Records become rows, attributes become columns, and primary or foreign keys preserve structure. To answer a connected question, the query joins rows across tables. A graph database stores the same entities as nodes and the key-based connection as a direct relationship, then follows that path through traversal.

DimensionRelational DBMSGraph Database
Primary structureTables, rows, columnsNodes, relationships, properties
Connection mechanismPrimary and foreign keysNamed relationships
Connected queriesJoin tables at query timeTraverse stored paths
SchemaUsually declared before insertFlexible labels and properties; constraints can still be enforced
Natural strengthsTransactions, aggregation, tabular reportingNetworks, hierarchies, paths, recommendations, multi-hop questions
Typical examplesOrders, ledgers, inventory, payrollFraud rings, social networks, dependencies, knowledge graphs
“No Joins” Needs Precision

The notes summarize Neo4j as “no joins.” The practical meaning is that connected queries do not reconstruct every relationship from matching foreign-key values. The graph stores adjacency directly and traverses it. That does not make traversal free, nor does it make relational databases obsolete; it removes join-centric modeling from domains whose main questions are already about relationships.

Neo4j and the Property-Graph Model

Neo4j is a graph database built around the property-graph model. Data is stored and managed as labeled nodes, directed relationships, and properties on both. Relationships are always physically directed from a start node to an end node, even when an application chooses to match them in either direction.

Flexible Schema Does Not Mean No Governance

Neo4j allows different nodes with the same label to carry different properties, and new relationship types can be introduced as the domain evolves. This flexibility is valuable when connected knowledge changes faster than a rigid table design. Production graphs should still use naming conventions, uniqueness constraints, indexes, required identifiers, and data-quality checks. Flexibility removes unnecessary rigidity; it does not remove responsibility.

ACID Still Matters

Graph structure does not trade away transactional guarantees. Neo4j supports the familiar ACID model:

PropertyMeaning for graph updates
AtomicityA multi-node, multi-relationship update succeeds completely or rolls back
ConsistencyConstraints and transaction rules remain valid before and after the update
IsolationConcurrent operations do not expose incomplete intermediate graph states
DurabilityCommitted nodes and relationships survive failure and restart

Cypher: Query by Describing a Pattern

Cypher is Neo4j's declarative graph query language. “Declarative” means you describe the graph pattern and result you want; the database planner decides how to find it. Its notation mirrors the whiteboard model: parentheses represent nodes, bracketed arrows represent relationships, labels constrain entity categories, and properties constrain details. This chapter stays code-free, so the important lesson is the visual grammar rather than command syntax.

Visual ideaWhat Cypher expressesQuestion it supports
NodeAn entity, optionally restricted by label or propertiesFind a Player named Virat Kohli
Directed relationshipA named connection from one node to anotherWhich team does this player play for?
Undirected matchA relationship may be matched without caring about stored directionWho is connected as a teammate?
PathSeveral connected relationships matched togetherWhich players share a team with this captain?
Property filterOnly entities or relationships with matching key-value detailsWho has captained the team since 2024?
Think in Shapes

Before writing a graph query, sketch the answer shape: Person → relationship → Team, or Person → Team ← Person. The sketch identifies the node labels, relationship types, direction, filters, and number of hops that the query must express. Cypher is close to that sketch by design.

Why Graph Traversal Changes Retrieval

Graph retrieval follows relationships from a known or discovered starting node. A one-hop traversal asks about immediate neighbors; a multi-hop traversal asks how entities connect through intermediate nodes. This is valuable whenever the explanation is a path rather than a single matching record.

Hop countCricket questionGraph path
1 hopWhich team does Virat Kohli play for?Virat Kohli → PLAYS_FOR → Indian Cricket Team
2 hopsWho captains Virat Kohli's team?Virat Kohli → Team ← CAPTAIN_OF ← Rohit Sharma
3 hopsWhich players share a team led by the same captain?Player → Team ← Captain, then Team ← Player

The database can return both the answer entities and the path that connects them. That path is useful evidence: an application can explain not only what it found, but why the result follows from stored relationships.

Graph RAG With LangChain and LangGraph

Traditional vector RAG retrieves text chunks that are semantically similar to a question. Graph RAG retrieves entities, relationships, neighborhoods, and paths that satisfy a structured meaning. The two approaches are complementary: vector search is strong for fuzzy language and broad discovery, while graph traversal is strong for explicit relationships, constraints, and multi-hop evidence.

When It Is Probably Not the First Choice

A graph database is not automatically better for every dataset. A simple ledger, flat event stream, columnar analytics workload, or document archive may be better served by relational, time-series, analytical, or document storage. Use a graph when relationships are central to both the domain and the questions — especially when those questions change or span several hops.

Decision questionIf the answer is “yes”
Are relationships as important as the entities themselves?A property graph deserves serious consideration
Do queries repeatedly cross several foreign-key relationships?Traversal may simplify the model and query shape
Must the system return an explanation path?Graph paths provide natural provenance
Does the domain evolve with new entity and relationship types?Flexible graph modeling may reduce migration friction
Are most workloads simple aggregates over stable tables?Keep the relational model unless another requirement dominates

Designing a Useful Knowledge Graph

The hardest part is rarely drawing circles and arrows. It is choosing stable entity boundaries, meaningful relationship types, and properties that support real questions without copying every source sentence into the graph.

Design stepQuestion to askCommon failure
Start from questionsWhich decisions or paths must the application retrieve?Modeling the source format instead of the use case
Choose node identitiesWhat counts as the same person, team, product, or concept?Duplicate entities caused by spelling or aliases
Name relationshipsWhat precise verb connects these node types?Generic CONNECTED_TO edges with no semantic value
Choose directionWhich direction expresses the fact most naturally?Inconsistent inverse relationships
Place propertiesDoes this detail describe the entity or the connection?Putting “since” on a person instead of on CAPTAIN_OF
Preserve provenanceWhere did this node or relationship fact come from?Answers that cannot cite or verify their source
Constrain identifiersWhich IDs or names must be unique?A flexible graph that slowly accumulates contradictions
Model Relationships as Facts

If a connection has its own time, confidence, role, source, or status, it belongs as a relationship with properties — not as a text field buried inside one node. That decision is what makes the graph traversable, explainable, and useful to RAG.

Key Takeaways

Remember These Five

1. A knowledge graph is a semantic network. It models real-world entities and names the relationships between them, turning language such as “Rohit captains India” into a connected fact that can be traversed.

2. Neo4j uses the property-graph model. Nodes carry labels and properties; directed relationships connect a start node to an end node and can carry properties of their own.

3. Graph traversal replaces relationship reconstruction. Relational databases connect tables through keys and joins; a graph stores adjacency directly and follows paths. Choose according to the shape of the domain and its questions.

4. Cypher is declarative and visual. Think in node-relationship-node patterns, filters, direction, and path length. Describe the shape you need; let Neo4j plan the traversal.

5. Graph RAG retrieves connected evidence. LangChain or LangGraph can translate intent into graph retrieval, collect paths from Neo4j, and give the LLM both facts and provenance — often alongside vector search in a hybrid design.
