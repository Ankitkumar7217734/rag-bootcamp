Chapter 2: Core Components of RAG

A deep dive into the three phases of RAG — ingestion, retrieval, and generation — and the tools that power each one.

Overview

In Chapter 1 we saw what RAG is and why it matters. Now we break open the architecture and examine each component in detail. RAG is a pipeline with three distinct phases. Understanding each phase — and the decisions made within it — is what separates a working RAG system from a great one.

Phase 1: Document Ingestion & Preprocessing

This is the offline phase — it runs before any user ever asks a question. Its job is to take raw documents and transform them into a format that can be searched by meaning, not just keywords.

Step 1 — Load Your Data Sources

RAG can ingest virtually any structured or unstructured data source:

PDFs / DocsAnnual reports, manuals, policiesEnterprise knowledge bases
Web PagesProduct docs, Wikipedia, blogsResearch assistants

CSV / SpreadsheetsInventory, pricing, schedulesCustomer support, e-commerce

Step 2 — Splitting Documents into Chunks

An LLM has a fixed context size — the maximum amount of text it can read at once. A 500-page PDF cannot be sent whole. The Document Splitter breaks it into smaller, overlapping pieces called chunks.

Why Overlap Matters

If a chunk ends at page 10 and the next starts at page 11, a sentence that spans the page boundary gets cut in half. Overlap (e.g., pages 9–19 instead of 11–20) ensures no meaning is lost at the seams. A typical overlap is 10–20% of the chunk size.

Step 3 — Generating Embeddings

An embedding model converts each chunk of text into a vector — a list of numbers that encodes the semantic meaning of that text. Chunks with similar meaning end up with numerically similar vectors, even if they use completely different words.

[0.6, 0.5, 0.4, 0.1, 0.7, ...]   ← "What is a neural network?"
[0.6, 0.5, 0.3, 0.2, 0.6, ...]   ← "Explain deep learning models"
[0.1, 0.9, 0.2, 0.8, 0.1, ...]   ← "Best recipe for chocolate cake"

The first two sentences are semantically close (high vector similarity); the third is unrelated (low similarity). Popular embedding models include:

Step 4 — Storing in a Vector Database

The generated vectors (plus the original text and metadata) are stored in a Vector Database. This is a specialized database optimized to perform fast similarity searches across millions of vectors.

Vector DBTypeBest For
FAISSIn-memory libraryLocal prototyping, speed
ChromaDBOpen-source, embeddedSmall-to-mid scale, fast setup
PineconeManaged cloud serviceProduction at scale
DataStax (Astra)Managed (Cassandra-based)Enterprise, hybrid workloads
WeaviateOpen-source / cloudMulti-modal (text + images)

Each stored entry looks something like this:

{
  "id": "doc_42_chunk_3",
  "vector": [0.6, 0.5, 0.4, ...],
  "text": "Revenue increased by 10% in Q3...",
  "metadata": {
    "source": "Tesla_Annual_Report_2023.pdf",
    "page": 42,
    "date": "2023-10-15"
  }
}

The metadata field is critical for citations — it tells the LLM where the retrieved fact came from, enabling transparent, auditable answers.

Ingestion Phase — Full Flow Summary

Key Insight

The ingestion phase runs once (or periodically when docs change). Every user query then benefits from the pre-built index — no re-processing needed. This is what makes RAG fast at query time.
