Chapter 17: Corrective RAG (CRAG) — Grading Retrieval & Self-Correcting

Don't trust retrieval blindly. Grade every document, and when the store comes up short, rewrite the question and go to the web before you answer.

A More Detailed Explanation

Addressing Limitations of Basic RAG

Traditional RAG systems rely heavily on the accuracy of retrieved documents. If the retrieved information is flawed or incomplete, the generated response can also be inaccurate.

CRAG's Core Components

1. Retrieval Evaluator: This component assesses the quality and relevance of retrieved documents.

2. Generative Model: This model generates the initial response based on the retrieved information.

3. Refinement and Correction: CRAG employs strategies like knowledge refinement or web search to address issues identified by the retrieval evaluator.

Benefits of CRAG

1. Improved Accuracy: By evaluating and correcting retrieved information, CRAG helps ensure the accuracy of generated responses.

2. Enhanced Relevance: CRAG can identify and filter out irrelevant information, making the generated response more relevant.

3. Increased Robustness: CRAG can handle cases where the initial retrieval process is not perfect, leading to a more robust RAG system.

Overview

Every RAG system you've built so far shares one silent assumption: the documents the retriever hands back are good enough to answer from. Chapter 13 poked a hole in that by grading documents and looping back to rewrite. Corrective RAG (CRAG) turns that instinct into a full, self-contained workflow.

CRAG is an advanced RAG technique that improves the accuracy and relevance of generated answers by adding self-reflection and self-grading of retrieved documents. It evaluates the quality of what the retriever returned and applies a corrective action when the retrieval is weak — refining the query and falling back to web search — rather than generating an answer from irrelevant context.

This chapter builds the complete CRAG graph from 2-CorrectiveRAG.ipynb: a retrieval grader with structured output, a RAG generator, a question re-writer tuned for web search, a Tavily search tool, and a LangGraph StateGraph that wires them into a retrieve → grade → (generate | rewrite → search → generate) loop.

The Core Idea

Basic RAG answers from whatever the retriever returns. CRAG adds a grader between retrieval and generation. If every document is relevant, it generates the answer directly. If any document is irrelevant, it triggers a corrective action: re-write the question into a web-search-friendly form, run a Tavily search, append those results to the context, and only then generate. The system reflects on its own retrieval quality and fixes it before answering.

Step 1 — Build the Index

We index three of Lilian Weng's blog posts (agents, prompt engineering, adversarial attacks) into a FAISS store. Documents are loaded with WebBaseLoader, split with a tiktoken-aware RecursiveCharacterTextSplitter (500-token chunks, no overlap), embedded with OpenAI embeddings, and exposed as a retriever. 

### Build Index

from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.document_loaders import WebBaseLoader
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings

# Set embeddings
embd = OpenAIEmbeddings()

# Docs to index
urls = [
    "https://lilianweng.github.io/posts/2023-06-23-agent/",
    "https://lilianweng.github.io/posts/2023-03-15-prompt-engineering/",
    "https://lilianweng.github.io/posts/2023-10-25-adv-attack-llm/",
]

# Load
docs = [WebBaseLoader(url).load() for url in urls]
docs_list = [item for sublist in docs for item in sublist]

# Split
text_splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
    chunk_size=500, chunk_overlap=0
)
doc_splits = text_splitter.split_documents(docs_list)

# Add to vectorstore
vectorstore = FAISS.from_documents(
    documents=doc_splits,
    embedding=OpenAIEmbeddings()
)

retriever = vectorstore.as_retriever()

This is the same knowledge base pattern from earlier chapters — the interesting part isn't the store, it's what we do after retrieval.

Step 2 — The Retrieval Grader

This is the heart of CRAG. We ask an LLM to return a binary relevance score for a single (question, document) pair. Using with_structured_output against a Pydantic model forces the answer into a clean 'yes'/'no' field — no parsing free-form text. 

### Retrieval Grader

from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


# Data model
class GradeDocuments(BaseModel):
    """Binary score for relevance check on retrieved documents."""

    binary_score: str = Field(
        description="Documents are relevant to the question, 'yes' or 'no'"
    )

# LLM with function call
llm = ChatOpenAI(model="gpt-3.5-turbo-0125", temperature=0)
structured_llm_grader = llm.with_structured_output(GradeDocuments)

# Prompt
system = """You are a grader assessing relevance of a retrieved document to a user question. \n
    If the document contains keyword(s) or semantic meaning related to the question, grade it as relevant. \n
    Give a binary score 'yes' or 'no' score to indicate whether the document is relevant to the question."""
grade_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", system),
        ("human", "Retrieved document: \n\n {document} \n\n User question: {question}"),
    ]
)

# chain the prompt with the LLM
retrieval_grader = grade_prompt | structured_llm_grader
question = "agent memory"
docs = retriever.invoke(question)
doc_txt = docs[1].page_content
print(retrieval_grader.invoke({"question": question, "document": doc_txt}))
binary_score='no'
Why binary, not a score?

A binary yes/no is deliberately coarse. CRAG doesn't need a precise relevance ranking — it needs a trigger. One irrelevant document is enough to flip the whole flow into corrective mode, so a crisp binary decision per document is exactly the right signal. (Note the grader is evaluated on docs[1] here, and that particular chunk grades as 'no'.)

Step 3 — The Generator

A standard concise RAG chain: stuff the retrieved documents into a prompt and ask for a three-sentence answer. This is the inline equivalent of the classic hub.pull("rlm/rag-prompt"). 

### Generate

from langchain_core.output_parsers import StrOutputParser
from langchain_core.prompts import ChatPromptTemplate

# Prompt (inline equivalent of the old hub.pull("rlm/rag-prompt"))
prompt = ChatPromptTemplate.from_messages(
    [
        (
            "human",
            "You are an assistant for question-answering tasks. Use the following pieces "
            "of retrieved context to answer the question. If you don't know the answer, "
            "just say that you don't know. Use three sentences maximum and keep the answer "
            "concise.\n"
            "Question: {question} \n"
            "Context: {context} \n"
            "Answer:",
        ),
    ]
)

# LLM
llm = ChatOpenAI(model_name="gpt-3.5-turbo", temperature=0)


# Post-processing
def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)


# Chain
rag_chain = prompt | llm | StrOutputParser()

# Run
generation = rag_chain.invoke({"context": docs, "question": question})
print(generation)
Agent memory in a LLM-powered autonomous agent system consists of short-term memory for in-context
learning and long-term memory for retaining and recalling information over extended periods. The
long-term memory allows the agent to leverage an external vector store for fast retrieval of
information. Memory plays a crucial role in enabling agents to learn from past experiences and
improve their future actions.

Step 4 — The Question Re-Writer

When retrieval fails, the corrective action starts by improving the question. The re-writer reasons about the underlying semantic intent and rewrites the query into a form that's optimized for web search — often expanding a terse phrase like “agent memory” into a full natural-language question. 

### Question Re-writer

# LLM
llm = ChatOpenAI(model="gpt-3.5-turbo-0125", temperature=0)

# Prompt
system = """You a question re-writer that converts an input question to a better version that is optimized \n
     for web search. Look at the input and try to reason about the underlying semantic intent / meaning."""
re_write_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", system),
        (
            "human",
            "Here is the initial question: \n\n {question} \n Formulate an improved question.",
        ),
    ]
)

question_rewriter = re_write_prompt | llm | StrOutputParser()
question_rewriter.invoke({"question": question})
'What is the role of memory in artificial intelligence agents?'

Step 5 — The Web Search Tool

The fallback knowledge source is Tavily, a search API built for LLMs. It returns the top results, which the web-search node will fold into the document list. 

### Search

from langchain_community.tools.tavily_search import TavilySearchResults

web_search_tool = TavilySearchResults(k=3)

Step 6 — Graph State

The whole workflow shares one TypedDict state. Each node reads some keys and returns updated ones. The web_search flag is the switch the grader flips and the conditional edge reads. 

from typing import List
from typing_extensions import TypedDict


class GraphState(TypedDict):
    """
    Represents the state of our graph.

    Attributes:
        question: question
        generation: LLM generation
        web_search: whether to add search
        documents: list of documents
    """

    question: str
    generation: str
    web_search: str
    documents: List[str]

Step 7 — The Graph Nodes

The workflow is five node functions plus one decision function. Every node follows the same contract: it receives the current state dict, does one job, and returns only the keys it changed. We'll build them one at a time instead of all at once — each is small once you see it in isolation. They share a single import, LangGraph's Document, used when we wrap web results back into the document list: 

from langchain_classic.schema import Document

retrieve & generate — the two endpoints

These are the bookends of the graph. retrieve is the ordinary RAG lookup: take the question, hit the FAISS retriever, drop the documents into state. generate is the final step: feed the (possibly corrected) documents and question to the rag_chain from Step 3 and store the answer under generation. Neither makes a decision — they just move data through. 

def retrieve(state):
    """Retrieve documents."""
    print("---RETRIEVE---")
    question = state["question"]
    documents = retriever.invoke(question)
    return {"documents": documents, "question": question}


def generate(state):
    """Generate answer."""
    print("---GENERATE---")
    question = state["question"]
    documents = state["documents"]
    generation = rag_chain.invoke({"context": documents, "question": question})
    return {"documents": documents, "question": question, "generation": generation}

grade_documents — the quality gate

This is the node that makes CRAG “corrective.” It loops over every retrieved document and runs the retrieval_grader from Step 2 on each one. Relevant documents are kept in filtered_docs; irrelevant ones are dropped. The important line is in the else branch: the first document that grades no flips web_search from "No" to "Yes" — and that flag is the entire trigger for the corrective path. 

def grade_documents(state):
    """Determine whether the retrieved documents are relevant to the question."""
    print("---CHECK DOCUMENT RELEVANCE TO QUESTION---")
    question = state["question"]
    documents = state["documents"]

    # Score each doc
    filtered_docs = []
    web_search = "No"
    for d in documents:
        score = retrieval_grader.invoke(
            {"question": question, "document": d.page_content}
        )
        grade = score.binary_score
        if grade == "yes":
            print("---GRADE: DOCUMENT RELEVANT---")
            filtered_docs.append(d)
        else:
            print("---GRADE: DOCUMENT NOT RELEVANT---")
            web_search = "Yes"
            continue
    return {"documents": filtered_docs, "question": question, "web_search": web_search}
The one subtle rule

grade_documents keeps only the relevant documents in filtered_docs, and flips web_search to "Yes" if any document failed. So even when the store had some good chunks, one bad chunk still sends the flow through web search — CRAG treats partial relevance as a reason to enrich the context, not skip it.

transform_query & web_search — the corrective action

These two only run when grading failed. transform_query rewrites the question into a web-search-friendly form using the re-writer from Step 4. web_search then sends that improved question to Tavily, joins the results into a single Document, and appends it to the surviving documents — so generate later sees the good local chunks plus fresh web context, not one or the other. 

def transform_query(state):
    """Transform the query to produce a better question."""
    print("---TRANSFORM QUERY---")
    question = state["question"]
    documents = state["documents"]
    better_question = question_rewriter.invoke({"question": question})
    return {"documents": documents, "question": better_question}


def web_search(state):
    """Web search based on the re-phrased question."""
    print("---WEB SEARCH---")
    question = state["question"]
    documents = state["documents"]
    docs = web_search_tool.invoke({"query": question})
    web_results = "\n".join([d["content"] for d in docs])
    web_results = Document(page_content=web_results)
    documents.append(web_results)
    return {"documents": documents, "question": question}

decide_to_generate — the conditional edge

This one isn't a node — it's the router. After grading, it reads the web_search flag and returns the name of the next node to run: "generate" if every document was relevant, or "transform_query" if any wasn't. That returned string is what the graph's conditional edge in Step 8 maps to an actual node. 

### Edges


def decide_to_generate(state):
    """Decide whether to generate an answer, or re-generate a question."""
    print("---ASSESS GRADED DOCUMENTS---")
    state["question"]
    web_search = state["web_search"]
    state["documents"]

    if web_search == "Yes":
        # All documents have been filtered check_relevance
        # We will re-generate a new query
        print("---DECISION: ALL DOCUMENTS ARE NOT RELEVANT TO QUESTION, TRANSFORM QUERY---")
        return "transform_query"
    else:
        # We have relevant documents, so generate answer
        print("---DECISION: GENERATE---")
        return "generate"

Step 8 — Wiring the Graph

Now assemble the StateGraph. The only conditional edge is out of grade_documents; everything else is a straight line. Notice the two paths reconverge at generate: the “good retrieval” path jumps straight there, while the “corrective” path detours through transform_query → web_search_node first. 

from langgraph.graph import END, StateGraph, START

workflow = StateGraph(GraphState)

# Define the nodes
workflow.add_node("retrieve", retrieve)                # retrieve
workflow.add_node("grade_documents", grade_documents)  # grade documents
workflow.add_node("generate", generate)                # generate
workflow.add_node("transform_query", transform_query)  # transform_query
workflow.add_node("web_search_node", web_search)       # web search

# Build graph
workflow.add_edge(START, "retrieve")
workflow.add_edge("retrieve", "grade_documents")
workflow.add_conditional_edges(
    "grade_documents",
    decide_to_generate,
    {
        "transform_query": "transform_query",
        "generate": "generate",
    },
)
workflow.add_edge("transform_query", "web_search_node")
workflow.add_edge("web_search_node", "generate")
workflow.add_edge("generate", END)

# Compile
app = workflow.compile()

This is what app.get_graph(xray=True).draw_mermaid_png() renders — the compiled CRAG graph. The dotted edges out of grade_documents are the conditional branch: one dashes straight down to generate (relevant docs), the other detours through transform_query → web_search_node (corrective path) before rejoining.

CRAG vs Agentic RAG

Chapter 13's Agentic RAG also graded documents and could rewrite. So what's different? The key distinction is where the correction goes.

AspectAgentic RAG (Ch. 13)Corrective RAG (Ch. 17)
Who decides to retrieve The LLM agent, via tool calls A fixed graph edge — retrieval always happens first
Grading Relevance grade on retrieved docs Per-document binary grade, filters the doc list
Correction on failure Rewrite the question & retrieve from the same store again Rewrite & fall back to web search (new knowledge source)
Control flow Agent loop (model-driven) Deterministic graph (edge-driven)
Best when Multiple stores, model should choose One store that may not cover everything — web fills the gap

Key Takeaways

Remember These Four

1. Grade before you generate. CRAG inserts a retrieval evaluator between retrieval and generation — a binary yes/no relevance check per document — so the LLM never answers from context it hasn't vetted.

2. Any irrelevant document triggers correction. The grader keeps the good chunks but flips web_search = "Yes" on the first bad one, routing the flow through the corrective branch.

3. Correction = re-write + web search. The corrective action improves the question for web search, runs Tavily, and appends those results to the surviving documents — enriching context instead of answering from a thin retrieval.

4. It's a deterministic graph, not an agent loop. One conditional edge out of grade_documents decides between generate and transform_query → web_search_node → generate. The two paths always reconverge at generate.
