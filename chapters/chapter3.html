Chapter 3: Data Ingestion & Parsing

How raw files — text, PDFs, Word docs, spreadsheets, JSON, and databases — become clean, searchable Document objects ready for a RAG pipeline.

Overview

Before RAG can retrieve anything, it first has to ingest your data. Ingestion is the very first stage of the pipeline: take messy real-world files in every shape and format, and turn them into a single, uniform structure the rest of the system can work with.

That uniform structure in LangChain is the Document. No matter whether the source is a 19-page research PDF, a CSV of products, or a SQL table, the output is always the same: a list of Document objects, each carrying its text and metadata. This chapter walks through how to load and parse every common data source.

The Golden Rule

Garbage in, garbage out. A RAG system can only retrieve what was ingested well. If parsing drops a table, mangles a paragraph, or loses the source, every answer downstream inherits that flaw. Ingestion is where quality is won or lost.

The Document — RAG's Universal Container

Every loader in LangChain returns the same object: a Document. It has just two parts:

  • page_content — the actual text (a string).
  • metadata — a dictionary of extra info: source file, page number, author, date, or anything you add.
from langchain_core.documents import Document

doc = Document(
    page_content="This is a simple document. It contains some text "
                 "to demonstrate the Document class in LangChain.",
    metadata={
        "source": "example.txt",
        "page": 1,
        "author": "John Doe",
        "date": "2026-05-20",
        "custom_field": "custom_value",
    },
)

print(doc.page_content)
print(doc.metadata)

Metadata is not optional decoration — it is what lets the LLM cite its sources later. A retrieved chunk that knows it came from page 42 of the Tesla report produces a trustworthy, auditable answer.

Text Files (.txt)

The simplest source. TextLoader reads one file into one Document. The file path is automatically captured in metadata["source"].

from langchain_community.document_loaders import TextLoader

loader = TextLoader("data/text_files/coalindia_notice.txt", encoding="utf-8")
documents = loader.load()

print(f"Loaded {len(documents)} documents.")
print(documents[0].page_content[:100])
print(documents[0].metadata)
# {'source': 'data/text_files/coalindia_notice.txt'}

Loading a Whole Folder

When you have many files, DirectoryLoader loads them all at once using a glob pattern.

from langchain_community.document_loaders import DirectoryLoader, TextLoader

directory_loader = DirectoryLoader(
    "data/text_files",
    glob="**/*.txt",            # every .txt file, recursively
    loader_cls=TextLoader,
    show_progress=True,
    loader_kwargs={"encoding": "utf-8"},
)
documents = directory_loader.load()

Text Splitting Strategies

An LLM (and an embedding model) can only read a limited amount of text at once. A long document must be broken into smaller chunks. How you split matters — split badly and you slice sentences in half; split well and each chunk stays meaningful. There are three core strategies.

1 — CharacterTextSplitter

Splits on a single separator (e.g. a newline) and counts characters. Simple, but it will not break a long line that has no separator — so chunks can overflow your target size.

from langchain_classic.text_splitter import CharacterTextSplitter

splitter = CharacterTextSplitter(
    chunk_size=100,
    chunk_overlap=20,
    separator="\n",
)
chunks = splitter.split_documents(documents)

2 — RecursiveCharacterTextSplitter (recommended)

The workhorse of RAG. It tries a list of separators in order — paragraphs, then lines, then spaces, then characters — falling back until each chunk fits. This keeps natural units (paragraphs, sentences) together whenever possible.

from langchain_classic.text_splitter import RecursiveCharacterTextSplitter

text = ("This is a long text without any natural breakpoints. It will be "
        "split into smaller chunks based on the specified chunk size and overlap.")

splitter = RecursiveCharacterTextSplitter(
    chunk_size=50,
    chunk_overlap=10,
    separators=[""],     # fall all the way back to character-level
)
chunks = splitter.split_text(text)
for c in chunks:
    print(c)

Output — notice the overlap: the end of one chunk repeats at the start of the next, so meaning is never lost at the seam:

chunk 1: 'This is a long text without any natural breakpoint'
chunk 2: 'breakpoints. It will be split into smaller chunks'
chunk 3: 'er chunks based on the specified chunk size and ov'
chunk 4: 'ize and overlap.'

3 — TokenTextSplitter

Splits by tokens (the units the model actually counts) instead of characters. This is the most accurate way to respect a model's real context limit.

from langchain_classic.text_splitter import TokenTextSplitter

token_splitter = TokenTextSplitter(chunk_size=10, chunk_overlap=2)
token_chunks = token_splitter.split_text(text)
Which Should I Use?

RecursiveCharacterTextSplitter is the sensible default for almost all text and PDFs. Reach for TokenTextSplitter when you need to pack chunks right up to a model's token limit, and CharacterTextSplitter only when your data has one clean, reliable separator.

PDF Documents

PDFs are the hardest common format. They store text in a complex layout (not a simple stream), may contain scanned images that need OCR, and often leave behind extraction artifacts. LangChain offers several loaders with different trade-offs.

from langchain_community.document_loaders import (
    PyPDFLoader,        # fast, simple, page-by-page
    PyMuPDFLoader,      # accurate, rich metadata, better layouts
    UnstructuredPDFLoader,  # element-aware, slowest
)

loader = PyMuPDFLoader("data/pdf/RAG_Research paper.pdf")
pages = loader.load()
print(f"Loaded {len(pages)} pages")
print(pages[0].page_content[:300])

Cleaning Extraction Artifacts

Raw PDF text is full of stray whitespace, broken ligatures (ﬁ → fi), and page numbers. A small cleaning function fixes the most common issues.

import re

def clean_pdf_text(raw_text):
    # 1. Collapse excessive whitespace and newlines
    cleaned = " ".join(raw_text.split())
    # 2. Fix broken ligatures
    cleaned = cleaned.replace("ﬁ", "fi").replace("ﬂ", "fl")
    # 3. Remove "page 1 of 10" style footers
    cleaned = re.sub(r"page \d+ of \d+", "", cleaned, flags=re.IGNORECASE)
    return cleaned

A Production-Grade PDF Processor

In practice you combine three things: an accurate loader, the cleaning step, and a recursive splitter — plus enriched metadata on every chunk (page number, total pages, character count). This class does all of it.

from typing import List
from langchain_core.documents import Document
from langchain_community.document_loaders import PyMuPDFLoader
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter

class SmartPDFProcessor:
    """Accurate loading + cleaning + smart chunking with rich metadata."""

    def __init__(self, chunk_size=1000, chunk_overlap=100):
        self.text_splitter = RecursiveCharacterTextSplitter(
            chunk_size=chunk_size,
            chunk_overlap=chunk_overlap,
            separators=[" "],
        )

    def process_pdf(self, pdf_path: str) -> List[Document]:
        pages = PyMuPDFLoader(pdf_path).load()
        total_pages = len(pages)
        processed_chunks = []

        for page_num, page in enumerate(pages):
            cleaned_text = self._clean_text(page.page_content)
            if not cleaned_text.strip():        # skip blank pages
                continue

            chunks = self.text_splitter.create_documents(
                texts=[cleaned_text],
                metadatas=[{
                    **page.metadata,
                    "page": page_num + 1,
                    "total_pages": total_pages,
                    "chunk_method": "smart_pdf_processor",
                    "char_count": len(cleaned_text),
                }],
            )
            processed_chunks.extend(chunks)
        return processed_chunks

    def _clean_text(self, raw_text: str) -> str:
        cleaned = " ".join(raw_text.split())
        cleaned = cleaned.replace("ﬁ", "fi").replace("ﬂ", "fl")
        return re.sub(r"page \d+ of \d+", "", cleaned, flags=re.IGNORECASE)

# Usage
processor = SmartPDFProcessor()
docs = processor.process_pdf("data/pdf/RAG_Research paper.pdf")
print(f"Processed PDF into {len(docs)} chunks.")   # → 85 chunks
Example Output Metadata

Every resulting chunk carries traceable context, e.g.:

source: RAG_Research paper.pdf · page: 1 · total_pages: 19 · char_count: 2903 · chunk_method: smart_pdf_processor

Microsoft Word Documents (.docx)

Word files have two loaders, and the choice depends on whether you want plain text or structure.

Method 1 — Docx2txtLoader (whole document)

Returns the entire document as one clean text blob. Fast and simple.

from langchain_community.document_loaders import Docx2txtLoader

loader = Docx2txtLoader("data/word_files/proposal.docx")
documents = loader.load()
print(documents[0].page_content[:200])
# Project Proposal: RAG Implementation
# Executive Summary
# This proposal outlines the implementation of a ...

Method 2 — UnstructuredWordDocumentLoader (by element)

With mode="elements", each heading, paragraph, list item, and table becomes its own Document with a category in its metadata (Title, NarrativeText, ListItem, Table). This structural awareness is great for precise retrieval.

from langchain_community.document_loaders import UnstructuredWordDocumentLoader

loader = UnstructuredWordDocumentLoader("data/word_files/proposal.docx", mode="elements")
docs = loader.load()

for d in docs[:4]:
    print(d.metadata["category"], "→", d.page_content)
# Title          → Project Proposal: RAG Implementation
# Title          → Executive Summary
# NarrativeText  → This proposal outlines the implementation of a ...
# Title          → Objectives

Tables are even preserved as HTML in the metadata, so a row like "Research · 2 weeks · Technology evaluation report" keeps its structure instead of collapsing into a wall of words.

Method 2 — Intelligent Processing (richer control)

Often you want a nicely formatted sentence-like chunk and structured metadata you can filter on later. Build the Document yourself with pandas.

import pandas as pd
from typing import List
from langchain_core.documents import Document

def process_csv_intelligently(filepath: str) -> List[Document]:
    df = pd.read_csv(filepath)
    documents = []
    for idx, row in df.iterrows():
        content = (
            f"Product Information:\n"
            f"  Name: {row['Product']}\n"
            f"  Category: {row['Category']}\n"
            f"  Price: ${row['Price']}\n"
            f"  Stock: {row['Stock']}\n"
            f"  Description: {row['Description']}"
        )
        documents.append(Document(
            page_content=content,
            metadata={
                "source": filepath,
                "row_index": idx,
                "product_name": row["Product"],
                "category": row["Category"],
                "price": row["Price"],
                "stock": row["Stock"],
            },
        ))
    return documents

Excel — Multiple Sheets

Excel files can hold several sheets. Loop over them with pandas and turn each sheet into its own document.

import pandas as pd
from langchain_core.documents import Document

def process_excel(filepath: str):
    documents = []
    excel_file = pd.ExcelFile(filepath)
    for sheet_name in excel_file.sheet_names:
        df = pd.read_excel(excel_file, sheet_name=sheet_name)
        content = (
            f"Sheet Name: {sheet_name}\n\n"
            f"Columns: {', '.join(df.columns)}\n\n"
            f"Row Count: {len(df)}\n\n"
            f"{df.to_string(index=False)}"
        )
        documents.append(Document(
            page_content=content,
            metadata={"source": filepath, "sheet_name": sheet_name,
                      "columns": df.columns.tolist(), "row_count": len(df)},
        ))
    return documents

JSON & Nested Data

JSON is everywhere in APIs and logs, but it is nested — the challenge is flattening it without losing context. Our sample:

{
  "company": "TechCorp",
  "employees": [
    {
      "id": 1,
      "name": "John Doe",
      "role": "Software Engineer",
      "skills": ["Python", "JavaScript", "React"],
      "projects": [
        {"name": "RAG System",    "status": "In Progress"},
        {"name": "Data Pipeline", "status": "Completed"}
      ]
    }
  ]
}

Method 1 — JSONLoader with a jq Schema

JSONLoader uses a jq query to pluck out exactly the part you want. Here, .employees[] extracts one document per employee.

from langchain_community.document_loaders import JSONLoader

loader = JSONLoader(
    file_path="data/json_files/company_data.json",
    jq_schema=".employees[]",   # one document per employee
    text_content=False,         # keep the full JSON object
)
docs = loader.load()
print(f"Loaded {len(docs)} employee documents")   # → 2

Method 2 — Custom Flattening (context preserved)

For readable, retrieval-friendly chunks, flatten the nested structure into natural language yourself.

import json
from typing import List
from langchain_core.documents import Document

def process_json_intelligently(filepath: str) -> List[Document]:
    with open(filepath) as f:
        data = json.load(f)

    documents = []
    for emp in data.get("employees", []):
        content = (
            f"Employee Profile:\n"
            f"  Name: {emp['name']}\n"
            f"  Role: {emp['role']}\n"
            f"  Skills: {', '.join(emp['skills'])}\n\n"
            f"  Projects:"
        )
        for proj in emp.get("projects", []):
            content += f"\n  - {proj['name']} (Status: {proj['status']})"

        documents.append(Document(
            page_content=content,
            metadata={
                "source": filepath,
                "data_type": "employee_profile",
                "employee_id": emp["id"],
                "employee_name": emp["name"],
                "role": emp["role"],
            },
        ))
    return documents

The result is a clean, self-contained profile per employee — exactly the kind of chunk an LLM can answer questions about:

Employee Profile:
  Name: John Doe
  Role: Software Engineer
  Skills: Python, JavaScript, React

  Projects:
  - RAG System (Status: In Progress)
  - Data Pipeline (Status: Completed)

Method 2 — Custom SQL → Documents

For RAG, the powerful move is to turn each table — and the relationships between tables — into documents. A JOIN becomes a human-readable chunk.

import sqlite3
from typing import List
from langchain_core.documents import Document

def sql_to_documents(db_path: str) -> List[Document]:
    conn = sqlite3.connect(db_path)
    cursor = conn.cursor()
    documents = []

    # One document per table
    cursor.execute("SELECT name FROM sqlite_master WHERE type='table';")
    for (table_name,) in cursor.fetchall():
        cursor.execute(f"PRAGMA table_info({table_name});")
        columns = [col[1] for col in cursor.fetchall()]
        cursor.execute(f"SELECT * FROM {table_name}")
        rows = cursor.fetchall()

        content = (f"Table: {table_name}\n"
                   f"Columns: {', '.join(columns)}\n"
                   f"Total Records: {len(rows)}\n\nSample Records:\n")
        for row in rows[:5]:
            content += f"{dict(zip(columns, row))}\n"

        documents.append(Document(
            page_content=content,
            metadata={"source": db_path, "table_name": table_name,
                      "num_records": len(rows), "data_type": "sql_table"},
        ))

    # One document for the relationships (a JOIN)
    cursor.execute("""
        SELECT e.name, e.role, p.name AS project_name, p.status
        FROM employees e
        JOIN projects p ON e.id = p.lead_id
    """)
    rel_content = "Employee-Project Relationships:\n\n"
    for name, role, project, status in cursor.fetchall():
        rel_content += f"{name} ({role}) leads {project} - Status: {status}\n"

    documents.append(Document(
        page_content=rel_content,
        metadata={"source": db_path, "data_type": "sql_relationships"},
    ))

    conn.close()
    return documents

The relationship document reads like plain English — perfect for retrieval:

Employee-Project Relationships:

John Doe (Senior Developer) leads RAG Implementation - Status: Active
Jane Smith (Data Scientist) leads Data Pipeline - Status: Completed
Mike Johnson (Product Manager) leads Customer Portal - Status: Planning
Jane Smith (Data Scientist) leads ML Platform - Status: Active

Key Takeaways

1. Every source, no matter how different, ends up as a list of Document objects — text plus metadata.

2. Always enrich metadata (source, page, row, table). It powers citations and metadata filtering at retrieval time.

3. Choose your splitter deliberately — RecursiveCharacterTextSplitter with sensible overlap is the safe default.

4. Clean before you chunk. Artifacts removed here never pollute your embeddings later.
