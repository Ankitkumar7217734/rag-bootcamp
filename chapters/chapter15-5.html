Answer Synthesis from Multiple Sources

Pull context from internal docs, YouTube, Wikipedia, and ArXiv into one coherent answer — then see how all five building blocks compose into Autonomous RAG.

Overview

Every sub-chapter so far retrieved from one knowledge source — a text file, a web scrape, or a single vector store. Real questions don't work that way. “What are transformer agents and how are they evolving in recent research?” needs your internal experiment notes and Wikipedia's definition and recent ArXiv papers and maybe a video explainer. No single store has all of that.

Answer synthesis from multiple sources is the technique of collecting information from several retrieval tools, labelling each chunk by where it came from, and asking the LLM to merge them into one coherent response. This sub-chapter builds a graph with four retrieval nodes (internal docs, YouTube, Wikipedia, ArXiv) and one synthesizer node that combines everything at the end.

Where this fits

From 7-answersynthesis.ipynb, building block 5 of 5 — the last hands-on graph in Chapter 15. After the code walkthrough, this page closes with the Autonomous RAG definition, the complete workflow diagram, and a comparison with Agentic RAG from Chapters 13–14.

Why It's Needed

Most real-world queries are multifaceted (need several kinds of information), ambiguous (need refinement), or open-ended (don't map to one document). A single vector DB is insufficient. The agent must decide what to fetch from where, pull from multiple tools, and produce one human-like response that cites evidence from each source.

The diagram shows four sources fanning out and merging into one synthesizer. In our LangGraph implementation, the four retrievals run as sequential nodes (not truly parallel) because each writes to a different field in the state — they don't conflict, and sequential wiring is simpler to debug. The synthesizer is always last: it reads all four fields and produces final_answer.

What We're Building

Four small loader functions (one per source), four retrieval node functions (one per source), one synthesizer node, and a five-node graph wired in a chain. The state holds a separate slot for each source's output so nothing gets overwritten during collection.

Step 1 — Source Loaders & Tools

Before writing any graph nodes, set up the four data sources. Each loader is a small, self-contained function that returns either a retriever (for FAISS-backed sources) or a plain string (for API-backed sources).

Imports and LLM

We use GPT-4o-mini here — synthesis is mostly formatting and merging, so a smaller model keeps cost down. Wikipedia and ArXiv come from LangChain community integrations you saw in Chapter 14. 

import os
from typing import List
from pydantic import BaseModel
from langchain.chat_models import init_chat_model
from langchain_openai import OpenAIEmbeddings
from langchain_classic.schema import Document
from langchain_classic.vectorstores import FAISS
from langchain_community.document_loaders import TextLoader, ArxivLoader
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from langchain_classic.tools import WikipediaQueryRun
from langchain_classic.utilities import WikipediaAPIWrapper
from langgraph.graph import StateGraph, END

llm = init_chat_model("openai:gpt-4o-mini")

Internal docs — load_text_retriever

Same pattern as 15.1: load research_notes.txt, chunk, embed, return a FAISS retriever. Wrapping it in a function means you can point at any text file without duplicating the ingestion boilerplate. 

def load_text_retriever(file_path):
    docs = TextLoader(file_path, encoding="utf-8").load()
    splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
    chunks = splitter.split_documents(docs)
    vs = FAISS.from_documents(chunks, OpenAIEmbeddings())
    return vs.as_retriever()

YouTube — load_youtube_retriever

In production you'd use YoutubeLoader to pull a real transcript. Here we mock the transcript as a single Document so the graph runs without a network call to YouTube. The mock content mentions agentic AI, feedback loops, and temporal reasoning — topics that will show up in the synthesized answer. 

def load_youtube_retriever():
    content = """
    This video explains how agentic AI systems rely on feedback loops, memory, and tool use.
    It compares them to traditional pipeline-based LLMs. Temporal reasoning and autonomous tasking are emphasized.
    """
    doc = Document(page_content=content, metadata={"source": "youtube"})
    vectorstore = FAISS.from_documents([doc], OpenAIEmbeddings())
    return vectorstore.as_retriever()

Wikipedia — wikipedia_search

Wraps the LangChain Wikipedia tool. Returns a plain string (article summary), not Documents. The print statement is a simple trace so you can see when the node fires during graph.invoke(). 

def wikipedia_search(query: str) -> str:
    print("🌐 Searching Wikipedia...")
    return WikipediaQueryRun(api_wrapper=WikipediaAPIWrapper()).run(query)

ArXiv — arxiv_search

Uses ArxivLoader to fetch up to two paper abstracts matching the query. If nothing is found, returns a fallback string so the synthesizer still has something to work with rather than crashing on an empty result. 

def arxiv_search(query: str) -> str:
    print("📄 Searching ArXiv...")
    results = ArxivLoader(query).load()
    return "\n\n".join(doc.page_content for doc in results[:2]) or "No relevant papers found."

Initialise the retrievers

text_retriever = load_text_retriever("research_notes.txt")
youtube_retriever = load_youtube_retriever()
Gotcha — call .run(), not the object

A WikipediaQueryRun instance is not directly callable. Writing WikipediaQueryRun(...)(query) raises TypeError: 'WikipediaQueryRun' object is not callable. Always invoke it with .run(query). This is the one fix applied to the source notebook so the graph runs end-to-end.

Step 2 — State & Retrieval Nodes

The state schema

One field per source, plus the question and final answer. Wikipedia and ArXiv store plain strings because their loaders return text directly; the FAISS-backed sources store Document lists. 

class MultiSourceRAGState(BaseModel):
    question: str
    text_docs: List[Document] = []
    yt_docs: List[Document] = []
    wiki_context: str = ""
    arxiv_context: str = ""
    final_answer: str = ""

Four retrieval nodes — one per source

Each node reads state.question, calls its source, and writes to exactly one state field. They don't touch each other's fields, which is why sequential execution is safe.

Internal docs: standard FAISS retrieval against your experiment notes.

def retrieve_text(state: MultiSourceRAGState) -> MultiSourceRAGState:
    docs = text_retriever.invoke(state.question)
    return state.model_copy(update={"text_docs": docs})

YouTube: same pattern, different retriever.

def retrieve_yt(state: MultiSourceRAGState) -> MultiSourceRAGState:
    docs = youtube_retriever.invoke(state.question)
    return state.model_copy(update={"yt_docs": docs})

Wikipedia: calls the API wrapper, stores the returned summary string.

def retrieve_wikipedia(state: MultiSourceRAGState) -> MultiSourceRAGState:
    result = wikipedia_search(state.question)
    return state.model_copy(update={"wiki_context": result})

ArXiv: fetches paper abstracts, stores as a single concatenated string.

def retrieve_arxiv(state: MultiSourceRAGState) -> MultiSourceRAGState:
    result = arxiv_search(state.question)
    return state.model_copy(update={"arxiv_context": result})

Step 3 — The Synthesizer

This is the only node that reads all sources. It builds a labelled context block — each section tagged with [Internal Docs], [Wikipedia], etc. — so the LLM can attribute facts to the right source and weigh them appropriately. Without labels, the model can't tell your private notes from Wikipedia's general knowledge. 

def synthesize_answer(state: MultiSourceRAGState) -> MultiSourceRAGState:
    context = ""
    context += "\n\n[Internal Docs]\n" + "\n".join([doc.page_content for doc in state.text_docs])
    context += "\n\n[YouTube Transcript]\n" + "\n".join([doc.page_content for doc in state.yt_docs])
    context += "\n\n[Wikipedia]\n" + state.wiki_context
    context += "\n\n[ArXiv]\n" + state.arxiv_context

    prompt = f"""You have retrieved relevant context from multiple sources. Now synthesize a complete and coherent answer.

Question: {state.question}

Context:
{context}

Final Answer:"""
    answer = llm.invoke(prompt).content.strip()
    return state.model_copy(update={"final_answer": answer})

The prompt is deliberately simple: show the model everything, ask for one answer. In a production system you might add instructions about prioritising internal docs over Wikipedia, or citing sources inline — but the labelled blocks alone go a long way.

Step 4 — Build & Run

Wire the five nodes in sequence. Each retrieval node feeds the next; the chain ends at the synthesizer. You could refactor the four retrievals to run in parallel with LangGraph sub-graphs, but the sequential version is easier to trace when something goes wrong.

Graph wiring

builder = StateGraph(MultiSourceRAGState)
builder.add_node("retrieve_text", retrieve_text)
builder.add_node("retrieve_yt", retrieve_yt)
builder.add_node("retrieve_wiki", retrieve_wikipedia)
builder.add_node("retrieve_arxiv", retrieve_arxiv)
builder.add_node("synthesize", synthesize_answer)

builder.set_entry_point("retrieve_text")
builder.add_edge("retrieve_text", "retrieve_yt")
builder.add_edge("retrieve_yt", "retrieve_wiki")
builder.add_edge("retrieve_wiki", "retrieve_arxiv")
builder.add_edge("retrieve_arxiv", "synthesize")
builder.add_edge("synthesize", END)

graph = builder.compile()

Invoke and print

Watch the console during invoke — you should see the Wikipedia and ArXiv print statements fire as those nodes execute. That confirms all four sources were actually queried before synthesis. 

question = "What are transformer agents and how are they evolving in recent research?"
state = MultiSourceRAGState(question=question)
result = graph.invoke(state)

print("✅ Final Answer:\n")
print(result["final_answer"])
Output — Four Sources, One Answer

Final Answer: Transformer agents combine the transformer architecture with an agentic control loop. From the internal notes: tool-augmented prompting (LangGraph + Wikipedia + SQL) and dynamic RAG pipelines outperform static RAG. From the YouTube transcript: agentic systems rely on feedback loops, memory, and tool use, with an emphasis on temporal reasoning and autonomous tasking. From Wikipedia: the transformer's attention mechanism underlies modern LLMs. From ArXiv: research is pushing tool-use, planning, and multi-agent coordination. Synthesised, the trend is clear — transformers are moving from passive text generators toward autonomous, tool-using agents.

Notice how the answer weaves together evidence from all four sources rather than copying one verbatim. That's synthesis doing its job — the LLM is summarising and reconciling, not just concatenating.

Bringing It Together: Autonomous RAG

You've now built all five ingredients separately across sub-chapters 15.1–15.5. Autonomous RAG is what you get when you wire them into one self-managing loop. You don't need a single mega-graph to understand the concept — each building block is a reusable pattern you compose based on what your use case needs.

What is Autonomous RAG?

Autonomous RAG is a Retrieval-Augmented Generation system where the LLM (or agent) is capable of reasoning, planning, acting, reflecting, and improving — on its own — without manual control over each step.

It combines:

· Agentic reasoning — like ReAct or LangGraph agents
· Self-reflection & self-correction — grade and retry
· Dynamic tool selection — choose the right source per query
· Multi-source retrieval — docs, Wikipedia, ArXiv, APIs…

Core Components

Mapping the five sub-chapters onto roles, a full Autonomous RAG system has seven moving parts. Six are required; memory is optional but valuable for logging bad queries and improving over time.

The Complete Flow

This diagram shows how all five techniques compose into one vertical pipeline. Read it top to bottom: plan and decompose the query, reason through steps (CoT), act via a ReAct tool loop (Chapter 14), check whether context is sufficient (iterative retrieval across multiple sources), synthesize an answer, and finally reflect. If reflection passes, the graph ends. If not, it loops back to gather more context — the self-correcting heartbeat of an autonomous system.

The numbered stages map onto the five techniques you built — query planning, chain-of-thought, the ReAct tool loop (Chapter 14), iterative retrieval across multiple sources, and self-reflection. Autonomy is the loop at the bottom: stage 6 feeds back into stage 4 until the answer is good enough, with no human deciding when to retry.

Agentic RAG vs. Autonomous RAG

You've been building toward this since Chapter 13. Agentic RAG uses an agent to make retrieval decisions. Autonomous RAG is self-managing — reflection, retry, and multi-source synthesis are built in, not bolted on as optional extras. The difference is degree of independence.

ConceptAgentic RAGAutonomous RAG
Definition A RAG system that uses an agentic approach — an LLM reasons, plans, and acts using tools A RAG system that operates independently, with full self-management of planning, retrieving, reflection, and improvement
Focus Structured reasoning and tool use (ReAct, LangGraph, etc.) Complete autonomy in task execution, retry, and learning
Behavior Think → Act → Observe → Answer Think → Act → Reflect → Retry → Learn → Answer
Retry Logic Optional — usually static agent plans Built-in retry/refine strategies (context + answer reflection)

Planner Often present (manual or LLM-generated plans) Always present — triggers multi-step workflows adaptively
Learning Loop Not always present May log feedback, improve over time

Key Takeaways

Remember These Four

1. Autonomous RAG is a composition, not a single trick. It stacks chain-of-thought, self-reflection, query planning & decomposition, iterative retrieval, and multi-source synthesis into one self-managing graph.

2. The conditional edge is what makes it autonomous. Pipelines become loops when a reflect node routes back to retrieve/refine — and an attempts >= 2 guard keeps the loop from running forever.

3. Reflection turns a one-shot guess into a self-correcting answer. Having the LLM grade its own output (YES/NO + explanation) is the cheapest, highest-leverage upgrade over vanilla RAG.

4. Agentic RAG uses an agent; Autonomous RAG is self-managing. Plan → reason → act → reflect → retry, with no human choosing when to try again.
