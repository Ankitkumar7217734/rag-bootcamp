Chapter 14: ReAct Agents — Reasoning, Acting & Observing

How create_react_agent turns an LLM into a self-directed loop that picks the right tool, reads the result, and reasons until it has enough to answer.

Overview

ReAct (Reasoning + Acting) is an agent framework where an LLM alternates between two modes: reasoning — decomposing the problem and deciding what to do next — and acting — calling a tool and reading the result back. Each observation can trigger another round of reasoning, creating a Think → Act → Observe loop that runs until the model is confident enough to produce a final answer.

LangGraph ships a prebuilt create_react_agent that encodes this loop for you. You supply an LLM and a list of tools; the prebuilt handles the internal node structure, ToolNode execution, and the routing logic that decides whether to call another tool or stop and respond. The result is a compiled subgraph you can drop into a StateGraph as a single node.

In this chapter we build two ReAct agents. The first uses a FAISS-backed retriever loaded from a blog post plus a Wikipedia tool. The second introduces a reusable tool factory that wires any text file into a retriever, then assembles a four-tool agent covering Wikipedia, ArXiv, and two private knowledge bases.

The Core Idea

Think: the LLM reasons about the query and selects which tool to call.
Act: the chosen tool (RAG retriever, Wikipedia, ArXiv…) is executed.
Observe: the LLM reads the result and decides: answer now, or call another tool?
This cycle repeats until the model produces a final response — or a configured limit is hit.

What is ReAct?

Reasoning + Acting

ReAct was introduced in a 2022 paper by Yao et al. and has become the dominant paradigm for tool-calling agents. The name is a portmanteau of Reasoning and Acting. Two limitations motivate it:

Chain-of-thought prompting improves reasoning but cannot pull in new information — the LLM is limited to its training data. Tool-use alone (just calling an API) doesn't help if the LLM can't interpret the result and decide whether to query again. ReAct combines both: reason, act, read the result, reason again, and repeat.

PhaseWhat happensWho drives it
Think LLM reasons about the current state and selects the next tool and its input LLM
Act The selected tool executes — RAG retrieval, Wikipedia lookup, ArXiv search… Tool (Python function)
Observe The tool result is appended to the message history; LLM decides what to do next LLM + LangGraph ToolNode
Final Answer LLM stops calling tools and returns a synthesised, grounded response LLM

Why it matters for RAG

Vanilla RAG fires one retrieval, then generates. If the retrieval misses, the answer is wrong and you never know. ReAct lets the agent retrieve, recognise the result is insufficient, call a second source (e.g. Wikipedia), and synthesise both before answering. This multi-hop, self-correcting behaviour is the core upgrade ReAct brings to RAG pipelines.

Part 1: A Basic ReAct Agent

We start with a two-tool agent: a FAISS retriever built from Lilian Weng's LLM agents blog post, and a Wikipedia tool. create_react_agent wires them into a prebuilt Think → Act → Observe graph automatically.

Step 1 — Imports & Building the RAG Retriever

import os
from langgraph.prebuilt import create_react_agent
from langchain_classic.agents import Tool
from langchain_classic.tools import WikipediaQueryRun
from langchain_classic.utilities import WikipediaAPIWrapper
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
from langchain_community.document_loaders import WebBaseLoader
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from langgraph.graph import END, StateGraph
from typing import Annotated, TypedDict, Sequence
from langchain_core.messages import BaseMessage, HumanMessage, AIMessage
from langgraph.graph.message import add_messages

# Load content from Lilian Weng's LLM-powered agents blog post
docs = WebBaseLoader("https://lilianweng.github.io/posts/2023-06-23-agent/").load()
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

embedding = OpenAIEmbeddings()
vectorstore = FAISS.from_documents(chunks, embedding)
retriever = vectorstore.as_retriever()

Step 2 — Wrapping the Retriever as a Tool

LangGraph tools need three things: a name, a description the LLM reads to decide when to use the tool, and a callable that accepts a single string query and returns a string result. 

def retriever_tool_func(query: str) -> str:
    print("📚 Using RAGRetriever tool")
    docs = retriever.invoke(query)
    return "\n".join([doc.page_content for doc in docs])

retriever_tool = Tool(
    name="RAGRetriever",
    description="Use this tool to fetch relevant knowledge base info",
    func=retriever_tool_func
)

Step 3 — Adding Wikipedia

wiki_tool = WikipediaQueryRun(api_wrapper=WikipediaAPIWrapper())

Step 4 — Initialise the LLM

from langchain.chat_models import init_chat_model

os.environ["OPENAI_API_KEY"] = os.getenv("OPENAI_API_KEY")
llm = init_chat_model("openai:gpt-4o")

Step 5 — Create the ReAct Agent

create_react_agent from langgraph.prebuilt builds the entire Think → Act → Observe subgraph from the model and tool list. It returns a compiled StateGraph you can plug straight into an outer graph as a node. 

tools = [retriever_tool, wiki_tool]
react_node = create_react_agent(llm, tools)
Deprecation Note

In LangGraph v1.0+, create_react_agent has moved to langchain.agents. The import above still works in earlier versions but produces a deprecation warning. Update to from langchain.agents import create_react_agent for new projects.

Step 6 — Define State & Build the Graph

The outer graph is intentionally minimal: one node that runs the full ReAct loop, then terminates. The internal Think → Act → Observe cycle is handled inside react_node. 

class AgentState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], add_messages]

builder = StateGraph(AgentState)
builder.add_node("react_agent", react_node)
builder.set_entry_point("react_agent")
builder.add_edge("react_agent", END)

graph = builder.compile()

Step 7 — Run the Agent

user_query = "What is an agent loop and how does Wikipedia describe autonomous agents?"
state = {"messages": [HumanMessage(content=user_query)]}
result = graph.invoke(state)

print("\n✅ Final Answer:\n", result["messages"][-1].content)
Output — Two-Source Synthesis

Tools called: RAGRetriever → Wikipedia (internally by the LLM)

Agent Loop (from RAG source): An iterative process where an LLM-backed agent learns from its actions and improves over time, enabling autonomous scientific discovery and planning — as seen in AutoGPT and BabyAGI.

Autonomous Agents (Wikipedia): An AI system that perceives its environment, takes actions to achieve goals independently, and can improve performance through machine learning. Studied across cognitive science, ethics, and philosophy.

Part 2: A Generic Multi-Tool Agent

The second half of the notebook introduces a reusable pattern: a factory function that turns any text file into a retriever tool in one call. We then assemble a four-tool agent covering Wikipedia, ArXiv, and two internal knowledge bases.

A Reusable Retriever Factory

Instead of hand-writing a retriever closure for each knowledge base, make_retriever_tool_from_text takes a file path, name, and description and returns a ready-to-use Tool. The closure captures the retriever, so each call produces an independent tool with its own vector store. 

from langchain_community.document_loaders import ArxivLoader, TextLoader

def make_retriever_tool_from_text(file, name, desc):
    docs = TextLoader(file, encoding="utf-8").load()
    chunks = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50).split_documents(docs)
    vs = FAISS.from_documents(chunks, OpenAIEmbeddings())
    retriever = vs.as_retriever()

    def tool_func(query: str) -> str:
        print(f"📚 Using tool: {name}")
        results = retriever.invoke(query)
        return "\n\n".join(doc.page_content for doc in results)

    return Tool(name=name, description=desc, func=tool_func)

Wikipedia & ArXiv Tools

In Part 2 the Wikipedia tool is wrapped in a Tool object (rather than passed raw as WikipediaQueryRun) to ensure full compatibility with ToolNode. The ArXiv tool fetches recent academic papers via ArxivLoader and truncates results to keep context manageable. 

# Wikipedia — wrapped for ToolNode compatibility
wiki_tool = Tool(
    name="Wikipedia",
    description="Use this tool to fetch general world knowledge from Wikipedia.",
    func=WikipediaQueryRun(api_wrapper=WikipediaAPIWrapper()).run
)

# ArXiv — fetch recent academic papers
def arxiv_search(query: str) -> str:
    print("🧪 Searching ArXiv...")
    results = ArxivLoader(query).load()
    return "\n\n".join(doc.page_content[:1000] for doc in results[:2]) or "No papers found."

arxiv_tool = Tool(
    name="ArxivSearch",
    description="Use this tool to fetch recent academic papers on technical topics.",
    func=arxiv_search
)

Internal Document Retrievers

Two private text files are turned into retriever tools with a single factory call each. The description is the LLM's only signal for when to use a tool — a precise description routes queries to the right knowledge base without any explicit routing code. 

internal_tool_1 = make_retriever_tool_from_text(
    "sample_docs.txt",
    "InternalTechDocs",
    "Search internal tech documents for proprietary architecture and methods."
)

internal_tool_2 = make_retriever_tool_from_text(
    "research_notes.txt",
    "InternalResearchNotes",
    "Search internal research notes for experimental results and agent designs."
)

Assembling the Multi-Tool Agent

Recreate react_node with all four tools, then build the outer graph. The outer graph structure is identical to Part 1 — one node, one edge to END. Swapping in more tools requires only changing what you pass to create_react_agent. 

# Re-create react_node with the full four-tool set
tools = [wiki_tool, arxiv_tool, internal_tool_1, internal_tool_2]
react_node = create_react_agent(llm, tools)

class AgentState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], add_messages]

builder = StateGraph(AgentState)
builder.add_node("agentic_rag", react_node)
builder.set_entry_point("agentic_rag")
builder.add_edge("agentic_rag", END)

graph = builder.compile()

Running the Multi-Tool Agent

query = "What do our internal research notes say about transformer variants, and what does ArXiv suggest recently?"

state = {"messages": [HumanMessage(content=query)]}
result = graph.invoke(state)

print("\n✅ Final Answer:\n", result["messages"][-1].content)
Output — Multi-Tool Synthesis

Tools called: InternalResearchNotes → ArxivSearch

From internal research notes: Recent transformer variants explore external tool API integration (TALM, Toolformer). Sensory, short-term, and long-term memory concepts are central for tasks like HotpotQA and FEVER.

From ArXiv: Transformers continue to evolve across domains. Vision Transformers (ViTs) replace CNN-style patch processing. The 2017 "Attention Is All You Need" paper laid the groundwork for cross-domain applications including image recognition and autonomous driving.

Tool Design Requirements

Every LangGraph-compatible tool must satisfy three requirements. The description is the most important: the LLM never sees the tool's source code — it decides when to call a tool solely based on the description string.

RequirementWhy it mattersExample
name Identifies the tool in the message history and in structured tool-call responses RAGRetriever, Wikipedia, ArxivSearch
description The LLM reads this to decide whether a given query should go to this tool "Use this tool to fetch recent academic papers on technical topics."
func A callable accepting a single str query and returning a str result retriever_tool_func, arxiv_search, tool_func
Tip: Description quality determines routing accuracy

If two tools have vague descriptions the LLM will pick arbitrarily — or call both when one would do. Write descriptions that state precisely what knowledge lives in each tool.

Vague: "Search documents."
Precise: "Search internal research notes for experimental results and agent designs."

ReAct vs. Custom Agentic RAG

Chapter 13 built a custom Agentic RAG graph with explicit relevance grading and a controlled rewrite loop. create_react_agent is a higher-level abstraction — faster to prototype but less controllable. Here is when to reach for each.

Aspectcreate_react_agent (this chapter)Custom Agentic RAG (Ch. 13)
Setup complexity Minimal — one function call Higher — define nodes, edges, grader
Loop control Automatic (prebuilt Think/Act/Observe) Manual — you control every edge and condition
Relevance grading None — raw tool output goes straight to LLM Explicit structured-output grader per retrieved doc
Rewrite on bad retrieval Not built-in Yes — query rewrite node feeds back into retrieval
Multi-tool routing LLM decides solely from tool descriptions LLM routes via descriptions + your custom graph logic
Loop safety Bounded by LangGraph's recursion limit You control attempt counters and fallback branches
Best for Rapid prototyping, multi-source search agents Production RAG with quality gates and controlled recovery

Key Takeaways

Four Things to Remember

1. ReAct = Think → Act → Observe, repeated until the model is satisfied. create_react_agent encodes this loop — supply an LLM and a list of tools and the subgraph is built for you.

2. Every tool needs three things: a name, a description the LLM reasons from, and a callable that accepts and returns a string. The description is the routing mechanism — write it precisely.

3. make_retriever_tool_from_text is a reusable factory: load a file, chunk it, embed it, wrap the retriever in a Tool. One pattern that scales to any number of private knowledge bases without repeated boilerplate.

4. Use create_react_agent for speed and multi-source breadth; use a custom graph (Chapter 13) when you need explicit relevance grading, controlled rewrite loops, or fine-grained quality gates before generation.
