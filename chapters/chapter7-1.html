Combining Dense & Sparse Retrieval

The first hybrid strategy — fuse keyword search and semantic search so you get the exact-match power of one and the meaning-awareness of the other.

Two Ways to Search Text

There are two fundamentally different ways to search text, and each is blind where the other sees clearly. Hybrid retrieval runs both and blends their scores.

Sparse Retrieval — Exact Keywords

Sparse retrieval matches exact words using classic information-retrieval techniques like TF-IDF and BM25. Text is turned into a sparse matrix — a huge vector that is mostly zeros, with a non-zero entry only for the words actually present. For the sentence "I want to have food", over a vocabulary it looks like:

"I want to have food"  →  [1, 0, 0, 0, 1, 0, ...]   # mostly zeros = "sparse"

Sparse search is exact-match: it shines when the query and document share literal terms (a product code, a rare medical word like "osteoporosis", an API name), but it has no concept of meaning — "build app" and "create LLM system" look completely unrelated to it.

Dense Retrieval — Semantic Meaning

Dense retrieval is what we built in Chapters 4–5: embed the text into a dense vector (every dimension carries a number) and compare with cosine similarity, backed by FAISS or ChromaDB. It captures meaning, so it matches paraphrases and synonyms — but it can overlook an exact keyword that a user specifically typed.

Sparse (BM25 / TF-IDF)Dense (Embeddings)
Matches onExact keywordsSemantic meaning
RepresentationSparse matrix (mostly zeros)Dense vector
StrengthRare terms, codes, exact phrasingSynonyms, paraphrasing, intent
WeaknessNo understanding of meaningCan miss specific keywords
Backed byBM25, TF-IDF, BoWFAISS, ChromaDB + cosine similarity

Hybrid = Sparse + Dense

Hybrid retrieval runs both and blends their scores, so you get the semantic power of embeddings and the exact match of keywords — best of both worlds. The standard recipe is a weighted sum:

Score_hybrid = α · Score_dense + (1 − α) · Score_sparse

  Score_dense  = cosine similarity(query, vector store)
  Score_sparse = TF-IDF / BM25(query, document)
  α            = weight (e.g. 0.5 = equal blend)

The weight α (alpha) tunes the balance: α = 1 is pure semantic search, α = 0 is pure keyword search, α = 0.5 weighs them equally.

Worked Example — α = 0.5

Query: "build application using LLM". Suppose dense and sparse each score three documents:

DocDense scoreSparse scoreHybrid (α = 0.5)
D1 "LangChain helps build LLM apps"0.850.600.725
D2 "Pinecone is used for vector search"0.400.200.300
D3 "Eiffel Tower is in Paris"0.100.100.100

D1 wins on both axes (0.5×0.85 + 0.5×0.60 = 0.725) and ends up clearly on top — the hybrid score rewards a document that is both semantically on-topic and shares the literal keywords "build", "LLM".

Hybrid Retrieval in LangChain

LangChain wraps this in EnsembleRetriever: give it a dense retriever and a sparse (BM25Retriever) one, plus weights, and it fuses their rankings for you.

from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
from langchain_community.retrievers import BM25Retriever
from langchain_classic.retrievers import EnsembleRetriever
from langchain_core.documents import Document

docs = [
    Document(page_content="LangChain is a framework for building applications powered by large language models"),
    Document(page_content="RAG stands for Retrieval Augmented Generation which combines retrieval with LLM generation"),
    Document(page_content="LangChain supports agents, chains, memory, and retrieval components"),
    Document(page_content="Vector databases like FAISS and ChromaDB store embeddings for semantic search"),
    Document(page_content="Hybrid search combines dense retrieval using embeddings with sparse retrieval using BM25"),
]

# Dense retriever — FAISS + OpenAI embeddings (semantic)
embeddings_model = OpenAIEmbeddings(model="text-embedding-3-small",
                                    dimensions=1536, api_key=openai_api_key)
dense_vectorstore = FAISS.from_documents(docs, embeddings_model)
dense_retriever = dense_vectorstore.as_retriever()

# Sparse retriever — BM25 (exact keywords)
sparse_retriever = BM25Retriever.from_documents(docs)
sparse_retriever.k = 3

# Hybrid — blend them 70% dense / 30% sparse
hybrid_retriever = EnsembleRetriever(
    retrievers=[dense_retriever, sparse_retriever],
    weights=[0.7, 0.3],
)

results = hybrid_retriever.invoke("How can I build an application using LLMs?")
for i, doc in enumerate(results):
    print(f"\ndoc {i}:\n{doc.page_content}")

The weights=[0.7, 0.3] argument is exactly the α from the formula — here favouring semantic meaning while still letting exact keywords break ties. Dropping the hybrid_retriever into the same LCEL RAG chain from Chapter 5 needs no other change.

Why Combine Them — The Payoff

#BenefitWhy it matters
1Boosts recallBM25 catches exact matches dense misses; dense captures meaning when keywords differ. Together you miss less.
2Handles synonyms & rephrasingSemantic search matches "create app" to "build LLM system"; BM25 still catches the literal "LLM".
3More robustServes both keyword searchers and natural-language phrasers.
4Respects lexical importanceBM25 weights rare technical/legal/medical terms heavily.
5Bridges document diversityAdapts to mixed corpora — clean docs and messy ones alike.
6Easy to tuneOne weight: score = 0.7·dense + 0.3·sparse.
7Tolerates misspellingsDense models handle typos like "llm aplication" that BM25 fails on.
Key Takeaway

Hybrid retrieval = dense + sparse, blended by a single weight α. It lifts recall by covering both meaning and exact keywords. In LangChain it's one object — EnsembleRetriever(retrievers=[dense, sparse], weights=[...]).
