Chapter 21: Vectorless RAG — Reasoning-Based Retrieval with PageIndex

Twenty chapters of chunk–embed–retrieve, and now the assumption itself goes: no chunking, no embeddings, no vector database. Build a tree from the document, hand it to the LLM, and let it reason its way to the answer — the way a human expert scans a table of contents.

Vectorless RAG in One Paragraph

Every retrieval system so far has answered “which chunks are most similar to this query?” — and similarity is not the same thing as relevance. A chunk about “market conditions” can outscore the actual answer section simply because it shares more words with the question. Vectorless RAG asks a different question entirely: “which section of this document would a human expert open?” It parses the document into a hierarchical tree — a smart table of contents with a summary at every node — and lets the LLM reason over that structure to pick the exact sections that contain the answer. No chunks, no embeddings, no vector DB — and every answer arrives with the section title and page number it came from.

Overview

This chapter is built from two sources: the Vectorless vs Traditional RAG deck (the theory — when each approach wins and why) and the PageIndex crash-course notebook (the practice — a complete reasoning-based RAG pipeline over a real 34-page PDF, the Krish Naik Academy “Advanced Route of Learning AI” syllabus). PageIndex is the reference implementation of the vectorless idea: a cloud API (plus an open-source, self-hostable version) that turns any PDF into a tree index an LLM can navigate.

The Core Idea

Similarity ≠ relevance. Cosine similarity measures how much two pieces of text overlap in meaning-space — it has no idea what the document is about, how it is organized, or what the question actually needs. Vectorless RAG replaces the similarity score with reasoning: give the LLM the document's structure (titles, summaries, page numbers, nesting) and ask it, in plain language, which sections contain the answer. The retrieval step becomes an explainable chain of thought instead of an opaque number.

Why Vector RAG Fails on Professional Documents

First, Why RAG at All

The deck opens with the problem every RAG chapter in this bootcamp has been solving. You can't just feed everything to the LLM, for three reasons:

  • Context windows are limited — even 1M tokens can't hold a corporate knowledge base.
  • Long context degrades quality — LLMs lose recall in the middle of huge prompts.
  • It's expensive — tokens cost money; less context means lower bills.

So we retrieve relevant context on demand: fetch only the pieces that matter for this query, then ask the LLM. The big question — the one this whole chapter turns on — is how do we retrieve?

The Traditional Answer, and Its Failure Modes

Traditional RAG's five-step pipeline is the one you built in Chapters 2–5: chunk documents into 200–500-token pieces, embed each chunk into a vector, store the vectors in a database (Chroma, FAISS, Pinecone), search top-k by cosine similarity, and generate from the retrieved chunks. It's powerful — but the deck is honest about where it breaks:

StrengthsWeaknesses
Massive scale — millions of docs, millisecond lookups Chunking destroys context — “As defined in Section 3.2…” is meaningless when retrieved alone
Mature ecosystem — Chroma, FAISS, Pinecone, Qdrant, Weaviate Similarity ≠ relevance — embeddings can match the wrong things, confidently
Cheap retrieval — one embedding + ANN search per query No cross-section reasoning — can't answer “compare risks vs mitigations”
Great for factoids — short, lookup-style questions Hard to explain — why was this chunk picked? A cosine score isn't an answer
Domain agnostic — works on any text: blogs, tickets, PDFs Embedding drift — change the embedding model and you re-embed everything

Notice what the weaknesses have in common: they all bite hardest on long, professionally structured documents — annual reports, 10-Ks, legal contracts, regulatory filings, research papers, textbooks. Exactly the documents where the author already organized the content into a hierarchy, and exactly the structure that chunking throws away.

The Vectorless Answer — Navigate Like a Human

Vectorless RAG skips embeddings entirely and lets the LLM navigate the document the way a person would. The document becomes a hierarchical index — a tree whose nodes are the document's own chapters, sections, and sub-sections, each carrying a short summary. Retrieval is a guided descent:

  1. Build the tree — parse the document structure (headings, sections) and generate a summary at each node. Done once, offline.
  2. The LLM reads the root summaries — “which chapter is most likely to contain the answer?”
  3. Descend the tree — repeat the question at each level until reaching a leaf section.
  4. Read the full section — no chunking, full context preserved.
  5. Answer and cite the path — the system returns the answer and the navigation path it took.

The tradeoffs are real — they're just different tradeoffs:

StrengthsWeaknesses
Preserves document context — sections stay whole, no broken references Higher per-query cost — one or more LLM calls to traverse the tree
Cross-section reasoning — the LLM can compare, contrast, synthesize Higher latency — hundreds of ms to a few seconds per query
Explainable retrieval — returns the navigation path, not a cosine score Doesn't scale to millions — tens to thousands of docs, not internet-scale
No embedding pipeline — skip the embed/index/refresh complexity Needs structured docs — random blog posts? A tree adds little value
Shines on structure — reports, contracts, filings, textbooks Young ecosystem — PageIndex and a few others; tooling is still maturing
Keep Both Frames in Mind

Traditional RAG asks “what looks like the query?” and wins on scale, speed, and cost. Vectorless RAG asks “where would the answer live?” and wins on reasoning, context, and explainability. They are not competitors — the end of this chapter shows how production systems combine them. But first, let's build the vectorless side for real.

Why No Chunking?

Instead of cutting the document into arbitrary 500-token pieces, PageIndex respects the document's natural section boundaries — chapters, sub-sections, paragraphs — as the author intended. A section is a unit of meaning; a chunk is a unit of token count. Chapter 6 spent a whole chapter approximating meaning-boundaries with semantic chunking — a tree index gets them for free, because the author already drew them.

LLM Tree Search — The Core of Vectorless Retrieval

This is where PageIndex fundamentally differs from vector RAG. Put the two retrieval mechanics side by side: 

# Vector RAG retrieval:
query → embed → cosine_similarity(query_vec, all_chunk_vecs) → top-k chunks
# Problem: finds what's SIMILAR, not what's RELEVANT

# PageIndex retrieval:
query + tree → LLM reasons → "nodes 0010 and 0014 contain the answer"
# Advantage: the LLM understands document structure, context, and intent

The llm_tree_search Function

The whole trick is one prompt. Send the LLM the query plus the document tree, and ask it to return the relevant node IDs with its reasoning. Two details make this production-worthy rather than a toy:

  • Tree compression. The full tree carries the complete text of every section — far too many tokens to send on every query. The inner compress() helper rebuilds the tree with only node_id, title, page, and the first 150 characters of each summary — enough signal to navigate by, at a fraction of the cost.
  • Forced JSON. response_format={"type": "json_object"} guarantees the reply parses — a thinking string (the audit trail) and a node_list array (the retrieval result).
# ── LLM Tree Search Function ─────────────────────────────────────────────────

def llm_tree_search(query: str, tree: list, model: str = "gpt-4o") -> dict:
    """
    Core PageIndex retrieval:
    Sends the query + document tree to an LLM.
    LLM reasons over the structure and returns relevant node_ids.

    Returns: dict with 'thinking' (reasoning) and 'node_list' (node IDs)
    """

    # Compress tree to save tokens — only send titles + short summaries
    def compress(nodes):
        out = []
        for n in nodes:
            entry = {
                "node_id": n["node_id"],
                "title":   n["title"],
                "page":    n.get("page_index", "?"),
                "summary": n.get("text", "")[:150]  # first 150 chars
            }
            if n.get("nodes"):
                entry["children"] = compress(n["nodes"])
            out.append(entry)
        return out

    compressed_tree = compress(tree)

    prompt = f"""You are given a query and a document's tree structure (like a Table of Contents).
Your task: identify which node IDs most likely contain the answer to the query.
Think step-by-step about which sections are relevant.

Query: {query}

Document Tree:
{json.dumps(compressed_tree, indent=2)}

Reply ONLY in this exact JSON format:
{{
  "thinking": "<your step-by-step reasoning>",
  "node_list": ["node_id1", "node_id2"]
}}"""

    response = openai_client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"}
    )

    return json.loads(response.choices[0].message.content)

The Full Pipeline — Search, Retrieve, Generate

Three steps make the complete system: tree search picks the node IDs, retrieval fetches those nodes' actual content, and generation writes a grounded answer with citations. Each step is a small function.

Step 2 — Retrieve the Nodes

The tree search returned IDs; now walk the tree and collect the matching nodes (children included, so the recursion mirrors print_tree): 

# ── Helper: Find nodes by ID ─────────────────────────────────────────────────

def find_nodes_by_ids(tree: list, target_ids: list) -> list:
    """Recursively walk the tree and collect nodes matching target_ids."""
    found = []
    for node in tree:
        if node["node_id"] in target_ids:
            found.append(node)
        if node.get("nodes"):
            found.extend(find_nodes_by_ids(node["nodes"], target_ids))
    return found

Step 3 — Generate With Citations

Each retrieved node becomes a labeled context block — [Section: 'PEFT' | Page 13] — and the prompt requires the LLM to cite the section title and page number for every claim. Because the context arrives as named sections rather than anonymous chunks, the citations are real, checkable references into the original PDF: 

# ── Generate answer from retrieved nodes ─────────────────────────────────────

def generate_answer(query: str, nodes: list, model: str = "gpt-4o") -> str:
    """
    Takes retrieved nodes as context and generates a grounded answer.
    Instructs the LLM to cite section titles and page numbers.
    """
    if not nodes:
        return "⚠️ No relevant sections found in the document."

    # Build context string from retrieved nodes
    context_parts = []
    for node in nodes:
        context_parts.append(
            f"[Section: '{node['title']}' | Page {node.get('page_index', '?')}]\n"
            f"{node.get('text', 'Content not available.')}"
        )
    context = "\n\n---\n\n".join(context_parts)

    prompt = f"""You are an expert document analyst.
Answer the question using ONLY the provided context.
For every claim you make, cite the section title and page number in parentheses.
Be concise and precise.

Question: {query}

Context:
{context}

Answer:"""

    response = openai_client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": prompt}]
    )

    return response.choices[0].message.content

Putting It Together

One function chains the three steps, with verbose logging so you can watch every stage:

# ── The complete Vectorless RAG function ─────────────────────────────────────

def vectorless_rag(query: str, tree: list, verbose: bool = True) -> str:
    """
    Full end-to-end PageIndex RAG pipeline:

    Step 1: LLM Tree Search  → finds relevant node_ids
    Step 2: Node Retrieval   → fetches section content
    Step 3: Answer Generation → produces cited answer
    """
    if verbose:
        print(f"{'='*55}")
        print(f"🔍 Query: {query}")
        print(f"{'='*55}")

    # Step 1: Tree Search
    search_result  = llm_tree_search(query, tree)
    node_ids       = search_result.get("node_list", [])

    if verbose:
        print(f"\n🧠 Reasoning: {search_result.get('thinking', '')[:200]}...")
        print(f"🎯 Retrieved node IDs: {node_ids}")

    # Step 2: Retrieve nodes
    nodes = find_nodes_by_ids(tree, node_ids)

    if verbose:
        print(f"📄 Sections found: {[n['title'] for n in nodes]}")

    # Step 3: Generate answer
    answer = generate_answer(query, nodes)

    if verbose:
        print(f"\n📝 Answer:\n{answer}")

    return answer

    tree=pageindex_tree

Notice What's Missing

No FAISS.from_texts. No HuggingFaceEmbeddings. No similarity_search(k=4). The entire retrieval layer that Chapters 4–5 built is replaced by one LLM call over a JSON tree — and the “index” is just a JSON file you could check into git. That's the “vectorless” in Vectorless RAG.

Expert-Guided Retrieval — Domain Knowledge as a Prompt

Here's the feature the notebook calls “the killer feature no one talks about.” Suppose your retrieval keeps missing because your domain has conventions the model doesn't know — in finance, EBITDA questions are answered in the MD&A section; in legal, definitions live in the annexes. With vector RAG, teaching the retriever this kind of knowledge means fine-tuning the embedding model — expensive, slow, and it has to be redone for every domain.

With tree search, domain expertise is just more prompt text. You write routing rules like a senior analyst's playbook and inject them into the retrieval prompt: 

# ── Define domain expert rules ───────────────────────────────────────────────
# Routing rules that tell the LLM WHERE to look for specific queries.
# Think of it as encoding a senior analyst's institutional knowledge.

FINANCIAL_EXPERT_RULES = """

- EBITDA, profitability queries    → MD&A section (Management Discussion & Analysis)

Injecting the Rules Into Tree Search

The expert version of the search function is llm_tree_search with one extra prompt block — the rules — and an instruction to follow them: 

# ── Expert-guided tree search ────────────────────────────────────────────────

def llm_tree_search_with_expert(
    query: str,
    tree: list,
    expert_rules: str,
    model: str = "gpt-4o"
) -> dict:
    """
    Same as llm_tree_search() but with domain expert rules injected.
    The LLM uses these rules to guide its reasoning.
    """

    def compress(nodes):
        out = []
        for n in nodes:
            entry = {"node_id": n["node_id"], "title": n["title"],
                     "page": n.get("page_index", "?"),
                     "summary": n.get("text", "")[:150]}
            if n.get("nodes"):
                entry["children"] = compress(n["nodes"])
            out.append(entry)
        return out

    prompt = f"""You are a domain expert analyzing a document.
Find all node IDs that most likely contain the answer to the query.
Use the expert routing rules below to guide your reasoning.

Query: {query}

Document Tree:
{json.dumps(compress(tree), indent=2)}

Expert Routing Rules (follow these carefully):
{expert_rules}

Reply ONLY in this JSON format:
{{
  "thinking": "<your reasoning, referencing the expert rules>",
  "node_list": ["node_id1", "node_id2"]
}}"""

    response = openai_client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"}
    )
    return json.loads(response.choices[0].message.content)

Why This Is the Killer Feature

Domain adaptation became prompt engineering. A financial analyst, a lawyer, or a doctor can read those routing rules, correct them, and extend them — no ML expertise, no training run, no re-indexing. Compare the vector-RAG equivalent: collect labeled query–chunk pairs, fine-tune the embedding model, re-embed the corpus, re-evaluate. PageIndex makes the system's retrieval policy a human-readable, human-editable document.

Multi-turn chat works the same way — keep the message history yourself and send the whole conversation each turn, exactly like the raw OpenAI chat format: 

# ── Multi-turn conversation ───────────────────────────────────────────────────
conversation_history = []

def chat_with_doc(user_message: str, doc_id: str) -> str:
    """Chat with a document, maintaining conversation history."""
    global conversation_history

    conversation_history.append({"role": "user", "content": user_message})

        messages=conversation_history,

Your situationPickBecause

The Two-Question Decision Heuristic

The deck compresses the whole decision into two questions:

Q1 — Is the answer in one localized chunk, or does it require navigating structure? Localized points to traditional; structural points to vectorless.

Q2 — What does your corpus look like?

  • Millions of docs → Traditional RAG — vector search wins on scale.
  • Hundreds to thousands → either works — pick based on query type (Q1).
  • Tens of long, well-structured docs (TOC, headings) → likely vectorless — tree navigation is reliable.
  • Mixed structure → hybrid — vector to filter, tree to reason.
  • Truly unstructured → Traditional RAG — a tree adds little value.

The Hybrid Pattern

Which brings us to where serious production systems actually land: both. Vectors are a filter; trees are a microscope.

They're Not Competitors

Pure vector search and pure tree navigation are both extremes. The right pick depends on the document, not the hype: long structured filings → vectorless; mixed knowledge base → vector; big system → hybrid. If you remember one sentence from the deck, make it this one.

Key Takeaways

Remember These Four

1. Similarity ≠ relevance — that's the whole motivation. Cosine similarity finds chunks that look like the query, and on long professional documents that's routinely the wrong section. Vectorless RAG replaces the similarity score with LLM reasoning over the document's own structure — and posts 98.7% on FinanceBench vs ~80% for vector RAG.

2. The tree is the index. PageIndex turns a PDF into a hierarchy of nodes — node_id, title, page_index, summary, children — stored as plain JSON. Retrieval is llm_tree_search (LLM picks node IDs, with its thinking as an audit trail) → find_nodes_by_ids → generate_answer with (Section, Page) citations on every claim.

3. Domain expertise is just prompt text. Expert routing rules — “EBITDA → MD&A section”, “fine-tuning questions → M9” — get injected into the retrieval prompt and immediately reshape what gets retrieved. No embedding fine-tune, no re-indexing, and the rules stay human-readable and human-editable.

4. Pick by document, and go hybrid at scale. Millions of mixed docs, factoid queries, tight latency → vector RAG. Tens-to-thousands of long structured docs, cross-section reasoning, audit trails → vectorless. Serious production systems chain them: vector filter to ~10 candidates, then tree navigation inside each — scale from vectors, precision from trees.
