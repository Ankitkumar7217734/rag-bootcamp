Chains, Tools & the Router Agent

This is where the graph stops being a fixed pipeline and starts thinking: the LLM decides whether to answer directly or reach for a tool — and that decision is the agent.

What We're Building

A chain in LangGraph ties together four ideas you'll use in every agent:

  1. How to use chat messages as the graph state.
  2. How to use chat models in graph nodes.
  3. How to bind tools to the LLM.
  4. How to execute the tool call in a graph node.

By the end, the graph will look at a user message and route itself: a question like "what is 2 plus 2" triggers a tool call, while "what is Machine Learning" gets answered directly by the model. That self-routing is the simplest possible agent.

Messages — the Four Roles

LangChain represents a conversation as a list of typed messages, each capturing a different role:

Message typeRepresents
HumanMessageA message from the user.
AIMessageA message from the chat model.
SystemMessageInstructions that steer the model's behaviour.
ToolMessageThe result returned from a tool call.

Each message carries its content, an optional author name, and optional response_metadata (often populated by the provider on AI messages). You can build and pretty-print a conversation by hand:

from langchain_core.messages import HumanMessage, AIMessage, SystemMessage, ToolMessage
from pprint import pprint

messages = [AIMessage(content="please tell me how can I help", name="LLMModels")]
messages.append(HumanMessage(content="hi, how are you?", name="User"))
messages.append(AIMessage(content="which programming language you want to learn", name="LLMModels"))
messages.append(HumanMessage(content="I want to learn RAG", name="Ankit"))

for message in messages:
    message.pretty_print()

Tools — Giving the LLM Hands

An LLM on its own can only produce text. Tools let it interact with external systems — an API, a third-party service, or a database. The flow is: a query comes in as natural language, the model decides whether to call a tool, and if it does, the tool runs and returns a result the model can use.

Defining a tool

A tool is just a Python function with a docstring — LangChain reads the signature and docstring to build the JSON schema the model sees. Here's a simple add:

def add(a: int, b: int) -> int:
    """
    Args:
        a (int): first number
        b (int): second number

    Returns:
        int: sum of a and b

    Example:
        >>> add(1, 2)
        3
    """
    return a + b

Binding the tool

Binding attaches the tool's schema to the model so the model knows it can call it. The model still decides whether to. We call bind_tools and then invoke with a question that needs arithmetic:

# bind the function to the llm
llm_with_tools = llm_groq.bind_tools([add])

tool_calls = llm_with_tools.invoke([HumanMessage(content="what is 1+2?", name="user")])
tool_calls.tool_calls
[{'name': 'add',
  'args': {'a': 1, 'b': 2},
  'id': '2q8js3923',
  'type': 'tool_call'}]

Crucially, the model did not compute 3 itself. It returned a structured request — "call add with a=1, b=2" — and left the actual execution to us. That separation is what makes tool calling reliable: the LLM picks the tool and the arguments; deterministic code does the work.

Using Messages as State (with the Reducer)

To turn this into a graph, the state is again a list of messages with the add_messages reducer — so each node appends rather than overwrites (see Chapter 11.2 for why this matters):

from typing_extensions import TypedDict
from typing import Annotated
from langchain_core.messages import AnyMessage
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]

The node simply invokes the tool-bound model on the running messages:

def llm_tools(state: State) -> State:
    return {"messages": [llm_with_tools.invoke(state["messages"])]}

A first graph wires that single node between START and END:

from IPython.display import display, Image
from langgraph.graph import StateGraph, START, END

builder = StateGraph(State)
builder.add_node("llm_tools", llm_tools)
builder.add_edge(START, "llm_tools")
builder.add_edge("llm_tools", END)

graph = builder.compile()
display(Image(graph.get_graph().draw_mermaid_png()))

Invoke it with an arithmetic question and the model produces a tool call — but note there's no tool result yet, because nothing in this graph actually runs the tool:

message = graph.invoke({"messages": "what is 2 plus 2"})
for msg in message["messages"]:
    msg.pretty_print()

The graph stopped at the request. To close the loop we need a node that executes the tool and a way to decide when to route there.

Executing Tools — ToolNode + tools_condition

LangGraph ships two prebuilt helpers that do exactly this:

HelperWhat it does
ToolNode(tools)A ready-made node that runs whatever tool the last AI message asked for and returns a ToolMessage with the result.
tools_conditionA prebuilt conditional edge: if the latest AI message contains a tool call, route to the tools node; otherwise route to END.
 
tools = [add]

from langgraph.prebuilt import ToolNode, tools_condition

builder = StateGraph(State)
builder.add_node("llm_tools", llm_tools)
builder.add_node("tools", ToolNode(tools))

builder.add_edge(START, "llm_tools")
builder.add_conditional_edges(
    "llm_tools",
    # If the latest message from the assistant is a tool call -> route to "tools".
    # If it is NOT a tool call -> route to END.
    tools_condition,
)
builder.add_edge("tools", END)

graph = builder.compile()
display(Image(graph.get_graph().draw_mermaid_png()))

This is the router. The llm_tools node feeds into tools_condition, which sends the flow one of two ways:

Why this is a Router — and an Agent

We can think of this as a router, where the chat model routes between a direct response or a tool call based on the user input. And that makes it a simple agent: the LLM is directing the control flow — either by calling a tool or by just responding directly. The decision lives in the model, not in hard-coded if statements.

Running it both ways

Now an arithmetic question flows all the way through — the model requests add, the tools node runs it, and a ToolMessage carries back the answer 4:

Same graph, two completely different paths — chosen at runtime by the model. That's the essence of an agent.

From router to full agent loop

In this graph the tools node goes straight to END. A complete agent instead loops the tool result back to llm_tools (the dashed arrow above), so the model can read the tool output and decide whether to answer or call another tool. That single extra edge — tools → llm_tools instead of tools → END — turns the one-shot router into an agent that can chain multiple tool calls. We build exactly that in the next chapter.

Scaling Up — Multiple Tools

Nothing limits you to one tool. You bind a list, and the model picks the right one per query. A common upgrade is real external tools — for example, Arxiv and Wikipedia search from langchain_community:

from langchain_community.tools import ArxivQueryRun, WikipediaQueryRun
from langchain_community.utilities import ArxivAPIWrapper, WikipediaAPIWrapper

api_wrapper_arxiv = ArxivAPIWrapper(top_k_results=2, doc_content_chars_max=500)
arxiv = ArxivQueryRun(api_wrapper=api_wrapper_arxiv)
print(arxiv.name)        # -> arxiv

arxiv.invoke("Attention is all you need")
'Published: 2021-05-06
Title: Do You Even Need Attention? A Stack of Feed-Forward Layers Does Surprisingly
       Well on ImageNet
Authors: Luke Melas-Kyriazi
Summary: The strong performance of vision transformers on image classification...'

To build a multi-tool chatbot, you bind the whole list and reuse the exact same router pattern — only the tools list grows:

tools = [add, arxiv]                      # mix custom + external tools
llm_with_tools = llm_groq.bind_tools(tools)

builder = StateGraph(State)
builder.add_node("llm_tools", llm_tools)
builder.add_node("tools", ToolNode(tools))   # ToolNode handles ALL tools

builder.add_edge(START, "llm_tools")
builder.add_conditional_edges("llm_tools", tools_condition)
builder.add_edge("tools", END)
graph = builder.compile()

Now "what is 2 plus 2" routes to add, "summarise the Attention paper" routes to arxiv, and "what is Machine Learning" skips tools entirely — all decided by the model. The ToolNode + tools_condition pattern doesn't change; you just hand it more tools.

Key Takeaways

Remember These Five

1. A conversation is a list of typed messages — Human, AI, System, Tool — and the graph state carries them with the add_messages reducer.

2. A tool is a documented Python function; llm.bind_tools([...]) shows the model its schema. The model returns a structured tool call (name + args) — it does not execute the tool itself.

3. ToolNode runs the requested tool and returns a ToolMessage; tools_condition is the prebuilt conditional edge that routes to tools when there's a tool call and to END otherwise.

4. Together they form a router: the LLM decides between answering directly and calling a tool — a simple agent, because the model directs the control flow.

5. Loop tools back to the LLM (instead of to END) to get a full agent that chains tool calls; bind a list of tools to scale from one tool to many without changing the graph.
