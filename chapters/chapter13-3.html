Running the Agent — the Loop & Hardening

The graph is compiled. Now we run it on three deliberately different questions to see all three paths — straight through, routed to the other store, and stuck in the rewrite loop — and learn why that last one means you must always bound the loop.

Invoking the Graph

Running the graph is one line. Because the state uses the add_messages reducer, you can pass a bare string and LangGraph wraps it as the first HumanMessage for you:

graph.invoke({"messages": "What is Langgraph?"})

The print statements we put in each node act as a built-in trace — they tell you exactly which path the graph took on this run. Let's read all three.

Scenario 1 — A LangGraph Question (the happy path)

"What is Langgraph?" is squarely in store A's wheelhouse. The trace flows straight down the graph — agent calls the retriever, the docs grade as relevant, generate writes the answer:

---CALL AGENT---
---CHECK RELEVANCE---
---DECISION: DOCS RELEVANT---
---GENERATE---

Behind those four lines, the messages list grew from one entry to four. Printing them out shows the full conversation the graph built:

================================ Human Message ================================
What is Langgraph?

================================== Ai Message ==================================
Tool Calls:
  retriever_vector_db_blog (8e9yyh72q)
  Args:
    query: Langgraph

================================= Tool Message =================================
Name: retriever_vector_db_blog
LangGraph overview - Docs by LangChain ... LangGraph is a low-level orchestration
framework and runtime for building, managing, and deploying long-running, stateful agents.

================================== Ai Message ==================================
LangGraph is a low-level orchestration framework for building stateful, multi-actor
applications with LLMs. It models an application as a graph of nodes and edges over a
shared state, and is used in production by companies such as Uber, Klarna, and LinkedIn.

Trace the four messages against the four trace lines: the agent emitted a tool call to retriever_vector_db_blog with the query it wrote itself ("Langgraph"); the retrieve node ran it and appended a tool message with the docs; grade_documents scored them relevant; and generate appended the final answer. That's the full agent → retrieve → grade → generate path.

Scenario 2 — A LangChain Question (routing in action)

Now ask about LangChain instead. The path through the graph is identical, but the agent picks the other tool — retriever_vector_langchain_blog — entirely on the strength of the tool descriptions:

graph.invoke({"messages": "What is Langchain?"})
---CALL AGENT---
---CHECK RELEVANCE---
---DECISION: DOCS RELEVANT---
---GENERATE---

The only difference is invisible in the trace prints but plain in the tool call — a different tool, a different store:

================================== Ai Message ==================================
Tool Calls:
  retriever_vector_langchain_blog (3f2vxzbc2)
  Args:
    query: What is Langchain?

This is the payoff of building two stores in 13.1. We never wrote a single line of routing logic — no keyword matching, no if "langchain" in question. The agent read the two descriptions and chose correctly. Add a third store tomorrow and the routing extends itself.

Scenario 3 — An Off-Topic Question (the loop fires)

Here's the interesting case. Ask something neither store knows about. Machine learning isn't in the LangGraph or LangChain docs, so the docs grade as not relevant — and the rewrite loop kicks in:

graph.invoke({"messages": "What is Machine learning?"})
---CALL AGENT---
---CHECK RELEVANCE---
---DECISION: DOCS NOT RELEVANT---
---TRANSFORM QUERY---
---CALL AGENT---
---CHECK RELEVANCE---
---DECISION: DOCS NOT RELEVANT---
---TRANSFORM QUERY---
---CALL AGENT---
---CHECK RELEVANCE---
---DECISION: DOCS NOT RELEVANT---
---TRANSFORM QUERY---
...

Read the cycle: agent → retrieve → grade (✗) → rewrite → agent → …, over and over. Each pass the agent dutifully retrieves, the grader correctly says "not relevant," and the rewriter produces yet another phrasing of a machine-learning question. But no rephrasing can conjure machine-learning content out of stores that simply don't contain any. The loop has no natural exit.

"What is Langchain?"agent → retrieve (store B) → grade ✓ → generateGrounded answer

Hardening — Always Bound the Loop

The self-correcting loop is the best and the most dangerous part of Agentic RAG. The same back-edge that lets the system recover from one bad retrieval will, on an unanswerable question, spin forever — burning tokens and latency — until LangGraph's built-in recursion limit finally throws a GraphRecursionError. Relying on that error as your stop condition is not a plan. You need a deliberate exit.

The fix is to make the loop count. Three patterns, smallest to largest:

GuardHowBehaviour when it trips
Recursion limitgraph.invoke(..., {"recursion_limit": 10})Hard stop — raises an error after N super-steps. A blunt safety net, not a UX.
Attempt counterAdd an attempts int to the state; increment it in rewrite; have the edge route elsewhere once it hits a cap.Controlled exit you design — stop after, say, 3 rewrites.
Fallback branchOn giving up, route to a web-search node, or a node that returns an honest "I couldn't find that."The user gets a useful answer or an honest miss — never a hang.
The loop is a feature you must fence

Don't read Scenario 3 as a bug — it's the grading working correctly, refusing to answer machine-learning questions from LangGraph docs. That refusal is exactly what stops the system hallucinating. The lesson isn't "remove the loop," it's "bound the loop": give it a counter and a graceful fallback so a self-correcting system stays self-correcting without becoming a runaway.

A Note on the Real World

Run any of this and the first thing you'll see is warnings, not errors:

DeprecationWarning: `langchain-community` is being sunset ...
USER_AGENT environment variable not set, consider setting it to identify your requests.

This is normal. langchain-community integrations are gradually moving into standalone packages, and WebBaseLoader politely asks you to identify your scraper (which is why we passed a User-Agent header back in 13.1). Neither blocks the pipeline — but reading the warnings instead of ignoring them is how you keep a RAG system healthy as the ecosystem shifts underneath it.

Key Takeaways

Remember These Five

1. graph.invoke({"messages": "..."}) accepts a bare string because add_messages wraps it as the first HumanMessage. The node prints are your free, built-in trace.

2. On an on-topic question the path is agent → retrieve → grade ✓ → generate, and the messages list grows Human → AI(tool call) → Tool(docs) → AI(answer).

3. Routing is free. A LangChain question fires the LangChain tool and a LangGraph question fires the LangGraph tool — chosen from the tool descriptions, with zero routing code.

4. When no store can answer, the grader keeps returning "not relevant" and the rewrite loop never exits on its own — proof the self-check is working, and a hazard if left unbounded.

5. Always bound the loop: a recursion limit, an attempt counter in the state, and a fallback branch (web search or an honest "I don't know") turn a potential runaway into a graceful one.
