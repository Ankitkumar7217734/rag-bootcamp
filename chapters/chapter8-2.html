Query Decomposition

The second technique — when one question secretly contains several, split it. Break a complex query into atomic sub-questions, answer each on its own, then stitch the answers back together.

What is Query Decomposition?

Query decomposition is the process of taking a complex, multi-part question and breaking it into simpler, atomic sub-questions that can each be retrieved and answered individually.

Consider: "How does LangChain use memory and agents compared to CrewAI?" That single sentence is really asking three or four different things at once. A single retrieval will pull a muddle of partially-relevant chunks. Decomposition splits it into focused sub-questions, each of which retrieves clean, on-point context.

Why Use Query Decomposition?

  • Complex queries often involve multiple concepts — one search can't serve them all.
  • LLMs or retrievers may miss parts of a multi-part question.
  • It enables multi-hop reasoning — answering in steps.
  • It allows parallelism, especially in multi-agent frameworks (each sub-question is independent).

How Query Decomposition Works

A decomposer (an LLM + prompt) splits the user query into sub-questions. Each sub-question runs through its own retriever → LLM call, producing a partial answer. Finally an answer combiner / synthesiser merges the partial answers into one final response.

Query Decomposition in LangChain

First, a decomposition chain — a prompt that asks the LLM to split the question into 2–4 sub-questions:

from langchain.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from langchain_classic.document_loaders import TextLoader
from langchain.chains.combine_documents import create_stuff_documents_chain

# Step 1: Load, chunk, embed, build an MMR retriever
loader = TextLoader("langchain_crewai_dataset.txt")
docs = loader.load()
splitter = RecursiveCharacterTextSplitter(chunk_size=300, chunk_overlap=50)
chunks = splitter.split_documents(docs)

embedding = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
vectorstore = FAISS.from_documents(chunks, embedding)
retriever = vectorstore.as_retriever(
    search_type="mmr", search_kwargs={"k": 4, "lambda_mult": 0.7}
)

# Step 3: Query decomposition chain
decomposition_prompt = PromptTemplate.from_template("""
You are an AI assistant. Decompose the following complex question into 2 to 4
smaller sub-questions for better document retrieval.

Question: "{question}"

Sub-questions:
""")
decomposition_chain = decomposition_prompt | llm | StrOutputParser()

Next, a per-sub-question QA chain, and a pipeline that loops over each sub-question — retrieve, answer, collect:

# Step 4: QA chain run once per sub-question
qa_prompt = PromptTemplate.from_template("""
Use the context below to answer the question.

Context:
{context}

Question: {input}
""")
qa_chain = create_stuff_documents_chain(llm=llm, prompt=qa_prompt)

# Step 5: Full decomposition RAG pipeline
def full_query_decomposition_rag_pipeline(user_query):
    # 1. Decompose into sub-questions
    sub_qs_text = decomposition_chain.invoke({"question": user_query})
    sub_questions = [
        q.strip("-•1234567890. ").strip()
        for q in sub_qs_text.split("\n") if q.strip()
    ]

    # 2. Retrieve + answer each sub-question independently
    results = []
    for subq in sub_questions:
        docs = retriever.invoke(subq)
        result = qa_chain.invoke({"input": subq, "context": docs})
        results.append(f"Q: {subq}\nA: {result}")

    # 3. Combine the partial answers
    return "\n\n".join(results)

# Step 6: Run
query = "How does LangChain use memory and agents compared to CrewAI?"
final_answer = full_query_decomposition_rag_pipeline(query)
print("✅ Final Answer:\n")
print(final_answer)
Parsing Gotcha

The decomposer returns sub-questions as free-form text (often a numbered or bulleted list, sometimes with a preamble like "Here's a breakdown:"). The list-comprehension strips bullet characters and numbering — but a stray preamble line can sneak in as a bogus "sub-question." In production, prompt the LLM for a strict format (e.g. JSON) or filter out non-question lines before looping.

The Trade-Off

Decomposition is powerful but not free. Its major disadvantage is cost and latency:

Major Disadvantage — More Calls

One decomposed query becomes many LLM calls and many retrieval calls — one decomposition call, then a retrieve + answer pair per sub-question, plus an optional synthesis call. For a 4-way split that's ~9 model/retriever round-trips instead of 2. More accuracy on complex questions, but slower and more expensive. Reserve it for genuinely multi-part queries.

Key Takeaway

Query decomposition splits a complex question into atomic sub-questions, answers each with its own retrieval, then combines them. It unlocks multi-hop reasoning and parallelism — at the cost of extra LLM and retrieval calls.
