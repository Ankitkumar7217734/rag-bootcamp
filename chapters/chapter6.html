Chapter 6: Advanced Chunking & Preprocessing — Semantic Chunking

A chunk cut in the wrong place poisons everything downstream. This chapter is about splitting documents by meaning instead of by character count — so each chunk is a complete, self-contained thought.

Overview

Everything so far has assumed your documents are already sliced into chunks. Chapter 3 used a RecursiveCharacterTextSplitter with a chunk_size and chunk_overlap — fixed knobs that cut text every N characters. That works, but it's blind: it counts characters, not ideas. A 500-character window can slice a sentence in half, split a definition from its example, or glue two unrelated topics into one chunk.

Semantic chunking fixes this. Instead of splitting by length, it splits by meaning — using embedding similarity between sentences to decide where one idea ends and the next begins. The result is chunks that are coherent, not cut off mid-thought.

Why Chunking Quality Decides Everything

Better chunks → better retrieval → better grounding → better answers. Retrieval can only return what chunking produced. If a chunk is half an idea, the LLM gets half a context — and hallucinates the rest. Good chunks are self-contained, contextually rich, and logically separated.

The Problem with Fixed-Size Chunking

Classic splitting takes two parameters and never looks at the text itself:

ParameterMeaningTypical value
chunk_sizeMax characters (or tokens) per chunk500
chunk_overlapCharacters repeated between neighbouring chunks to preserve context50

The overlap is a band-aid — it duplicates a little text at each boundary hoping a split idea survives in one of the two chunks. But the cut point is still arbitrary. Consider a document that drifts across three topics:

1. LangChain is a framework for building LLM-powered apps.
2. It integrates with tools like OpenAI and Pinecone.
3. The Eiffel Tower is located in Paris.
4. France is a popular tourist destination.

A fixed-size splitter might pack sentences 2 and 3 into the same chunk simply because they fit the character budget — fusing "LLM tooling" with "Paris landmarks." Semantic chunking instead notices that sentence 3 is about something else and starts a fresh chunk there:

Chunk 1: ["LangChain is a framework...", "It integrates with tools..."]
Chunk 2: ["The Eiffel Tower is located in Paris.",
          "France is a popular tourist destination."]

Same input, but now each chunk holds one coherent idea — exactly what makes retrieval accurate.

How Semantic Chunking Works

The whole technique is five steps. Segment the document into sentences, embed each one, measure how similar each sentence is to its neighbour, and start a new chunk wherever that similarity drops below a threshold.

Step 1 — Document Segmentation

Split the document into its smallest natural units — usually sentences (or paragraphs). This is the granularity at which we'll decide boundaries.

Step 2 — Sentence Embedding

Convert each sentence into a vector representation using an embedding model — OpenAI, a HuggingFace model, or a Sentence-Transformer. Now every sentence is a point in meaning-space, and sentences about the same topic sit close together (Chapter 4).

Step 3 — Semantic Similarity Check

Walk through adjacent sentences and compute the cosine similarity between each neighbouring pair of embeddings. A high score means the two sentences are about the same thing; a low score means the topic just changed.

Step 4 — Merging of Sentences

Merge adjacent sentences into the same chunk as long as their similarity stays at or above a threshold. The threshold is the one dial you tune — common values sit around 0.75–0.95.

ThresholdEffectResult
High (e.g. 0.95)Only near-identical sentences mergeMany small, tightly-focused chunks
Medium (e.g. 0.80)Related sentences merge, topic shifts breakBalanced, coherent chunks
Low (e.g. 0.60)Loosely related sentences mergeFewer, larger chunks (risk of mixing topics)

Step 5 — Form Chunks

Wherever similarity falls below the threshold, close the current chunk and start a new one. Each finished group of merged sentences becomes one semantic chunk, ready to embed and store.

Worked Example

For sentences S1–S4 with a threshold of 0.80: sim(S1,S2)=0.95 ≥ 0.80 → merge. sim(S2,S3)=0.20 < 0.80 → break. sim(S3,S4)=0.93 ≥ 0.80 → merge. The result is [S1, S2] as Chunk 1 and [S3, S4] as Chunk 2 — the topic shift at S3 became the boundary.

Building a Semantic Chunker from Scratch

The clearest way to understand the algorithm is to write it. The core is a single pass over the sentences, comparing each to the one before it.

from langchain_openai import OpenAIEmbeddings
from sklearn.metrics.pairwise import cosine_similarity
import os, dotenv

dotenv.load_dotenv()

embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",
    dimensions=1536,
    api_key=os.getenv("openai_api_key"),
)

text = """
langchain is a framework for building applications with LangChain.
langchain provides modular abstractions to combine LLM elements with tools like OpenAI and Pinecone.
You can create chains, agents, memory, and retrieval.
The Eiffel Tower in Paris.
France is a popular tourist destination.
"""

# Step 1: split into sentences
sentences = [s.strip() for s in text.split("\n") if s.strip()]

# Step 2: embed each sentence
sentence_embeddings = embeddings.embed_documents(sentences)

# Step 3: parameters
threshold = 0.7
chunks = []
current_chunk = [sentences[0]]

# Step 4: walk adjacent pairs, merge while similar
for i in range(1, len(sentences)):
    sim = cosine_similarity(
        [sentence_embeddings[i - 1]],
        [sentence_embeddings[i]],
    )[0][0]

    if sim >= threshold:
        current_chunk.append(sentences[i])      # same topic → keep merging
    else:
        chunks.append(" ".join(current_chunk))  # topic shift → close chunk
        current_chunk = [sentences[i]]

# Step 5: don't forget the final chunk
chunks.append(" ".join(current_chunk))

for i, c in enumerate(chunks, 1):
    print(f"Chunk {i}: {c}\n")

The logic is exactly the five steps: embed, compare each neighbour, append when similar, and cut a new chunk when the similarity drops below threshold. The line after the loop matters — without it the last group of sentences never gets flushed.

Watch the Final Flush

The loop only closes a chunk when it sees a topic change. The very last chunk never hits a break, so you must append current_chunk once more after the loop ends — a classic off-by-one that silently drops your final sentences if you forget it.

Wrapping It as a Reusable Chunker

To use semantic chunking inside a real pipeline, wrap the algorithm in a class that speaks LangChain's language — taking Document objects in and returning chunked Documents out, so it slots in wherever a text splitter would.

from langchain_core.documents import Document
from openai import OpenAI
from sklearn.metrics.pairwise import cosine_similarity
import os


class ThresholdSemanticChunker:
    def __init__(self, model_name="text-embedding-3-small", threshold=0.7):
        self.client = OpenAI(api_key=os.getenv("openai_api_key"))
        self.model_name = model_name
        self.threshold = threshold

    def _get_embeddings(self, texts):
        response = self.client.embeddings.create(input=texts, model=self.model_name)
        return [item.embedding for item in response.data]

    def chunk(self, text):
        sentences = [s.strip() for s in text.split(".") if s.strip()]
        sentence_embeddings = self._get_embeddings(sentences)

        chunks = []
        current_chunk = [sentences[0]]
        for i in range(1, len(sentences)):
            sim = cosine_similarity(
                [sentence_embeddings[i - 1]],
                [sentence_embeddings[i]],
            )[0][0]
            if sim >= self.threshold:
                current_chunk.append(sentences[i])
            else:
                chunks.append(". ".join(current_chunk) + ".")
                current_chunk = [sentences[i]]
        chunks.append(". ".join(current_chunk) + ".")
        return chunks

    def split_documents(self, docs):
        result = []
        for doc in docs:
            for chunk in self.chunk(doc.page_content):
                # carry the original metadata onto every derived chunk
                result.append(Document(page_content=chunk, metadata=doc.metadata))
        return result

Because it exposes split_documents — the same method name as LangChain's built-in splitters — you can drop it straight into the ingestion pipeline from Chapter 3:

doc = Document(page_content=sample_text)

chunker = ThresholdSemanticChunker(threshold=0.7)
chunks = chunker.split_documents([doc])

for c in chunks:
    print(c.page_content)
Preserve the Metadata

Notice split_documents copies doc.metadata onto each new chunk. Chunking must never lose the source, page number, or tags attached to the original document — that metadata is what powers filtering and citations later in the RAG chain.

The Easy Way — LangChain's SemanticChunker

You don't have to hand-roll this in production. LangChain ships a battle-tested SemanticChunker in langchain_experimental that does the embedding, comparison, and splitting for you — you just hand it an embedding model.

from langchain_openai import OpenAIEmbeddings
from langchain_experimental.text_splitter import SemanticChunker
from langchain_community.document_loaders import TextLoader

# Load a document
loader = TextLoader("langchain_intro.txt")
documents = loader.load()

# Create the semantic chunker
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",
    api_key=os.getenv("openai_api_key"),
)
chunker = SemanticChunker(embeddings, breakpoint_threshold_type="percentile")

chunks = chunker.split_documents(documents)
for i, chunk in enumerate(chunks, 1):
    print(f"\nchunk {i}:\n{chunk.page_content}")

Instead of a single fixed number, SemanticChunker decides the break point statistically from the distribution of similarities in your document. The breakpoint_threshold_type chooses how:

breakpoint_threshold_typeHow it picks a break pointTends to produce
percentile (default)Break where the distance is above the Nth percentile of all gapsBalanced, robust default
standard_deviationBreak where the distance exceeds mean + N·σFewer breaks — only sharp topic shifts
interquartileBreak using the interquartile range of distancesResistant to outlier sentences
gradientBreak on the rate of change of similarity — good for dense, technical textSensitive to gradual drift

For the LangChain-and-Eiffel-Tower text, the chunker correctly groups the three LangChain sentences together and splits off the two Paris/France sentences — without you ever setting a character count.

Plugging Semantic Chunks into a RAG Pipeline

Semantic chunking is a drop-in replacement for the splitting step — everything downstream (embed → store → retrieve → generate) is unchanged from Chapter 5. Here is the full modular flow, from chunks to a grounded answer:

from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
from langchain.chat_models import init_chat_model
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableMap

# 1. Semantic chunks → vector store
embeddings = OpenAIEmbeddings(model="text-embedding-3-small",
                              api_key=os.getenv("openai_api_key"))
vectorstore = FAISS.from_documents(chunks, embeddings)
retriever = vectorstore.as_retriever()

# 2. Prompt grounded strictly in retrieved context
prompt = PromptTemplate.from_template(
    """Answer the question based only on the following context:

{context}

Question: {question}

Answer:"""
)

# 3. LCEL chain: retrieve → prompt → LLM → text
llm = init_chat_model(model="groq:llama-3.3-70b-versatile", temperature=0)

rag_chain = (
    RunnableMap({
        "context": lambda x: retriever.invoke(x["question"]),
        "question": lambda x: x["question"],
    })
    | prompt
    | llm
    | StrOutputParser()
)

result = rag_chain.invoke({"question": "what is langchain used for"})
print(result)

Because the chunks are semantically coherent, the retriever returns whole ideas rather than fragments — so the LLM answers from complete context instead of stitching together half-sentences.

When to Use Semantic Chunking

Semantic chunking is more accurate, but it isn't free — it calls an embedding model for every sentence during ingestion, which costs time and money that fixed-size splitting doesn't.

Fixed-size chunkingSemantic chunking
Splits onCharacter / token countMeaning (embedding similarity)
Boundary qualityArbitrary — can cut mid-ideaAligned with topic shifts
SpeedInstantSlower — embeds every sentence
CostFreeEmbedding API calls at ingest time
Best forUniform text, quick prototypes, huge corporaMixed-topic docs where retrieval quality matters
A Practical Rule

Reach for semantic chunking when documents weave through several topics and answer quality is the priority — knowledge bases, research papers, long articles. Stick with fixed-size chunking for uniform, single-topic text or when you're ingesting at massive scale and the per-sentence embedding cost adds up. Many teams do both: a fast splitter for bulk content, semantic chunking for the high-value documents.

Key Takeaways

Remember These Four

1. Semantic chunking splits a document by meaning, not character count — keeping each chunk a complete, self-contained idea instead of a fragment cut off mid-thought.

2. The algorithm is five steps: segment into sentences → embed each → compare adjacent cosine similarities → merge while above a threshold → start a new chunk at every drop.

3. The threshold is the dial: higher = more, smaller chunks; lower = fewer, larger chunks. LangChain's SemanticChunker picks it statistically via breakpoint_threshold_type (percentile, standard_deviation, interquartile, gradient).

4. It's a drop-in for the splitting step — embed, store, retrieve, and generate stay exactly as in Chapter 5 — but it trades ingest-time speed and embedding cost for sharper retrieval, so use it where quality matters most.
