Chapter 9: Multimodal RAG — Going Beyond Text

Documents aren't just walls of text. Charts, diagrams, screenshots, and photos carry critical information — and a RAG pipeline that ignores them leaves answers half-blind.

Overview

Every chapter so far has worked with text — text chunks, text embeddings, text retrieval. But real-world documents are multimodal: a PDF report has bar charts next to bullet points; a product catalog pairs descriptions with photos; a research paper includes figures, tables, and equations alongside prose.

A standard text-only RAG pipeline drops all images on the floor. It strips out charts, screenshots, and diagrams, then answers questions using only the words. When a user asks "What does the revenue chart show?", text-only RAG is guessing from nearby captions — it never actually sees the chart.

Multimodal RAG fixes this. It extracts both text and images from documents, embeds them into a shared vector space using a model that understands both modalities (like CLIP), and retrieves them together. Then a vision-capable LLM like GPT-4V reads the retrieved text and images to produce a fully-grounded answer.

What is Multimodal RAG?

Multimodal RAG extends retrieval-augmented generation to handle text AND images in a unified pipeline. A multimodal embedding model (CLIP) maps both into the same vector space; a vision-capable LLM (GPT-4V) processes both modalities to generate answers grounded in the full document — not just its words.

Why Multimodal RAG Matters

Consider a financial analyst asking "What was the Q3 revenue trend?" about a PDF report. A text-only RAG would retrieve paragraphs mentioning Q3 revenue and generate an answer from those words alone. But the report's bar chart — showing three quarters of accelerating growth with exact figures — carries far more information than any sentence. Without it, the answer is incomplete at best, misleading at worst.

The Core Idea

Multimodal RAG uses a unified embedding model (CLIP) that maps text and images into the same 512-dimensional vector space. A query about either modality can retrieve the other — ask in words, get back charts. Show an image, get back related text. Then a vision-capable LLM (GPT-4V) reads both together to answer.

The CLIP Model — A Shared Space for Text and Images

CLIP (Contrastive Language-Image Pre-training) is the engine that makes multimodal RAG possible. It was trained on 400 million image-text pairs from the internet, learning to map images and their captions to nearby points in a shared vector space.

How CLIP Works

CLIP has two encoders — one for text, one for images — that produce embeddings in the same 512-dimensional space. During training, it pushes matching (image, caption) pairs together and pulls non-matching pairs apart using a contrastive loss:

Initialising CLIP

Step 1 — Extracting Text and Images from PDFs

The pipeline starts with PyMuPDF (fitz), a fast PDF library that can extract both text content and embedded images. For each page, we do two things:

  1. Extract text with page.get_text(), then chunk it with RecursiveCharacterTextSplitter.
  2. Extract images with page.get_images(full=True), convert each to a PIL image, and encode it as base64 for later use with GPT-4V.
import fitz  # PyMuPDF
from langchain_core.documents import Document
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from PIL import Image
import base64, io

pdf_path = "multimodal_sample.pdf"
doc = fitz.open(pdf_path)

all_docs = []
all_embeddings = []
image_data_store = {}

splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=100)

for i, page in enumerate(doc):
    # --- Process text ---
    text = page.get_text()
    if text.strip():
        temp_doc = Document(page_content=text, metadata={"page": i, "type": "text"})
        text_chunks = splitter.split_documents([temp_doc])
        for chunk in text_chunks:
            embedding = embed_text(chunk.page_content)
            all_embeddings.append(embedding)
            all_docs.append(chunk)

    # --- Process images ---
    for img_index, img in enumerate(page.get_images(full=True)):
        xref = img[0]
        base_image = doc.extract_image(xref)
        image_bytes = base_image["image"]

        pil_image = Image.open(io.BytesIO(image_bytes)).convert("RGB")
        image_id = f"page_{i}_img_{img_index}"

        # Store as base64 for GPT-4V
        buffered = io.BytesIO()
        pil_image.save(buffered, format="PNG")
        img_base64 = base64.b64encode(buffered.getvalue()).decode()
        image_data_store[image_id] = img_base64

        # Embed image using CLIP
        embedding = embed_image(pil_image)
        all_embeddings.append(embedding)

        image_doc = Document(
            page_content=f"[Image: {image_id}]",
            metadata={"page": i, "type": "image", "image_id": image_id}
        )
        all_docs.append(image_doc)

doc.close()
Three Critical Actions Per Image

1. Convert the raw PDF image bytes to a PIL Image (RGB).

2. Encode the PIL image as base64 and store it — GPT-4V requires base64-encoded images.

3. Run the image through CLIP to get a 512-dim embedding for retrieval.

Each image gets a Document with type "image" and an image_id key in metadata. The image_data_store dictionary maps those IDs to the actual base64 bytes so we can reconstruct the multimodal message later.

Step 2 — Embedding Text and Images with CLIP

Both modalities go through CLIP to produce L2-normalised 512-dimensional vectors. Normalisation ensures cosine similarity between any two vectors falls in [−1, 1], making cross-modal comparison straightforward.

Text Embedding

import torch

def embed_text(text):
    """Embed text using CLIP — returns L2-normalised 512-dim vector."""
    inputs = clip_processor(
        text=[text], return_tensors="pt", padding=True, truncation=True
    )
    with torch.no_grad():
        outputs = clip_model.get_text_features(**inputs)
        if hasattr(outputs, 'last_hidden_state'):
            features = outputs.last_hidden_state[:, 0, :]
        else:
            features = outputs
        features = features / features.norm(dim=-1, keepdim=True)
        return features.squeeze().numpy()

Image Embedding

def embed_image(image):
    """Embed a PIL image using CLIP — returns L2-normalised 512-dim vector."""
    inputs = clip_processor(images=image, return_tensors="pt")
    with torch.no_grad():
        outputs = clip_model.get_image_features(**inputs)
        if hasattr(outputs, 'last_hidden_state'):
            features = outputs.last_hidden_state[:, 0, :]
        else:
            features = outputs
        features = features / features.norm(dim=-1, keepdim=True)
        return features.squeeze().numpy()
Why Normalise?

L2 normalisation projects every vector onto the unit hypersphere, making cosine similarity equivalent to a simple dot product. This is critical for unified retrieval — without it, an image vector with large magnitudes could dominate text vectors in similarity search, breaking cross-modal retrieval.

Step 3 — Building the Unified FAISS Vector Store

Once all text chunks and images have CLIP embeddings, we build a single FAISS index that holds both modalities side by side. The key insight: since both are in the same 512-dimensional space, FAISS doesn't need to know which is which — it just does nearest-neighbour search.

import numpy as np
from langchain_community.vectorstores import FAISS

embeddings_array = np.array(all_embeddings)

vector_store = FAISS.from_embeddings(
    text_embeddings=[
        (doc.page_content, emb)
        for doc, emb in zip(all_docs, embeddings_array)
    ],
    embedding=None,  # using precomputed CLIP embeddings
    metadatas=[doc.metadata for doc in all_docs]
)

The page content for images is a placeholder like [Image: page_0_img_0] — it won't be shown to the LLM directly. The actual base64 image data lives in image_data_store, keyed by image_id. The metadata "type" field tells us which kind of document we got back after retrieval.

Step 4 — Multimodal Retrieval

Retrieval is a single vector search. The user's question is embedded with CLIP's text encoder and searched against the unified FAISS index. The top-k results are a mix of text and image documents — whichever are most relevant to the query:

def retrieve_multimodal(query, k=5):
    """Unified retrieval using CLIP embeddings for both text and images."""
    query_embedding = embed_text(query)
    results = vector_store.similarity_search_by_vector(
        embedding=query_embedding,
        k=k
    )
    return results
The Magic of Unified Embeddings

Because CLIP maps text and images into the same space, a text query like "revenue chart" can retrieve an image of a bar chart — even though the query is words and the result is pixels. No separate index, no modality switching. The similarity is computed in the shared 512-dimensional space where both live.

Step 5 — Building the Multimodal Message for GPT-4V

Retrieval gives us a list of Documents — some are "text", some are "image". The final step is to assemble them into a single multimodal prompt that GPT-4V can process. Text chunks become text content blocks; image chunks are resolved through image_data_store and sent as base64 image_url blocks:

from langchain_classic.schema.messages import HumanMessage

def create_multimodal_message(query, retrieved_docs):
    """Create a multimodal message with both text and images for GPT-4V."""
    content = []

    content.append({
        "type": "text",
        "text": f"Question: {query}\n\nContext:\n"
    })

    text_docs = [d for d in retrieved_docs if d.metadata.get("type") == "text"]
    image_docs = [d for d in retrieved_docs if d.metadata.get("type") == "image"]

    # Add text context
    if text_docs:
        text_context = "\n\n".join([
            f"[Page {doc.metadata['page']}]: {doc.page_content}"
            for doc in text_docs
        ])
        content.append({
            "type": "text",
            "text": f"Text excerpts:\n{text_context}\n"
        })

    # Add images as base64 data URIs
    for doc in image_docs:
        image_id = doc.metadata.get("image_id")
        if image_id and image_id in image_data_store:
            content.append({
                "type": "text",
                "text": f"\n[Image from page {doc.metadata['page']}]:\n"
            })
            content.append({
                "type": "image_url",
                "image_url": {
                    "url": f"data:image/png;base64,{image_data_store[image_id]}"
                }
            })

    content.append({
        "type": "text",
        "text": "\n\nPlease answer the question based on the provided text and images."
    })

    return HumanMessage(content=content)
Content Block TypeWhat It CarriesHow GPT-4V Sees It
"text"Query, text excerpts, page markers, instructionsAs natural language tokens
"image_url"Base64-encoded PNG (data URI)As pixels — the vision encoder reads the chart directly

The message is a single HumanMessage with a list of content blocks. This is the standard OpenAI multimodal chat format: the model processes text blocks with its language head and image blocks with its vision encoder, fusing both into a unified understanding.

Step 6 — The Complete Multimodal RAG Pipeline

Putting it all together — retrieve, assemble the multimodal message, and invoke GPT-4V:

from langchain.chat_models import init_chat_model

llm = init_chat_model("openai:gpt-4.1")

def multimodal_pdf_rag_pipeline(query):
    """Main pipeline for multimodal RAG."""
    # Step 1: Retrieve relevant documents (text + images)
    context_docs = retrieve_multimodal(query, k=5)

    # Step 2: Build multimodal message
    message = create_multimodal_message(query, context_docs)

    # Step 3: Get response from GPT-4V
    response = llm.invoke([message])

    # Print what was retrieved
    print(f"\nRetrieved {len(context_docs)} documents:")
    for doc in context_docs:
        doc_type = doc.metadata.get("type", "unknown")
        page = doc.metadata.get("page", "?")
        if doc_type == "text":
            preview = doc.page_content[:100] + "..."
            print(f"  - Text from page {page}: {preview}")
        else:
            print(f"  - Image from page {page}")

    return response.content

Example Run — Revenue Report with Chart

queries = [

Why CLIP Over Separate Embedders?

You could embed text with a text model and images with an image model, but that gives you two incompatible vector spaces — a text query can only search text, and an image query can only search images. CLIP's entire value proposition is the shared space:

Key Takeaways

Remember These Five

1. Multimodal RAG extends retrieval to handle text AND images in a single pipeline — no more dropping charts and diagrams on the floor.

2. CLIP is the engine: a dual-encoder model that maps text and images into the same 512-dimensional vector space, enabling cross-modal retrieval with a single FAISS index.

3. PDF processing with PyMuPDF extracts both text (chunked with overlap) and images (converted to PIL, then base64). Each gets a CLIP embedding and a LangChain Document.

4. GPT-4V (or any vision-capable LLM) receives a multimodal message — text blocks for the query and retrieved passages, image_url blocks for retrieved charts — and fuses both to generate a fully-grounded answer.

5. The pipeline is ten steps end-to-end: parse → chunk → embed text → embed images → store base64 → build FAISS → embed query → retrieve → build multimodal prompt → generate. The magic is that steps 3 and 4 produce vectors in the same space.
