MMR — Maximal Marginal Relevance

The third hybrid strategy — stop the retriever from returning five copies of the same idea. MMR picks results that are relevant and different from each other, so the LLM gets coverage instead of repetition.

Relevance + Novelty

Run a similarity search and the top results are often near-duplicates — five chunks all making the same point. That wastes the LLM's context window. MMR (Maximal Marginal Relevance) is a diversity-aware technique that balances relevance and novelty, selecting documents that are both:

  • Relevant to the query, and
  • Diverse from each other (non-redundant).

It prevents the retriever from returning very similar documents that repeat the same content.

The MMR Formula

MMR(d) = λ · sim(d, q)  −  (1 − λ) · max sim(d, s)
                                      s ∈ S

  d = candidate document          q = the query
  S = documents already selected   sim = cosine similarity
  λ = trade-off between relevance and diversity

Read it as two competing terms: the first rewards relevance to the query; the second subtracts a penalty for how redundant the candidate is with what you've already picked. The dial is λ (lambda):

λ valueBehaviour
λ = 1Pure relevance — no diversity (↑ relevance)
λ = 0Pure diversity — ignores the query (low λ)
λ = 0.5Balanced — the typical default

Worked Example

Query: "How to use LangChain for RAG". Three candidate documents:

  • D1 — "LangChain enables retrieval with FAISS"
  • D2 — "LangChain can use Chroma and Pinecone too"
  • D3 — "LangChain agents can call external tools"

Step 1 — pick the most relevant. By cosine similarity to the query, D1 (0.95) wins and is selected first.

Step 2 — pick the second with MMR. Now compare D2 and D3, each scored for relevance and penalised for redundancy with the already-selected D1. The inter-document similarities are:

PairSimilarityInterpretation
sim(D1, D2)0.90Redundant — both about vector stores
sim(D1, D3)0.30Diverse — different topic

With λ = 0.7:

MMR(D2) = 0.7 × 0.93  −  0.3 × 0.90  =  0.651 − 0.27  =  0.381
MMR(D3) = 0.7 × 0.80  −  0.3 × 0.30  =  0.560 − 0.09  =  0.470   ← higher

Even though D2 has a higher raw relevance (0.93 vs 0.80), its heavy redundancy penalty drags it below D3. MMR picks D3 because it adds new information. The final ranking is D1, D3 — relevant and diverse — instead of the redundant D1, D2 you'd get from plain similarity search.

MMR in LangChain

You don't compute any of this by hand — every vector store exposes MMR through as_retriever by setting search_type="mmr" (you met this briefly in Chapter 5):

from langchain_community.vectorstores import FAISS

# Build the store as usual, then ask for MMR retrieval
retriever = vectorstore_openai.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 3},          # optionally: "fetch_k": 20, "lambda_mult": 0.7
)

# Drop it straight into the same LCEL RAG chain from Chapter 5
def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

rag_chain = (
    {"context": retriever | format_docs, "input": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

print(rag_chain.invoke("How does LangChain support agents and memory?"))

Under the hood: fetch_k controls how many candidates are pulled before diversifying, k how many survive, and lambda_mult is the λ from the formula.

When to Use MMR

  • In a RAG pipeline — to avoid feeding the LLM redundant documents (richer, more useful context).
  • Chatbots — FAQ bots, search apps, document browsers.
  • When the retriever already returns many results and you need diversity across them.
  • MMR + Hybrid Retrieval — combine diversity with dense + sparse for the strongest setup.
✗ When NOT to use MMRWhy you might skip it
Extremely short context windowYou may just want the top-1 most relevant
You need precision onlyNot focused on coverage
Documents are already diverseNo need to enforce diversity
You're already reranking with an LLMRedundancy is handled by post-filtering
Key Takeaway

MMR trades a little raw relevance for diversity: MMR(d) = λ·sim(d,q) − (1−λ)·max sim(d,s). The λ dial (LangChain's lambda_mult) sets the balance. Reach for search_type="mmr" whenever your top results tend to repeat themselves.

Chapter 7 Recap — All Three Strategies

Hybrid retrieval boosts recall, re-ranking boosts precision, and MMR boosts coverage. They stack into one strong pipeline: hybrid retrieve → re-rank → MMR diversify → LLM — each fixing a different failure mode of plain vector search.
