Chapter 4: Vector Embeddings & Vector Databases

How words and documents become numbers — the embeddings that capture meaning, the math that measures similarity, and the databases built to search them at scale.

Overview

Chapter 3 turned every messy file into a clean list of Document objects. But a computer cannot search text by meaning — it only understands numbers. This chapter is about the bridge between the two: the embedding.

An embedding is a way to translate words into the language computers understand: numbers. Once text lives as vectors, "find me something similar" becomes a geometry problem — and that is exactly what a vector database is built to solve, fast, at the scale of millions of records.

The Core Idea

Traditional databases match exact values — search "cat" and you get rows containing the literal word "cat". A vector database matches meaning — search "cat" and it can surface "kitten", "feline", or "tabby", because those live near each other in vector space.

What Is an Embedding?

Think of an embedding as a way to translate words into numbers — but not random numbers. The numbers are arranged so that things with similar meaning end up close together. Each word becomes a point (a vector) in space.

Real embeddings have hundreds or thousands of dimensions, but the idea is easiest to see in 2D. Here we hand-place a few animals on a grid:

import numpy as np
import matplotlib.pyplot as plt

# Simplified 2D example (real embeddings are higher dimensional)
word_embeddings = {
    "cat":    [1, 5],
    "dog":    [3, 1],
    "mouse":  [5, 8],
    "rabbit": [7, 3],
    "snake":  [9, 11],
    "tiger":  [11, 6],
}

fig, ax = plt.subplots(figsize=(6, 6))
for word, emb in word_embeddings.items():
    ax.scatter(emb[0], emb[1])
    ax.annotate(word, (emb[0], emb[1]))
ax.set_title("Word Embeddings in 2D")
ax.set_xlabel("Embedding Dimension 1")
ax.set_ylabel("Embedding Dimension 2")
ax.grid(True, alpha=0.5)
plt.show()

The position carries meaning. In a real embedding model, "cat" and "kitten" would sit almost on top of each other, while "cat" and "car" would be far apart — even though they share three letters. Embeddings encode meaning, not spelling.

Traditional Database vs Vector Database

Why do we need a new kind of database at all? Because the question we ask of text data has changed.

Traditional DatabaseVector Database

QueryWHERE name = 'cat'"find things like cat"

Great forTransactions, lookupsSearch, recommendations, RAG

A vector database stores embeddings and lets you apply a query vector against all of them to find the closest matches — for example pulling back the vector [0.2, 0.5, 0.7] as the nearest neighbor to your search. That "closest" calculation is where similarity math comes in.

Measuring Similarity — Cosine Similarity

If words are vectors, "how similar are two words?" becomes "how close are two vectors?" The most common measure in RAG is cosine similarity: it measures the angle between two vectors, ignoring their length.

  • Close to 1 → very similar (small angle, same direction)
  • Close to 0 → unrelated (perpendicular)
  • Close to -1 → opposite meaning
import numpy as np

def cosine_similarity(v1, v2):
    """Measures the angle between two vectors.
       ~1 = very similar, ~0 = unrelated, ~-1 = opposite."""
    dot_product = np.dot(v1, v2)
    norm_a = np.linalg.norm(v1)
    norm_b = np.linalg.norm(v2)
    return dot_product / (norm_a * norm_b)

The formula behind that one line is:

                 A · B
cos(θ) = ─────────────────────
            ‖A‖ × ‖B‖

Let's compare a cat to a kitten, and a cat to a car:

cat_vector    = [0.8,  0.6, 0.2]
kitten_vector = [0.9,  0.7, 0.1]
car_vector    = [-0.5, 0.3, 0.2]

print(cosine_similarity(cat_vector, kitten_vector))   # → 0.993  (very similar)
print(cosine_similarity(cat_vector, car_vector))      # → -0.27  (unrelated/opposite)

Cat and kitten score ~0.99 — nearly identical in meaning. Cat and car, despite the spelling, score negative. This single number is the engine behind every semantic search.

Feature Representation — The Movie Analogy

Why do similar things get similar vectors? Because each dimension can be thought of as a feature. Imagine scoring movies on three features — how much Action, Comedy, and Suspense each contains:

MovieActionComedySuspense
Iron Man0.950.20.6
Hulk0.960.40.3
Sherlock Holmes0.60.950.97

Each row is a vector. Iron Man [0.95, 0.2, 0.6] and Hulk [0.96, 0.4, 0.3] both point in the "high-action" direction, so their cosine similarity is high. That is exactly how a service like Netflix recommends: if you loved Iron Man, the nearest vectors — Hulk and other action films — are surfaced next.

The Key Insight

An embedding model learns thousands of these "features" automatically from huge amounts of data — you never hand-label them. The result: meaning becomes geometry, and recommendation, search, and RAG retrieval all reduce to "find the nearest vectors."

Embedding Models

You don't compute embeddings by hand — you use a pre-trained embedding model that has already learned meaning from massive datasets. It takes text, data, or documents in and returns vectors out. Two families dominate:

ProviderRunsCostNotes
HuggingFaceLocally / open-sourceFreeGreat for privacy, offline, and prototyping (e.g. all-MiniLM-L6-v2)

HuggingFace Embeddings (local & free)

The all-MiniLM-L6-v2 model is small, fast, and runs on your own machine — perfect for getting started without an API key.

from langchain_huggingface import HuggingFaceEmbeddings

huggingface_embeddings = HuggingFaceEmbeddings(
    model_name="sentence-transformers/all-MiniLM-L6-v2"
)

text = "Hello, I am a student."
embedding = huggingface_embeddings.embed_query(text)

print(embedding)
print(f"Embedding Length: {len(embedding)}")   # → 384

The model returns a 384-dimensional vector — 384 numbers that together represent the meaning of that sentence.

OpenAI Embeddings (hosted API)

OpenAI's models are accessed through an API key (kept safely in a .env file, never hard-coded).

from langchain_openai import OpenAIEmbeddings
import os, dotenv

dotenv.load_dotenv()
openai_api_key = os.getenv("openai_api_key")

embeddings = OpenAIEmbeddings(
    model="text-embedding-ada-002",
    api_key=openai_api_key,
)

text = "Hello, I am a student of Computer Science at the University of Delhi."
embedding = embeddings.embed_query(text)
print(f"Embedding Length: {len(embedding)}")   # → 1536

Batch Embeddings

Embedding one string at a time is wasteful. Use embed_documents() to vectorize a whole list in a single call — far faster for indexing a corpus.

texts = [
    "AI powers modern technology through machine learning and smart data processing",
    "Embeddings capture semantic meaning to enable intelligent search and recommendations",
    "Vector databases store high-dimensional embeddings in a structured manner",
    "Vector databases enable fast and efficient similarity search",
]

vectors = embeddings.embed_documents(texts)
print(len(vectors))         # → 4 (one vector per text)
print(len(vectors[0]))      # → 1536 (dimensions per vector)
embed_query vs embed_documents

Use embed_query() for a single search query, and embed_documents() for the batch of texts you're storing. Some models treat the two slightly differently under the hood, so using the right method matters for retrieval quality.

Putting It Together — Semantic Search

Now we can build the heart of retrieval. Embed a query, embed a set of documents, score every pair with cosine similarity, and return the closest matches. First, notice how paraphrases score highly even with no shared keywords:

sentences = [
    "The sun sets behind the distant mountain peaks",
    "Sunset paints the sky over remote mountain ranges",      # paraphrase ↑
    "A chef prepares fresh pasta in a small Italian restaurant",
    "A cook makes homemade noodles at a family-run Italian eatery",  # paraphrase ↑
    "Astronauts conduct experiments on the International Space Station",
    "Space station astronauts run scientific tests in orbit", # paraphrase ↑
]

vecs = embeddings.embed_documents(sentences)
for i in range(len(vecs)):
    for j in range(i + 1, len(vecs)):
        sim = cosine_similarity(vecs[i], vecs[j])
        print(f"Sentences {i} and {j}: {sim:.4f}")

Each paraphrase pair scores far higher than unrelated sentences — proof the model captures meaning, not words. Now the full search function:

def semantic_search(query, sentences, embeddings, top_k=3):
    """Return the top_k most similar sentences to the query."""
    query_embedding = embeddings.embed_query(query)
    document_embeddings = embeddings.embed_documents(sentences)

    scored = []
    for sentence, doc_emb in zip(sentences, document_embeddings):
        score = cosine_similarity(query_embedding, doc_emb)
        scored.append((score, sentence))

    # Sort by similarity, highest first
    scored.sort(reverse=True, key=lambda x: x[0])
    return scored[:top_k]

query = "What is the weather like in the mountains?"
results = semantic_search(query, sentences, embeddings, top_k=3)
for score, sentence in results:
    print(f"{score:.4f}  →  {sentence}")

The query shares almost no words with the documents, yet the two sunset/mountain sentences rise to the top. This is semantic search — and it is precisely the retrieval step that powers a RAG pipeline. The next chapter takes these vectors and stores them in a real vector database for search at scale.

Key Takeaways

Remember These Four

1. An embedding turns text into a vector of numbers where similar meaning → nearby points.

2. Cosine similarity measures the angle between two vectors: ~1 similar, ~0 unrelated, ~-1 opposite.

3. Use an embedding model (HuggingFace for local/free, OpenAI for hosted quality) — never compute vectors by hand. Batch with embed_documents().

4. Semantic search = embed the query, compare to stored vectors, return nearest neighbors. Vector databases make this fast at scale.
