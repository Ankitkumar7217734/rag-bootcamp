Iterative Retrieval

Don't settle for the first batch of documents. Reflect on the answer, rewrite the query, and retrieve again until the context is good enough.

Overview

In 15.2, the graph looped back to retrieval when the answer failed the reflector's check — but it searched with the same query every time. If the original question was vague or poorly phrased, retrying with identical words won't help. You need to rewrite the query before searching again.

Iterative Retrieval adds a refine node that sits between reflection and the next retrieval pass. When the reflector says the answer is incomplete, the refine node asks the LLM for a better search query, stores it in refined_question, and the retriever uses that on the next pass. Combined with the self-reflection pattern from 15.2, you get a graph that thinks about its answer and improves its search strategy.

Where this fits

From 6-iterativeretrieval.ipynb, building block 4 of 5. Next in 15.5: pulling from four different sources and synthesizing one answer.

Why Use It?

In standard RAG, a single retrieval is done and the LLM answers from it — if the documents were incomplete or irrelevant, the answer fails silently. In iterative RAG, the agent reflects on the retrieved content and the answer it produced; if it is unsure, it refines its search like a human researcher would, then retrieves with the improved query.

Trace the loop on the right side of the diagram: reflect says NO → refine rewrites the query → retrieve searches again with the new wording → answer → reflect again. The amber arrow back up to retrieve is the iterative part. The green path down to END fires when the answer is good enough or we've hit the attempt cap.

Step 1 — Setup & State

Same ingestion as 15.2 (sample_docs.txt). The state adds two fields that 15.2 didn't have: refined_question holds the rewritten search query, and verified is a boolean the reflector sets.

Imports, LLM, and retriever

import os
from typing import List
from pydantic import BaseModel
from langchain.chat_models import init_chat_model
from langchain_openai import OpenAIEmbeddings
from langchain_classic.schema import Document
from langchain_classic.vectorstores import FAISS
from langchain_community.document_loaders import TextLoader
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from langgraph.graph import StateGraph, END

llm = init_chat_model("openai:gpt-4o")

docs = TextLoader("sample_docs.txt", encoding="utf-8").load()
chunks = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50).split_documents(docs)
vectorstore = FAISS.from_documents(chunks, OpenAIEmbeddings())
retriever = vectorstore.as_retriever()

The state schema

class IterativeRAGState(BaseModel):
    question: str
    refined_question: str = ""
    retrieved_docs: List[Document] = []
    answer: str = ""
    verified: bool = False
    attempts: int = 0
FieldWhen it's setWhat it does
refined_questionAfter refine_query runsThe improved search query for the next retrieval pass. Empty on the first pass.
verifiedAfter reflect_on_answer runsTrue if the reflector said the answer is sufficient.
attemptsEach time generate_answer runsSafety counter — loop stops when this reaches 2.

Step 2 — The Nodes

Four nodes form the loop. The retrieve and answer nodes are familiar. The reflect node checks quality. The refine node is the new piece — it only runs when reflection fails.

Node 1 — retrieve_docs

The key line is state.refined_question or state.question. On the first pass, refined_question is empty, so the retriever uses the original question. On later passes, it uses the rewritten query from the refine node. 

def retrieve_docs(state: IterativeRAGState) -> IterativeRAGState:
    query = state.refined_question or state.question
    docs = retriever.invoke(query)
    return state.model_copy(update={"retrieved_docs": docs})

Node 2 — generate_answer

Always answers the original question (state.question), even if retrieval used a refined query. The refined query is only for finding better documents — the user's question hasn't changed. Also increments attempts. 

def generate_answer(state: IterativeRAGState) -> IterativeRAGState:
    context = "\n\n".join(doc.page_content for doc in state.retrieved_docs)
    prompt = f"""Use the following context to answer the question:

Context:
{context}

Question:
{state.question}
"""
    response = llm.invoke(prompt.strip()).content.strip()
    return state.model_copy(update={"answer": response, "attempts": state.attempts + 1})

Node 3 — reflect_on_answer

Ask the LLM whether the answer is complete. Sets verified = True if the response contains “yes”. This is a simpler check than 15.2's structured format — in production you'd want stricter parsing, but it works for the demo. 

def reflect_on_answer(state: IterativeRAGState) -> IterativeRAGState:
    prompt = f"""
Evaluate whether the answer below is factually sufficient and complete.

Question: {state.question}
Answer: {state.answer}

Respond 'YES' if it's complete, otherwise 'NO' with feedback.
"""
    feedback = llm.invoke(prompt).content.lower()
    verified = "yes" in feedback
    return state.model_copy(update={"verified": verified})

Node 4 — refine_query

Only reached when reflection fails. The LLM sees the original question and the incomplete answer, then suggests a better search query. That suggestion goes into refined_question, and the graph loops back to retrieve with it. 

def refine_query(state: IterativeRAGState) -> IterativeRAGState:
    prompt = f"""
The answer appears incomplete. Suggest a better version of the query that would help
retrieve more relevant context.

Original Question: {state.question}
Current Answer: {state.answer}
"""
    new_query = llm.invoke(prompt).content.strip()
    return state.model_copy(update={"refined_question": new_query})

Step 3 — Build the Loop

Wire the four nodes and add the conditional edge after reflect. Unlike 15.2, the “retry” path goes to refine first (not straight back to retrieve), and the conditional can route directly to END when done. 

builder = StateGraph(IterativeRAGState)

builder.add_node("retrieve", retrieve_docs)
builder.add_node("answer", generate_answer)
builder.add_node("reflect", reflect_on_answer)
builder.add_node("refine", refine_query)

builder.set_entry_point("retrieve")
builder.add_edge("retrieve", "answer")
builder.add_edge("answer", "reflect")

builder.add_conditional_edges(
    "reflect",
    lambda s: END if s.verified or s.attempts >= 2 else "refine"
)

builder.add_edge("refine", "retrieve")
graph = builder.compile()

Follow one iteration mentally. Pass 1: retrieve with original query → answer → reflect says NO → refine writes a better query. Pass 2: retrieve with refined query → answer → reflect says YES → END. If reflect keeps saying NO, the attempts >= 2 guard forces exit anyway.

PassSearch query usedWhat changes
1Original questionFirst retrieval; answer may be thin.
2Refined questionBetter-targeted retrieval after refine node.

Step 4 — Run It

The test query is deliberately vague — “agent loops and transformer-based systems?” — to show how the graph handles an underspecified question. On a good knowledge base, one pass is often enough. 

query = "agent loops and transformer-based systems?"
initial_state = IterativeRAGState(question=query)
final = graph.invoke(initial_state)

print("✅ Final Answer:\n", final["answer"])
print("\n🧠 Verified:", final["verified"])
print("🔁 Attempts:", final["attempts"])
Output — One Pass Was Enough

Final Answer: Agent loops (think → act → observe) and transformer-based systems are complementary. The agent loop provides adaptability and real-time decision-making; transformers supply the heavy language understanding inside the agent's thinking phase — powering memory (summarising past dialogue), planning (predicting future states), and tool interaction (parsing commands).

Verified: True  |  Attempts: 1 — the first retrieval was sufficient, so the refine loop never fired.

When Attempts: 1, the refine node never ran — that's a success, not a failure. The loop is there for when the first pass isn't enough. To see it fire, try a question your knowledge base can't answer well on the first try.

Key Takeaways

Remember These Three

1. Iterative retrieval fixes bad queries, not just bad answers — the refine node rewrites what you search for.

2. retrieve uses refined_question or question so the first pass uses the original and later passes use the rewrite.

3. Same safety pattern as 15.2: attempts >= 2 caps the loop.
