HyDE — Hypothetical Document Embeddings

The third technique, and the cleverest — don't search with the question at all. Have an LLM imagine the answer first, then search with that. Answers look like documents; questions don't.

What is HyDE?

HyDE (Hypothetical Document Embeddings) is a retrieval technique where, instead of embedding the user's query directly, you first generate a hypothetical answer (document) to the query using an LLM — and then embed that hypothetical document to search your vector store.

Why does this help? Because of a subtle mismatch: a question ("When was Steve Jobs fired from Apple?") and the passage that answers it ("Jobs was ousted from Apple in September 1985 after a boardroom struggle…") use very different words. Embedding the question searches for text that looks like a question. Embedding a hypothetical answer searches for text that looks like an answer — which is exactly what your documents are.

The Core Idea

HyDE bridges the gap between user intent and relevant content by matching on answer-style text rather than question-style text. The hypothetical answer doesn't even need to be factually correct — it just needs to look like the right kind of document so the embedding lands near the real ones.

It's especially useful when:

  1. Queries are short — too little signal to embed well.
  2. There's a language mismatch between the query and the documents.
  3. You want to retrieve based on answer content, not question words.

How HyDE Works

The query goes to an LLM that writes a hypothetical answer. That answer — not the original query — is embedded and sent to the retriever. The retriever returns the real top-k documents, which the LLM then uses to produce the grounded final answer.

HyDE from Scratch

The manual version makes the idea concrete: write a function that asks the LLM for a hypothetical answer, then feed that text (not the query) to the retriever.

from langchain_core.output_parsers import StrOutputParser
from langchain.prompts.chat import SystemMessagePromptTemplate, ChatPromptTemplate

def get_hyde_doc(query):
    template = """Imagine you are an expert writing a detailed explanation
    on the topic: '{query}'. Create a hypothetical answer for the topic."""

    system_message_prompt = SystemMessagePromptTemplate.from_template(template=template)
    chat_prompt = ChatPromptTemplate.from_messages([system_message_prompt])
    messages = chat_prompt.format_prompt(query=query).to_messages()
    response = llm.invoke(messages)
    return response.content   # the hypothetical document

# Generate a hypothetical answer
query = "When was Steve Jobs fired from Apple?"
print(get_hyde_doc(query=query))
# → "## The Fall and Rise of a Tech Titan ... ousted on September 17, 1985 ..."

# Retrieve using the hypothetical answer, NOT the raw query
matched_docs = base_retriever.invoke(get_hyde_doc(query))
print(matched_docs)

The LLM's generated paragraph reads like an encyclopedia entry — so when embedded, it lands right next to the real Wikipedia passages about Jobs' departure, which a bare question embedding might have missed.

LangChain's Built-In HyDE Embedder

LangChain ships HypotheticalDocumentEmbedder, which wraps the whole "generate → embed" step into a drop-in embedding function. You hand it an LLM and a base embeddings model, and it transparently does HyDE for every query.

from langchain.chains.hyde.base import HypotheticalDocumentEmbedder
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_community.vectorstores import Chroma

# Base embeddings (the real vectoriser)
base_embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")

# HyDE embedder using a built-in prompt template ('web_search')
hyde_embedding_function = HypotheticalDocumentEmbedder.from_llm(
    llm=llm,
    base_embeddings=base_embeddings,
    prompt_key="web_search",
)

# Store documents using the HyDE embedding function
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=hyde_embedding_function,
    persist_directory="output/langchain",
)

# Now similarity_search transparently uses HyDE under the hood
matched_docs = vectorstore.similarity_search(
    "What memory modules does LangChain provide?", k=4
)

The prompt_key selects one of LangChain's built-in HyDE prompt templates. Per the source code (the PROMPT_MAP), the options are:

Built-in prompt_key values
web_search · sci_fact · arguana · trec_covid
fiqa · dbpedia_entity · trec_news · mr_tydi

Prefer your own wording? Pass a custom_prompt instead of a prompt_key:

from langchain.prompts import PromptTemplate

custom = PromptTemplate.from_template(
    "Generate a concise hypothetical answer for this topic: {query}"
)

hyde_embedding_function = HypotheticalDocumentEmbedder.from_llm(
    llm=llm,
    base_embeddings=base_embeddings,
    custom_prompt=custom,
)

Why Use HyDE?

ProblemHow HyDE helps
User query and documents use different wordsEmbeds answer-style content instead of the question
Query is too vagueLLM-generated answer gives richer semantics
Better grounding neededEmbeds what the answer might look like
Zero-shot retrievalWorks well even without prior training

Benefits of HyDE

FeatureWhy it helps
Semantic intent modellingBetter than literal keyword matching
LLM-aware retrievalQuery is expanded into more context
GeneralisationWorks even if document phrasing differs from the question
Plug-and-playNo retraining; works with OpenAI, Cohere, Hugging Face
The Catch

HyDE adds an extra LLM call before every retrieval (to generate the hypothetical answer), so it costs more latency and tokens. And if the LLM hallucinates wildly off-topic, the embedding can drift. It shines on short or vocabulary-mismatched queries — for already-rich queries, plain embedding may be enough.

Key Takeaway

HyDE embeds a hypothetical answer instead of the question, so retrieval matches on answer-style content. Build it by hand (generate → embed → retrieve) or use LangChain's HypotheticalDocumentEmbedder as a drop-in embedding function with a prompt_key or custom_prompt.

Chapter 8 Recap — All Three Techniques

Query Expansion broadens one query; Query Decomposition splits a complex query into sub-questions; HyDE swaps the question for a hypothetical answer. All three reshape the query before retrieval so the LLM gets better context — better queries in, better answers out.
