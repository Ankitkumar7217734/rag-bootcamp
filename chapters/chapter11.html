Chapter 11: LangGraph — Building Stateful Workflows

Chapter 10 explained what an agent is. This chapter is where you actually build one — wiring an LLM, tools, and memory into a graph that decides its own control flow.

Overview

Up to now every pipeline in this course has been a straight line: load → chunk → embed → retrieve → generate. That works beautifully when the steps never change. But the moment you want the system to loop, branch, call a tool and come back, or pause for a human, a straight line stops being enough. You need a way to describe flow and state.

LangGraph is that tool. It's a library for building stateful, multi-actor applications with LLMs — exactly the agent and multi-agent workflows from the last chapter. Instead of hard-coding the order of steps, you describe your application as a graph: boxes (nodes) that do work, and arrows (edges) that decide what runs next.

The Core Idea

LangGraph models your application as a graph of nodes and edges over a shared state. Each node is a Python function that reads the state and returns an update; each edge decides which node runs next — and edges can be conditional, which is what lets an LLM steer its own control flow. The framework threads one state object through the whole run.

A few facts worth knowing before we build anything:

  • LangGraph is inspired by Pregel and Apache Beam; its public interface draws inspiration from NetworkX.
  • It's built by LangChain Inc. (the creators of LangChain) but can be used without LangChain.
  • It powers production-grade agents, trusted by LinkedIn, Uber, Klarna, GitLab, and many more.

What LangGraph adds on top of a plain script is fine-grained control over both the flow and the state of your application, via a central persistence layer. That persistence unlocks two features common to almost every serious agent:

FeatureWhat the persistence layer gives you
MemoryLangGraph persists arbitrary aspects of your application's state, supporting memory of conversations and other updates within and across interactions.
Human-in-the-loopBecause state is checkpointed, execution can be interrupted and resumed — allowing decisions, validation, and corrections at key stages via human input.

Lang + Graph — the Two Building Blocks

The name says it: Lang(uage models) + Graph. The graph is a DAG — a Directed Acyclic Graph in the simplest case: information flows in one direction through a set of stages. (LangGraph also supports cycles, which is what makes loops and agents possible — more on that in Chapter 11.3.) Every LangGraph is made of just two kinds of thing:

ComponentWhat it isIn code
1 · NodesThe workers. A node is a Python function that receives the state and returns a partial update to it. This is where an LLM call, a tool call, or any logic lives.graph.add_node("name", fn)
2 · EdgesThe flow of information. An edge connects one node to the next. A conditional edge runs a function to choose the next node at runtime.graph.add_edge(a, b) / add_conditional_edges(...)

Two special nodes bookend every graph. START is where user input enters the graph; END is the terminal node where the run stops. You connect your real nodes between them.

The State — the Data That Flows Through

Before adding any node, you define the state. The state schema is the input/output contract for every node and edge in the graph — it describes the structure and types of the data the graph carries. The simplest way to declare one is a TypedDict from Python's typing module, which gives type hints for each key.

from typing import TypedDict

class State(TypedDict):
    graph_info: str

Because the state is a TypedDict with the schema above, every node can read the key with state["graph_info"]. (LangGraph offers more ways to declare state — dataclasses and Pydantic models — which we cover next in Chapter 11.1.)

Building a Simple Graph, Step by Step

Let's build a tiny but complete graph that shows every concept at once: a node that starts the game, then a conditional edge that randomly routes to either a cricket node or a badminton node, then END.

Step 1 — Define the nodes

Each node takes the state and returns a dict that updates a state key. Here every node appends to graph_info:

def start_play(state: State):
    print("Start_Play node has been called")
    return {"graph_info": state["graph_info"] + " I am planning to play"}

def cricket(state: State):
    print("Cricket node has been called")
    return {"graph_info": state["graph_info"] + " Cricket"}

def badminton(state: State):
    print("Badminton node has been called")
    return {"graph_info": state["graph_info"] + " Badminton"}

Step 2 — Define the conditional router

A conditional edge is just a function that returns the name of the next node. The return type is annotated with Literal[...] to list the possible destinations. Here we flip a coin:

import random
from typing import Literal

def random_play(state: State) -> Literal["cricket", "badminton"]:
    if random.random() < 0.5:
        return "cricket"
    else:
        return "badminton"

Step 3 — Construct and compile the graph

Now we wire it together with StateGraph. We add the nodes, connect START to the first node, attach the conditional edge off start_play, and route both game nodes to END. Finally we compile() — which runs a basic structural check — and visualise it as a Mermaid diagram.

from IPython.display import display, Image
from langgraph.graph import StateGraph, START, END

# Build the graph
graph = StateGraph(State)

# Add the nodes
graph.add_node("start_play", start_play)
graph.add_node("cricket", cricket)
graph.add_node("badminton", badminton)

# Add the edges
graph.add_edge(START, "start_play")
graph.add_conditional_edges("start_play", random_play)
graph.add_edge("cricket", END)
graph.add_edge("badminton", END)

# Compile the graph to check for errors
graph_builder = graph.compile()

# Visualize the graph
display(Image(graph_builder.get_graph().draw_mermaid_png()))

Step 4 — Invoke the graph

To run the graph, call .invoke() with an initial state matching the schema. LangGraph feeds it through START and returns the final state:

graph_builder.invoke({"graph_info": "Hey my name is Ankit."})

On one run, the printed trace and result show the start node firing, then the coin landing on cricket:

Start_Play node has been called
Cricket node has been called

{'graph_info': 'Hey my name is Ankit. I am planning to play Cricket'}

Run it again and the conditional edge may route you to badminton instead — that's the whole point of add_conditional_edges: the path is decided at runtime, not fixed in advance. Swap random_play for a function that asks an LLM which way to go, and you've built the core of an agent.

The Five-Step Recipe for Every LangGraph

1. Define the state (a TypedDict / dataclass / Pydantic model). 2. Write the nodes as functions that take state and return an update. 3. Add nodes and edges (plain or conditional) to a StateGraph. 4. compile() to validate. 5. invoke() (or stream()) with an initial state. Every example in this chapter is a variation on these five steps.
