Chapter 12: Agents — Tools, ReAct & the Agentic Loop

Chapter 11 gave the LLM a graph to run in. This chapter gives it hands — real tools it can call, a loop to call them in, and a memory that survives between turns. This is where a chatbot becomes an agent.

Overview

So far the LLM has only ever talked. It can reason about text, but it can't search arXiv, look something up on Wikipedia, fetch today's news, or even reliably multiply two numbers. An agent fixes that: it's an LLM wired to a set of tools, running inside a loop that lets it decide for itself which tool to call, read the result, and decide what to do next — over and over — until the task is done.

This chapter builds exactly that, on top of LangGraph. We'll cover the four building blocks (chains, routers, tools, agents), the ReAct architecture that ties them together, how to write and bind real tools, how the ToolNode + tools_condition loop works, how a checkpointer gives the agent memory across turns, and how LangSmith lets you watch the whole thing run.

The Core Idea

An agent is an LLM that controls a loop. The model is the "brain"; the tools are its hands. On every turn it looks at the conversation and either answers directly (and the loop ends) or emits a tool call. The tool runs, its output is fed back in, and the model is asked again — act, observe, reason — until it has what it needs to respond.

Here is the shape of the agent we're going to build — the LLM sits at the centre, a conditional edge routes to tools whenever the model asks for one, and the tool result loops back into the LLM instead of ending the run:

Notice the cycle: tools flows back into the LLM, not into END. That single back-edge is the difference between a one-shot chain and an agent that can chain several tool calls together to solve a multi-step request.

From Chains to Agents — Four Building Blocks

It helps to see how an agent is assembled from simpler pieces you've already met. Each row adds one capability to the one above it:

#Building blockWhat it doesWho decides the path?
1ChainA fixed sequence of steps — prompt → LLM → parser. Always runs the same way.You (hard-coded)
2RouterA conditional edge picks one of several branches based on the input, then stops.The model, once
3ToolsExternal functions the LLM can call — search, math, APIs — that extend it beyond text.The model chooses which
4AgentA router with a loop: the model calls tools, observes results, and re-decides — repeatedly.The model, every turn

The jump from router (Chapter 11.3) to agent is precisely the loop we drew above. A router answers or calls a tool once; an agent keeps going until it's satisfied.

ReAct — the General Agent Architecture

ReAct (Reason + Act) is the general-purpose pattern behind almost every modern agent. It breaks each turn of the loop into three moves:

StepNameWhat happens
1ActThe model calls one or more specific tools — it decides the name and the arguments.
2ObserveEach tool runs; its output is passed back to the model as a tool message.
3ReasonBased on the tool's output and the input, the model decides what to do next — call another tool, or answer directly and finish.

That's the entire intuition. Everything below is just code that realises this loop in LangGraph.

Tools — Giving the LLM Hands

A tool is just a Python function the LLM is allowed to call. In LangChain you turn a function into a tool with the @tool decorator. Two things matter enormously: the docstring (the model reads it to decide when to use the tool) and the type hints (they define the arguments the model must fill in).

A custom tool — searching arXiv

import arxiv
from langchain_core.tools import tool
from langchain_community.tools import WikipediaQueryRun
from langchain_community.utilities import WikipediaAPIWrapper

@tool
def arxiv_search(query: str) -> str:
    """Search arXiv for academic papers related to the query."""
    client = arxiv.Client()
    search = arxiv.Search(
        query=query,
        max_results=2,
        sort_by=arxiv.SortCriterion.Relevance
    )
    results = []
    for paper in client.results(search):
        results.append(
            f"Title: {paper.title}\n"
            f"Summary: {paper.summary[:500]}\n"
            f"URL: {paper.entry_id}"
        )
    return "\n---\n".join(results) if results else "No results found."

print(arxiv_search.name)   # -> arxiv_search

Once decorated, the tool exposes a .name and an .invoke() method so you can test it directly, exactly as the agent will call it:

arxiv_search.invoke("Attention is all you need")
# 'Title: Do You Even Need Attention? ... URL: http://arxiv.org/abs/2105.02723v1 ...'

Plain functions as tools too

Math is something LLMs are famously bad at, so we hand it off to real functions. Note the careful docstrings and type hints — they are the tool's interface for the model:

def multiply(a: int, b: int) -> int:
    """Multiply a and b.

    Args:
        a: first int
        b: second int
    """
    return a * b

def add(a: int, b: int) -> int:
    """Adds a and b.

    Args:
        a: first int
        b: second int
    """
    return a + b

def divide(a: int, b: int) -> float:
    """Divide a and b.

    Args:
        a: first int
        b: second int
    """
    return a / b

# Combine every tool into one list the LLM can choose from
tools = [arxiv_search, wiki, tavily, add, divide, multiply]
Why the docstring matters

The model never sees your function body — only its name, signature, and docstring. A vague docstring means the model calls the tool at the wrong time or with the wrong arguments. Treat the docstring as a prompt: say precisely what the tool does and what each argument is.

Binding Tools to the LLM

Defining tools isn't enough — the model has to know about them. bind_tools() attaches the tool schemas to the LLM so that, instead of only producing text, the model can produce a structured tool call: a name plus arguments.

from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4-turbo", temperature=0)
llm_with_tools = llm.bind_tools(tools)

Now when we ask something that needs a tool, the model doesn't answer in prose — it returns an AIMessage with an empty content and a populated tool_calls list:

from langchain_core.messages import HumanMessage

llm_with_tools.invoke([HumanMessage(content="What is the recent AI News")]).tool_calls
# [{'name': 'tavily_search_results_json',
#   'args': {'query': 'recent AI news'},
#   'id': 'call_ZYgtqWUMCxsg0lKWB4Unt6Cs',
#   'type': 'tool_call'}]

The model chose Tavily and wrote the query argument itself. That's the "Act" step. But on its own, bind_tools only requests a tool call — nothing has actually run yet. We need a graph to execute the request and feed the result back. Enter LangGraph.

The State — a List of Messages

An agent's state is the conversation itself: a growing list of messages (human, AI, tool). We use the add_messages reducer so each node appends to the list instead of overwriting it — the same pattern from Chapter 11.2.

from typing_extensions import TypedDict
from typing import Annotated
from langchain_core.messages import AnyMessage
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]

Building the Agent Graph

Now we assemble the loop from the diagram. Two prebuilt LangGraph pieces do the heavy lifting:

PieceWhat it does
ToolNode(tools)A ready-made node that reads the latest tool call from the state, runs the matching tool, and appends the result as a tool message.
tools_conditionA ready-made conditional edge: if the last AI message contains a tool call, route to "tools"; otherwise route to END.
 
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode, tools_condition

# The LLM "brain" node
def tool_calling_llm(state: State):
    return {"messages": [llm_with_tools.invoke(state["messages"])]}

# Build the graph
builder = StateGraph(State)
builder.add_node("tool_calling_llm", tool_calling_llm)
builder.add_node("tools", ToolNode(tools))

builder.add_edge(START, "tool_calling_llm")
builder.add_conditional_edges(
    "tool_calling_llm",
    # If the latest message is a tool call -> route to "tools"
    # If it is NOT a tool call          -> route to END
    tools_condition,
)
builder.add_edge("tools", "tool_calling_llm")   # <-- the loop back-edge

graph = builder.compile()

The one line that makes this an agent rather than a router is builder.add_edge("tools", "tool_calling_llm"). Without it, the graph would run a tool once and stop. With it, the tool result flows back into the brain, which can then call another tool — looping until it answers directly and tools_condition finally routes to END.

Trace the loop: the model fired tavily and add together (Act), saw add return 10 (Observe), reasoned that it now needed multiply(10, 10), called it, observed 100, and only then wrote the final answer. It chained three tool calls across multiple passes through the loop — entirely on its own.

A telling failure — why memory matters

Ask the agent "Divide that by 5" in a fresh run and it flounders — it guesses divide(10, 5) and then apologises, because "that" refers to a previous answer it no longer has. Each invoke() starts from a blank state. To say "that number" and be understood, the agent needs to remember the last turn. That's the next section.

Memory — Checkpointers, Threads & Super-Steps

LangGraph can use a checkpointer to automatically save the graph state after each step. This built-in persistence layer is what gives an agent memory: it lets the graph pick up from the last state update instead of starting blank every time.

The easiest one to use is MemorySaver — an in-memory key-value store for graph state. You compile the graph with it and, just like that, the graph remembers:

from langgraph.checkpoint.memory import MemorySaver

memory = MemorySaver()
graph_memory = builder.compile(checkpointer=memory)

To keep separate conversations apart, every call passes a thread id in its config. All the checkpoints sharing a thread_id form one continuous conversation:

config = {"configurable": {"thread_id": "1"}}

# Turn 1
graph_memory.invoke({"messages": [HumanMessage(content="Add 12 and 13.")]}, config=config)
# -> "The result of adding 12 and 13 is 25."

# Turn 2 — "that number" now resolves, because the state persisted
graph_memory.invoke({"messages": [HumanMessage(content="add that number to 25")]}, config=config)
# -> calls add(25, 25) = 50

# Turn 3
graph_memory.invoke({"messages": [HumanMessage(content="then multiply that number by 2")]}, config=config)
# -> calls multiply(50, 2) = 100

Because the checkpointer replays the saved messages on every turn, "that number" resolves correctly — the agent carries 25 → 50 → 100 across three separate invoke() calls. To start a clean conversation, just hand it a different thread_id.

The four concepts of persistence

It's worth naming the pieces precisely, because they show up everywhere in LangGraph:

ConceptMeaning
GraphThe control flow of nodes and edges that transforms an initial state into a final one.
Super-stepOne tick of execution. Each sequential node is its own super-step; nodes that run in parallel share the same super-step.
CheckpointA snapshot of the state plus metadata (the values, what runs next, an id) saved at every super-step. This is what the checkpointer stores.
ThreadA collection of checkpoints sharing one thread_id — i.e. one continuous conversation. Different threads = different, isolated conversations.

Because every checkpoint is saved, two superpowers fall out for free: memory (replay the thread to continue a conversation) and human-in-the-loop (pause at a checkpoint, let a human inspect or edit the state, then resume).

LangSmith — Tracking & Monitoring

Once an agent loops through several tool calls, "it works" or "it doesn't" stops being a useful description — you need to see each step. LangSmith is LangChain's observability platform: with a couple of environment variables it captures every run as a trace you can open in the browser — each LLM call, each tool call, the inputs, the outputs, the latency, and the token cost.

import os
from dotenv import load_dotenv
load_dotenv()

os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_PROJECT"] = "ReAct-agent"
# plus your TAVILY / OPENAI / GROQ keys in the .env file

With tracing on, every invoke() shows up under the ReAct-agent project: you can click into the multi-tool run above and watch the brain → tools → brain loop unfold step by step. For a system where the model decides its own path, that visibility is not a luxury — it's how you debug.

Swapping the brain — provider-agnostic

Nothing in the graph is tied to OpenAI. Swap the brain for an open model on Groq and recompile — the tools, state, loop, and checkpointer are unchanged:

from langchain_groq import ChatGroq → llm = ChatGroq(model="llama-3.3-70b-versatile") → llm_with_tools = llm.bind_tools(tools).

Key Takeaways

Remember These Five

1. An agent is an LLM running a loop — it repeatedly acts (calls a tool), observes (reads the result), and reasons (decides the next step) until it answers. That loop is the ReAct architecture.

2. A tool is a function exposed via @tool; the model picks it by its name, signature, and docstring. bind_tools() lets the LLM emit structured tool_calls instead of plain text.

3. ToolNode runs the requested tool and tools_condition routes "tool call → tools, otherwise → END". The single back-edge tools → llm is what turns a router into an agent.

4. A checkpointer (e.g. MemorySaver) saves a checkpoint of the state at every super-step; checkpoints sharing a thread_id form one conversation. This is what gives the agent memory and enables human-in-the-loop.

5. LangSmith traces every LLM and tool call so you can actually watch — and debug — an agent that chooses its own path.
