Chapter 19: RAG with Persistent Memory using LangGraph

Give your RAG agent a memory. A checkpointer plus a thread id turns a one-shot Q&A into a real conversation that remembers what came before — so “look up common ways of doing it” just works.

Persistent Memory in One Paragraph

Every RAG graph we've built so far forgets everything the moment it returns an answer. Each invoke starts from a blank slate. Persistent memory fixes that: you attach a checkpointer when you compile the graph, and pass a thread id on every call. LangGraph then saves the full message history after each step and reloads it on the next call with the same thread id.

The payoff is a genuine multi-turn assistant. Ask “What is Task Decomposition?”, then follow up with “Can you look up some common ways of doing it?” — and the agent knows “it” means task decomposition, because the earlier turn is still in state. Same graph, one extra argument, and RAG becomes conversational.

Overview

Chapters 13–18 gave an agent the power to decide — when to retrieve, how to grade, when to re-write. But every one of those graphs was stateless between calls. Run it twice and the second run has no idea the first ever happened. That's fine for a single question; it falls apart the instant a user asks a natural follow-up.

This chapter adds the missing ingredient: short-term (conversational) memory. It's built straight from ragmemory.ipynb and comes in two flavors that share the exact same memory mechanism:

  • A custom three-node graph — query_or_respond → tools → generate — compiled with a MemorySaver checkpointer, so it can answer directly or retrieve, and remembers the whole thread.
  • The prebuilt ReAct agent (create_react_agent) with the same checkpointer — less code, and it can loop tool calls until it's satisfied.

The two compiled LangGraph diagrams below are the ones this chapter reproduces faithfully — the custom graph and the ReAct agent, exactly as graph.get_graph().draw_mermaid_png() rendered them in the notebook.

The Core Idea

Memory in LangGraph is not a feature you code inside your nodes — it's infrastructure you switch on at compile time. compile(checkpointer=memory) plus a thread_id is the entire mechanism. The graph's MessagesState already accumulates messages within one run; the checkpointer persists that accumulated state between runs and keys it by thread, so different users (or different conversations) never bleed into each other.

How Persistent Memory Works

Two pieces do all the work. A checkpointer is a place to save state — MemorySaver keeps it in RAM (great for notebooks); production swaps in a SQLite or Postgres saver so memory survives restarts. A thread id is the key under which one conversation's checkpoints are stored. Same thread id → same running conversation; a new thread id → a fresh, empty memory.

Follow-ups “look up ways of doing it” → “it” is unknown Resolves “it” from earlier turns
What you change — compile(checkpointer=memory) + a thread_id in config
Multiple users Can't tell them apart One thread id per conversation keeps them isolated

Step 1 — The Model & Embeddings

Start with a chat model and an embedding model. init_chat_model is the provider-agnostic constructor — "openai:gpt-5" gives the newest OpenAI model, and swapping providers is a one-string change. OpenAIEmbeddings (the text-embedding-ada-002 default) turns text into the vectors we'll search over.

llm = init_chat_model("openai:gpt-5")
from langchain_openai import OpenAIEmbeddings
embeddings = OpenAIEmbeddings()

Step 3 — The retrieve Tool

The whole point of an agentic RAG graph is that retrieval is a tool the model can choose to call, not a step that always fires. Wrapping the FAISS search in @tool exposes it to the LLM; the docstring is what the model reads to decide whether this tool is relevant.

Note it returns a tuple: a human-readable serialized string (which becomes the ToolMessage content the LLM sees) and the raw retrieved_docs (the artifact, available to downstream code). Top-k is 2 — two chunks per lookup. 

from langchain_classic.agents import tool


@tool()
def retrieve(query: str):
    """Retrieve the information related to the query"""
    retrieved_docs = vector_store.similarity_search(query, k=2)
    serialized = "\n\n".join(
        (f"Source: {doc.metadata}\nContent: {doc.page_content}")
        for doc in retrieved_docs
    )
    return serialized, retrieved_docs

Step 4 — The Three Nodes

The graph runs on MessagesState — LangGraph's built-in state whose single messages field uses the add_messages reducer, so returning a message appends it instead of overwriting the list. That accumulating list is exactly what the checkpointer will persist. Three nodes act on it. 

from langchain_core.messages import SystemMessage
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import END, MessagesState, StateGraph
from langgraph.prebuilt import ToolNode, tools_condition

Node 1 — query_or_respond

The decision node. It binds the retrieve tool to the LLM and invokes it on the current message history. The model then does one of two things: answer directly (a plain AIMessage, no tool call — e.g. for “Hello”), or emit a tool call asking to retrieve. Either way, the response is appended to state. 

# Step 1: Generate an AIMessage that may include a tool-call to be sent.
def query_or_respond(state: MessagesState):
    """Generate tool call for retrieval or respond."""
    llm_with_tools = llm.bind_tools([retrieve])
    response = llm_with_tools.invoke(state["messages"])
    # MessagesState appends messages to state instead of overwriting
    return {"messages": [response]}

Node 2 — tools

The executor. ToolNode is a prebuilt that reads the tool call off the last message, runs the matching function, and writes a ToolMessage with the result back into state. No hand wiring — hand it the list of tools and it dispatches. 

# Step 2: Execute the retrieval.
tools = ToolNode([retrieve])

Node 3 — generate

The answer writer, reached only after a retrieval. It has two jobs. First, it walks backwards through the message list to collect the most recent block of ToolMessages — the freshly retrieved context — and stops at the first non-tool message. 

# Step 3: Generate a response using the retrieved content.
def generate(state: MessagesState):
    """Generate answer."""
    # Get generated ToolMessages
    recent_tool_messages = []
    for message in reversed(state["messages"]):
        if message.type == "tool":
            recent_tool_messages.append(message)
        else:
            break
    tool_messages = recent_tool_messages[::-1]

Second, it folds those docs into a system prompt, filters the history down to the real conversation (human/system turns and final AI answers — dropping the intermediate tool-call messages), and asks the LLM for a concise, grounded answer. Returning that message appends it to state, where the checkpointer will save it. 

    # Format into prompt
    docs_content = "\n\n".join(doc.content for doc in tool_messages)
    system_message_content = (
        "You are an assistant for question-answering tasks. "
        "Use the following pieces of retrieved context to answer "
        "the question. If you don't know the answer, say that you "
        "don't know. Use three sentences maximum and keep the "
        "answer concise."
        "\n\n"
        f"{docs_content}"
    )
    conversation_messages = [
        message
        for message in state["messages"]
        if message.type in ("human", "system")
        or (message.type == "ai" and not message.tool_calls)
    ]
    prompt = [SystemMessage(system_message_content)] + conversation_messages

    # Run
    response = llm.invoke(prompt)
    return {"messages": [response]}
Why filter the messages before generating?

The raw state contains the user's question, the AI's tool-call message, and the tool's result. Feeding the tool-call plumbing back to the LLM would confuse it. generate keeps the human turns and any final AI answers, injects the retrieved text as a fresh system message, and drops the intermediate machinery — so the model sees a clean “here's the context, here's the conversation, now answer” prompt.

Step 5 — Wiring & Compiling with a Checkpointer

Now assemble the graph. Entry is query_or_respond. From there, the prebuilt tools_condition router inspects the last message: if it carries a tool call, go to tools; if not, the model already answered, so go straight to END. After tools runs, flow always continues to generate, then ends. 

# Build graph
graph_builder = StateGraph(MessagesState)

graph_builder.add_node(query_or_respond)
graph_builder.add_node(tools)
graph_builder.add_node(generate)

graph_builder.set_entry_point("query_or_respond")
graph_builder.add_conditional_edges(
    "query_or_respond",
    tools_condition,
    {END: END, "tools": "tools"},
)
graph_builder.add_edge("tools", "generate")
graph_builder.add_edge("generate", END)

The one line that grants memory is the last one: compile(checkpointer=memory). Without it, this graph is stateless like every previous chapter's. With it, every super-step's state is written to the MemorySaver and reloaded on the next call with the same thread id. 

memory = MemorySaver()
graph = graph_builder.compile(checkpointer=memory)

This is what graph.get_graph().draw_mermaid_png() renders — the compiled graph. The solid arrows are direct edges (tools→generate→__end__); the dashed arrows are the tools_condition branch out of query_or_respond, which either calls tools or skips straight to __end__.

Step 6 — Threads: Giving the Conversation an ID

A checkpointer stores many conversations. The thread_id in the config selects which one this call belongs to. Reuse the same id to continue a conversation; pick a new id to start a clean one. 

# Specify an ID for the thread
config = {"configurable": {"thread_id": "abc123"}}

First message — a plain greeting. The model answers directly with no tool call, so tools_condition routes straight to END. Nothing to retrieve. 

input_message = "Hello"
for step in graph.stream(
    {"messages": [{"role": "user", "content": input_message}]},
    stream_mode="values",
    config=config,
):

The ReAct Agent Version

The custom graph is explicit and teachable, but LangGraph ships a prebuilt that gives you the same memory-backed, tool-calling behavior in three lines. create_react_agent builds the classic reason → act → observe loop: the agent thinks, optionally calls a tool, reads the result, and loops until it has enough to answer. Pass the same MemorySaver and it's persistent too. 

from langgraph.prebuilt import create_react_agent

memory = MemorySaver()
agent_executor = create_react_agent(llm, [retrieve], checkpointer=memory)

Here's the compiled ReAct agent. The shape is the key difference from the custom graph: after tools runs, a solid edge loops back to agent instead of going to a separate generate node. That back-edge is the ReAct loop — the agent can retrieve, look at what it got, and decide to retrieve again before finally answering.

Give it a new thread id and a two-part question — answer the first part, then use the result to drive a second lookup. The ReAct loop handles the chaining on its own. 

config = {"configurable": {"thread_id": "def234"}}

input_message = (
    "What is the standard method for Task Decomposition?\n\n"
    "Once you get the answer, look up common extensions of that method."
)

for event in agent_executor.stream(
    {"messages": [{"role": "user", "content": input_message}]},
    stream_mode="values",
    config=config,
):
    event["messages"][-1].pretty_print()

Custom Graph vs ReAct Agent

Both remember. Both call the retrieve tool. The choice is about control versus convenience — the same trade-off from Chapter 14, now with memory attached.

AspectCustom GraphReAct Agent (create_react_agent)

Short-term vs long-term memory

The memory in this chapter is short-term — it remembers the current conversation (the thread). It is not the “long-term memory” Lilian Weng's post describes, where an agent stores and recalls facts across separate sessions via a vector store. Persistent short-term memory keeps a chat coherent; long-term memory (a topic for later) lets an agent remember you tomorrow. A production assistant usually wants both.

Key Takeaways

Remember These Four

1. Memory is switched on at compile time. graph.compile(checkpointer=MemorySaver()) is the entire mechanism — no memory code inside your nodes. The checkpointer saves state after every step and reloads it on the next call.

2. The thread_id selects the conversation. Passed in config["configurable"], it keys the stored checkpoints. Same id continues a chat; a new id starts fresh; different ids keep different users isolated.

3. Memory is what makes follow-ups work. “Look up common ways of doing it” only resolves because the earlier turns were reloaded from the checkpointer before the model ran. That's the whole point.

4. Same memory, two graphs. The custom query_or_respond→tools→generate graph gives you full control; create_react_agent gives you a looping ReAct agent in three lines. Both become persistent by adding a checkpointer and a thread id.
