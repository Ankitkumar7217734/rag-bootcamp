Chapter 18: Adaptive RAG — Routing & Self-Correcting Across Sources

Not every question deserves the same pipeline. Route each query to the right source first, then self-correct the answer until it is grounded and actually useful.

Adaptive RAG in One Paragraph

Adaptive RAG is a framework that dynamically adjusts its strategy for handling queries based on their complexity. It's like a smart assistant that knows when to dig deep for information and when to give a simple answer. Instead of one rigid pipeline, Adaptive RAG chooses the most appropriate retrieval method for each query, balancing speed and accuracy.

Concretely, Adaptive RAG unites two ideas: (1) query analysis — route the question to a vectorstore or the web — with (2) active / self-corrective RAG — grade documents, check the answer for hallucinations, and rewrite-and-retry until it holds up.

Overview

Chapter 17's Corrective RAG added one quality gate: grade the documents, and fall back to web search if they're weak. Adaptive RAG generalizes that instinct in two directions at once.

First, it makes a routing decision up front — before retrieving anything, a router LLM looks at the question and decides whether it belongs to the vectorstore (topics the index actually covers) or whether it needs a live web search. Second, after generating, it runs two self-reflection graders: one asks “is this answer grounded in the documents?” (hallucination check) and another asks “does it actually answer the question?” (usefulness check). Failing either sends the graph back to re-write the query or re-generate.

This chapter builds the complete Adaptive RAG graph from 4-AdaptiveRAG_1.ipynb: a query router, three graders (relevance, hallucination, answer), a generator, a re-writer, a Tavily web-search tool, and a LangGraph StateGraph that wires them into a self-correcting loop with a source-routing entry point.

The Core Idea

Corrective RAG always retrieves first, then reacts. Adaptive RAG decides where to look before it looks, and validates the answer twice before returning it. A router sends the question to the vectorstore or the web; the vectorstore path then runs the full self-reflective loop — grade the docs, generate, check for hallucinations, check that it answers the question — rewriting and retrying whenever a gate fails.

The Two Ideas Adaptive RAG Unites

The handwritten notes state it directly: Adaptive RAG is a strategy for RAG that unites (1) query analysis with (2) active / self-corrective RAG. Everything in this chapter is one of those two halves.

HalfWhat it doesBuilt from
1 · Query Analysis (routing) Classifies the incoming question and routes it to the right datasource — the vectorstore for covered topics, live web search for everything else. The RouteQuery router (Step 2)
2 · Active / Self-Corrective RAG Grades retrieved docs, generates, then checks the answer for hallucinations and usefulness — looping back to rewrite or re-generate when a check fails. The three graders + edges (Steps 3, 7)

RAG Chain vs RAG State Machine

The notes contrast the old and new mental models. Traditional RAG is a chain — a straight line with no way to react. Adaptive RAG is a state machine — a graph with decision points that can loop, branch, and recover.

RAG Chain (traditional)RAG State Machine (adaptive)
Shape Question → Index → Retrieval → Generation → Answer Question → Retrieve → Grade → decision → Generate / Re-write → Answer
Control flow Linear, one pass, no checks Cyclic — conditional edges can loop back
On bad retrieval Answers anyway Re-writes the query and retrieves again
On a bad answer Returns it Catches it (hallucination / answer graders) and retries
Single-Step vs Multi-Step vs Adaptive

(A) Single-Step — one retrieval for every query. Fast, but inaccurate on complex, multi-hop questions.

(B) Multi-Step — always iterate several retrieval rounds. Accurate on hard questions, but inefficient — it wastes rounds on “When was X born?”

(C) Adaptive — classify the query first, then spend the right amount of effort: straightforward questions get a direct answer, simple lookups get one retrieval, complex ones get the full iterative loop.

Step 1 — Build the Index

Same knowledge base as Corrective RAG — three of Lilian Weng's blog posts (agents, prompt engineering, adversarial attacks) chunked into a FAISS store. The one change from Chapter 17 is a chunk_overlap of 50 so ideas aren't cut cleanly at chunk boundaries. 

### Build Index

from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.document_loaders import WebBaseLoader
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings

# Set embeddings
embd = OpenAIEmbeddings()

# Docs to index
urls = [
    "https://lilianweng.github.io/posts/2023-06-23-agent/",
    "https://lilianweng.github.io/posts/2023-03-15-prompt-engineering/",
    "https://lilianweng.github.io/posts/2023-10-25-adv-attack-llm/",
]

# Load
docs = [WebBaseLoader(url).load() for url in urls]
docs_list = [item for sublist in docs for item in sublist]

# Split
text_splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
    chunk_size=500, chunk_overlap=50
)
doc_splits = text_splitter.split_documents(docs_list)

# Add to vectorstore
vectorstore = FAISS.from_documents(
    documents=doc_splits,
    embedding=OpenAIEmbeddings()
)

retriever = vectorstore.as_retriever()

Step 2 — The Router (Query Analysis)

This is the piece that makes RAG “adaptive.” Before any retrieval happens, the router reads the question and returns a structured choice: vectorstore or web_search. The Literal type on the Pydantic field constrains the LLM to exactly those two values, and the system prompt tells it what the vectorstore actually contains — that description is the entire basis for the routing decision. 

### Router

from typing import Literal

from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


# Data model
class RouteQuery(BaseModel):
    """Route a user query to the most relevant datasource."""

    datasource: Literal["vectorstore", "web_search"] = Field(
        ...,
        description="Given a user question choose to route it to web search or a vectorstore.",
    )


# LLM with function call
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
structured_llm_router = llm.with_structured_output(RouteQuery)

# Prompt
system = """You are an expert at routing a user question to a vectorstore or web search.
The vectorstore contains documents related to agents, prompt engineering, and adversarial attacks.
Use the vectorstore for questions on these topics. Otherwise, use web-search."""
route_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", system),
        ("human", "{question}"),
    ]
)

question_router = route_prompt | structured_llm_router

print(question_router.invoke({"question": "Who won the Cricket world cup 2023 "}))
print(question_router.invoke({"question": "What are the types of agent memory?"}))
datasource='web_search'
datasource='vectorstore'
Why the router matters

A cricket score isn't in three blog posts about LLMs, so routing it to the vectorstore would guarantee irrelevant retrieval. The router catches that before wasting a retrieval call, sending it straight to the web. “Agent memory,” on the other hand, is squarely in the index — so it takes the full RAG path. One cheap LLM call picks the right pipeline.

Step 3 — The Three Graders

Adaptive RAG has three structured-output graders, each a tiny Pydantic model with a binary yes/no field. Together they are the “self-reflection” in the diagram: one gate before generation, two after.

Retrieval Grader — are the docs relevant?

The same relevance gate as Corrective RAG: score each retrieved document against the question and keep only the relevant ones. The prompt explicitly says it needn't be stringent — the goal is just to filter out erroneous retrievals. 

### Retrieval Grader

# Data model
class GradeDocuments(BaseModel):
    """Binary score for relevance check on retrieved documents."""

    binary_score: str = Field(
        description="Documents are relevant to the question, 'yes' or 'no'"
    )


llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
structured_llm_grader = llm.with_structured_output(GradeDocuments)

system = """You are a grader assessing relevance of a retrieved document to a user question. \n
    If the document contains keyword(s) or semantic meaning related to the user question, grade it as relevant. \n
    It does not need to be a stringent test. The goal is to filter out erroneous retrievals. \n
    Give a binary score 'yes' or 'no' score to indicate whether the document is relevant to the question."""
grade_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", system),
        ("human", "Retrieved document: \n\n {document} \n\n User question: {question}"),
    ]
)

retrieval_grader = grade_prompt | structured_llm_grader
question = "agent memory"
docs = retriever.invoke(question)
doc_txt = docs[1].page_content
print(retrieval_grader.invoke({"question": question, "document": doc_txt}))
binary_score='yes'

Hallucination Grader — is the answer grounded?

New in Adaptive RAG. After generation, this grader checks whether the answer is actually supported by the retrieved facts, or whether the LLM made something up. A no here means “not grounded — re-generate.” 

### Hallucination Grader

class GradeHallucinations(BaseModel):
    """Binary score for hallucination present in generation answer."""

    binary_score: str = Field(
        description="Answer is grounded in the facts, 'yes' or 'no'"
    )


llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
structured_llm_grader = llm.with_structured_output(GradeHallucinations)

system = """You are a grader assessing whether an LLM generation is grounded in / supported by a set of retrieved facts. \n
     Give a binary score 'yes' or 'no'. 'Yes' means that the answer is grounded in / supported by the set of facts."""
hallucination_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", system),
        ("human", "Set of facts: \n\n {documents} \n\n LLM generation: {generation}"),
    ]
)

hallucination_grader = hallucination_prompt | structured_llm_grader
hallucination_grader.invoke({"documents": docs, "generation": generation})
GradeHallucinations(binary_score='yes')

Answer Grader — does it resolve the question?

Also new. An answer can be perfectly grounded and still miss the point. This final gate asks whether the generation actually addresses and resolves the user's question. A no means “grounded but unhelpful — rewrite the query and try again.” 

### Answer Grader

class GradeAnswer(BaseModel):
    """Binary score to assess answer addresses question."""

    binary_score: str = Field(
        description="Answer addresses the question, 'yes' or 'no'"
    )


llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
structured_llm_grader = llm.with_structured_output(GradeAnswer)

system = """You are a grader assessing whether an answer addresses / resolves a question \n
     Give a binary score 'yes' or 'no'. Yes' means that the answer resolves the question."""
answer_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", system),
        ("human", "User question: \n\n {question} \n\n LLM generation: {generation}"),
    ]
)

answer_grader = answer_prompt | structured_llm_grader
answer_grader.invoke({"question": question, "generation": generation})
GradeAnswer(binary_score='yes')

Step 4 — Generate, Re-Writer & Web Search

The three “worker” components. Between the graders in Step 3, these actually produce text.

The Generator

A standard RAG chain, but here the prompt is pulled from the LangSmith hub (rlm/rag-prompt, the canonical concise-answer RAG prompt) instead of being written inline. 

### Generate

The Question Re-Writer

Note the difference from Corrective RAG: there the re-writer optimized the question for web search. Here it optimizes for vectorstore retrieval — because when the loop rewrites, it retries against the same index, not the web.

Step 5 — Graph State

The shared state is smaller than Corrective RAG's — just question, generation, and documents. There's no web_search flag, because routing happens at the graph's entry edge instead of via a state field. 

from typing import List
from typing_extensions import TypedDict


class GraphState(TypedDict):
    """
    Represents the state of our graph.

    Attributes:
        question: question
        generation: LLM generation
        documents: list of documents
    """

    question: str
    generation: str
    documents: List[str]

Step 6 — The Nodes

Five node functions. retrieve, generate, grade_documents, and transform_query are close to their Corrective RAG versions; the new one is web_search, reached directly from the router. Note grade_documents here just filters — it doesn't set a flag, because the emptiness of the filtered list is what the decision edge reads later. 

from langchain_classic.schema import Document


def retrieve(state):
    """Retrieve documents."""
    print("---RETRIEVE---")
    question = state["question"]
    documents = retriever.invoke(question)
    return {"documents": documents, "question": question}


def generate(state):
    """Generate answer."""
    print("---GENERATE---")
    question = state["question"]
    documents = state["documents"]
    generation = rag_chain.invoke({"context": documents, "question": question})
    return {"documents": documents, "question": question, "generation": generation}


def grade_documents(state):
    """Determine whether the retrieved documents are relevant to the question."""
    print("---CHECK DOCUMENT RELEVANCE TO QUESTION---")
    question = state["question"]
    documents = state["documents"]

    # Score each doc
    filtered_docs = []
    for d in documents:
        score = retrieval_grader.invoke(
            {"question": question, "document": d.page_content}
        )
        grade = score.binary_score
        if grade == "yes":
            print("---GRADE: DOCUMENT RELEVANT---")
            filtered_docs.append(d)
        else:
            print("---GRADE: DOCUMENT NOT RELEVANT---")
            continue
    return {"documents": filtered_docs, "question": question}


def transform_query(state):
    """Transform the query to produce a better question."""
    print("---TRANSFORM QUERY---")
    question = state["question"]
    documents = state["documents"]
    better_question = question_rewriter.invoke({"question": question})
    return {"documents": documents, "question": better_question}


def web_search(state):
    """Web search based on the re-phrased question."""
    print("---WEB SEARCH---")
    question = state["question"]
    docs = web_search_tool.invoke({"query": question})
    web_results = "\n".join([d["content"] for d in docs])
    web_results = Document(page_content=web_results)
    return {"documents": web_results, "question": question}

Step 7 — The Decision Edges

Three functions drive every branch in the graph. None of them is a node — each returns a string that the graph maps to the next node. Read them in order and the whole control flow falls out.

route_question — the entry router

Runs at START. It calls the router from Step 2 and returns either "web_search" or "vectorstore" — the two possible first nodes. 

### Edges ###


def route_question(state):
    """Route question to web search or RAG."""
    print("---ROUTE QUESTION---")
    question = state["question"]
    source = question_router.invoke({"question": question})
    if source.datasource == "web_search":
        print("---ROUTE QUESTION TO WEB SEARCH---")
        return "web_search"
    elif source.datasource == "vectorstore":
        print("---ROUTE QUESTION TO RAG---")
        return "vectorstore"

decide_to_generate — after grading

Runs after grade_documents. If the filtered list is empty (every doc was irrelevant), it rewrites the query; otherwise it generates. 

def decide_to_generate(state):
    """Determine whether to generate an answer, or re-generate a question."""
    print("---ASSESS GRADED DOCUMENTS---")
    state["question"]
    filtered_documents = state["documents"]

    if not filtered_documents:
        # All documents have been filtered check_relevance
        # We will re-generate a new query
        print("---DECISION: ALL DOCUMENTS ARE NOT RELEVANT TO QUESTION, TRANSFORM QUERY---")
        return "transform_query"
    else:
        # We have relevant documents, so generate answer
        print("---DECISION: GENERATE---")
        return "generate"

grade_generation_v_documents_and_question — the double check

The most important edge — it runs after generate and applies both new graders in sequence. First the hallucination check: if the answer isn't grounded, return "not supported" and re-generate. If it is grounded, run the answer check: "useful" ends the graph, "not useful" rewrites the query and loops. 

def grade_generation_v_documents_and_question(state):
    """Determine whether the generation is grounded in the document and answers question."""
    print("---CHECK HALLUCINATIONS---")
    question = state["question"]
    documents = state["documents"]
    generation = state["generation"]

    score = hallucination_grader.invoke(
        {"documents": documents, "generation": generation}
    )
    grade = score.binary_score

    # Check hallucination
    if grade == "yes":
        print("---DECISION: GENERATION IS GROUNDED IN DOCUMENTS---")
        # Check question-answering
        print("---GRADE GENERATION vs QUESTION---")
        score = answer_grader.invoke({"question": question, "generation": generation})
        grade = score.binary_score
        if grade == "yes":
            print("---DECISION: GENERATION ADDRESSES QUESTION---")
            return "useful"
        else:
            print("---DECISION: GENERATION DOES NOT ADDRESS QUESTION---")
            return "not useful"
    else:
        pprint("---DECISION: GENERATION IS NOT GROUNDED IN DOCUMENTS, RE-TRY---")
        return "not supported"
Three edges, three recovery moves

Bad retrieval → decide_to_generate returns transform_query → rewrite & retrieve again.

Hallucinated answer → not supported → re-generate from the same documents.

Grounded but unhelpful answer → not useful → rewrite the query and run the loop again.

Step 8 — Wiring the Graph

Now assemble the StateGraph. The distinctive part is the conditional entry edge: instead of add_edge(START, ...), the graph uses add_conditional_edges(START, route_question, ...) so routing is the very first thing that happens. Two more conditional edges (out of grade_documents and generate) create the self-correcting loops. 

from langgraph.graph import END, StateGraph, START

workflow = StateGraph(GraphState)

# Define the nodes
workflow.add_node("web_search", web_search)            # web search
workflow.add_node("retrieve", retrieve)                # retrieve
workflow.add_node("grade_documents", grade_documents)  # grade documents
workflow.add_node("generate", generate)                # generate
workflow.add_node("transform_query", transform_query)  # transform_query

# Build graph
workflow.add_conditional_edges(
    START,
    route_question,
    {
        "web_search": "web_search",
        "vectorstore": "retrieve",
    },
)
workflow.add_edge("web_search", "generate")
workflow.add_edge("retrieve", "grade_documents")
workflow.add_conditional_edges(
    "grade_documents",
    decide_to_generate,
    {
        "transform_query": "transform_query",
        "generate": "generate",
    },
)
workflow.add_edge("transform_query", "retrieve")
workflow.add_conditional_edges(
    "generate",
    grade_generation_v_documents_and_question,
    {
        "not supported": "generate",
        "useful": END,
        "not useful": "transform_query",
    },
)

# Compile
app = workflow.compile()

This is what app.get_graph(xray=True).draw_mermaid_png() renders — the compiled Adaptive RAG graph. Solid arrows are direct edges; dashed arrows are the conditional branches (routing at __start__, decide_to_generate out of grade_documents, and the double-check out of generate).

Read the two traces

The first line of each trace is the router deciding the whole path: TO WEB SEARCH vs TO RAG. After that, the vectorstore route runs the extra CHECK DOCUMENT RELEVANCE step that the web route skips — but both routes converge on the same two-gate finish: CHECK HALLUCINATIONS then GRADE GENERATION vs QUESTION. That shared finish is the “self-reflection” every answer must pass, no matter where the documents came from.

Adaptive vs Corrective vs Agentic RAG

Three chapters, three flavors of “RAG that can recover.” The difference is what decision the system is allowed to make, and when.

AspectAgentic RAG (Ch. 13)Corrective RAG (Ch. 17)Adaptive RAG (Ch. 18)
Source routing LLM picks a retriever tool Always the vectorstore first Router picks vectorstore vs web up front
Doc grading Yes Yes Yes
Correction on weak docs Rewrite & retry Rewrite & go to web Rewrite & re-retrieve (same index)
Answer checks None None Hallucination + answer graders
Control flow Agent loop (model-driven) Deterministic graph Deterministic graph, router entry

Key Takeaways

Remember These Four

1. Adaptive RAG = query analysis + self-corrective RAG. A router chooses the datasource before retrieval; a self-reflective loop validates the answer after generation.

2. Routing happens at the entry edge. add_conditional_edges(START, route_question, ...) sends each question to web_search or retrieve based on a structured RouteQuery decision — no wasted retrieval on out-of-domain questions.

3. Two graders guard every answer. The hallucination grader checks grounding; the answer grader checks usefulness. Failing either loops back to re-generate or re-write — this is the “RAG state machine,” not a one-shot chain.

4. It's a graph, so it can loop. Three conditional edges (route, decide-to-generate, grade-generation) turn RAG into a cyclic state machine that recovers from bad retrieval and bad generation instead of returning either.
