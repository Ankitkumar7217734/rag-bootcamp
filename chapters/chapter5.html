Chapter 5: Vector Storage & Vector Databases

Embeddings are useless sitting in a Python list. This chapter is about where they live — the stores and databases that hold millions of vectors and find the nearest ones in milliseconds.

Overview

Chapter 4 turned text into vectors — lists of numbers where similar meaning lands nearby. But computing an embedding is only half the job. To actually retrieve, you need somewhere to keep those vectors and a way to ask, "of all the vectors I've stored, which are closest to this query?" — fast, even across millions of records.

That "somewhere" is a vector store or a vector database. They take your embedded chunks, build an index over them, and answer nearest-neighbour queries in milliseconds. This chapter covers the difference between the two, the single interface LangChain wraps around all of them, and four concrete backends — ChromaDB, FAISS, Pinecone, and Astra DB — ending with the payoff: plugging a store into a working RAG chain.

The Core Idea

A vector store does one thing brilliantly: given a query vector, it returns the stored vectors closest to it — ranked nearest neighbours, not exact matches. Everything in this chapter is a variation on "store the vectors, then search them." The engines differ in persistence, scale, and features — not in that core job.

Vector Store vs Vector Database

The two terms get used interchangeably, but there's a useful distinction. A vector store is a lightweight library or tool focused on storing and searching vectors efficiently — it finds the K nearest neighbours to a query vector, and not much else. A vector database is a full-featured database system designed for vector data at scale — adding metadata filters, database operations (CRUD), replication, sharding, and high availability on top of that search.

Vector StoreVector Database
What it isLightweight library for storing & searching vectorsFull database system for vector data at scale
Core functionSimple K-nearest-neighbour searchAdvanced search with filters, metadata & CRUD
ArchitectureIn-memory or local file, single machineDistributed — replication, sharding, high availability
Scale~1M vectorsBillions+
Query speedMicroseconds (in-memory)Milliseconds (network + scale)
Setup timeMinutesHours / days
CostFree / $$$$ / $$$$
DeploymentLocalCloud / managed
ExamplesFAISS, Annoy, ScaNN, NMSLIBPinecone, Weaviate, Qdrant, Milvus, Astra DB
Best forPrototypes, research, < 1M vectorsProduction, enterprise, billions of vectors

Notice the counter-intuitive row: a vector store is often faster — microseconds, because everything sits in local memory — while a database answers in milliseconds, the price of scaling to billions of vectors across a network. You trade raw latency for scale, durability, and features.

The Blurry Middle — ChromaDB

The line isn't sharp. ChromaDB is the classic straddler: it runs as a tiny local file like a store, yet offers persistence, metadata, and live updates like a database. Don't over-think the label — choose based on what your workload needs: just fast search over a fixed set of vectors, or a living datastore you keep writing to, filtering, and scaling.

So which side should you reach for? These two checklists mirror the PDF's decision guide:

Use a Vector Store when…Use a Vector Database when…
Building a proof of concept or learningBuilding production applications
Working with fewer than ~1 million vectorsScaling beyond millions of vectors
You need the fastest possible search speedYou require high availability & reliability
You have a limited budgetYou need advanced filtering & metadata search
You want full control over the implementationYou have multiple users / tenants
Building an embedded, single-machine appYou want managed infrastructure
The Simple Rule

Start with a vector store for prototyping and learning. Graduate to a vector database when you need production-scale features, reliability, and advanced querying capabilities.

One Interface, Many Engines

Here is the single most useful thing to know in this chapter: LangChain gives every vector store the same interface. Whether you choose Chroma, FAISS, Pinecone, or Astra, you call the same handful of methods. Switching engines is usually a one-line import change — your retrieval code stays put.

MethodWhat it does
.from_documents(docs, embedding)Embed a list of Documents and build the index in one call
.add_documents(docs)Embed and insert more documents into an existing store
.similarity_search(query, k)Return the k documents closest in meaning to the query
.similarity_search_with_score(query, k)Same, but also return each match's distance score
.as_retriever(...)Wrap the store as a retriever — the piece a RAG chain plugs into

Learn these five once and you've learned every backend below. The rest is just configuration.

ChromaDB — A Local-First Vector Database

ChromaDB is the workhorse for getting started: open-source, runs locally, and — crucially — persists to disk, so you embed your documents once and reopen the database later without paying to re-embed. Let's walk the full pipeline: load → split → embed → store → search.

Build and Persist the Store

We pick up where Chapter 3 left off — raw files become Documents, which we split into chunks, embed, and hand to Chroma. The persist_directory is what makes it durable.

from langchain_community.document_loaders import DirectoryLoader, TextLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma
import os, dotenv

dotenv.load_dotenv()

# 1. Load raw text files into Document objects
loader = DirectoryLoader("data", glob="*.txt", loader_cls=TextLoader)
docs = loader.load()

# 2. Split into overlapping chunks
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

# 3. Choose an embedding model
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",
    api_key=os.getenv("openai_api_key"),
)

# 4. Embed every chunk and store it — persisted to ./chroma_db
vector_store = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",
    collection_name="rag_collection",
)

print(f"Stored {vector_store._collection.count()} vectors")

That one from_documents call did a lot: it embedded each chunk, wrote the vectors plus their text and metadata to ./chroma_db, and built an index for fast search. Reopen the folder later and the data is already there — no re-embedding.

Search by Meaning

Now the database earns its keep. Ask a question in plain English and get back the chunks whose meaning is closest — even when they share no keywords with the query.

query = "What are the types of machine learning?"
results = vector_store.similarity_search(query, k=3)

for doc in results:
    print(doc.page_content[:100], "...")

To see how close each match is, ask for scores. Chroma returns a distance, so lower is better (0 would be identical):

query = "What is NLP?"
results = vector_store.similarity_search_with_score(query, k=3)

for doc, score in results:
    print(f"{score:.4f}  →  {doc.page_content[:80]}...")
    # lower distance = more similar
Distance vs Similarity

Watch the direction of the number. similarity_search_with_score on Chroma and FAISS returns a distance (lower = closer). Some managed databases instead return a similarity (higher = closer). Always check which one your backend gives before you sort or threshold on it.

Add Documents Later

A real knowledge base grows. Because Chroma is a database, you can insert new documents into the existing store without rebuilding it — embed and append:

from langchain_core.documents import Document

new_doc = Document(
    page_content="Reinforcement learning trains an agent to make decisions "
                 "by interacting with an environment to maximize rewards.",
    metadata={"source": "manual_addition", "topic": "reinforcement_learning"},
)

new_chunks = splitter.split_documents([new_doc])
vector_store.add_documents(new_chunks)

print(f"Total vectors now: {vector_store._collection.count()}")

FAISS — In-Memory Speed

FAISS (Facebook AI Similarity Search) is the speed specialist. It's a vector store in the strict sense — an in-memory index built for blazing-fast nearest-neighbour search over millions of vectors, with optional GPU acceleration. It doesn't persist on its own; you save and load it.

from langchain_community.vectorstores import FAISS

# Build the index in memory (same interface as Chroma)
vector_store = FAISS.from_documents(documents=chunks, embedding=embeddings)
print(f"{vector_store.index.ntotal} vectors indexed")

# Persistence is explicit — write the index to disk yourself
vector_store.save_local("faiss_index")

# Reload later. allow_dangerous_deserialization is required because the
# index is a pickle — only load files you created and trust.
vector_store = FAISS.load_local(
    "faiss_index",
    embeddings,
    allow_dangerous_deserialization=True,
)
print(f"Loaded {vector_store.index.ntotal} vectors")

Search works exactly like Chroma — same method names — and you can narrow results with a metadata filter, so only chunks tagged a certain way are eligible:

query = "What is deep learning?"

# Plain top-k search
results = vector_store.similarity_search(query, k=3)

# Only search chunks whose metadata topic == "DL"
filtered = vector_store.similarity_search(query, k=3, filter={"topic": "DL"})
Chroma or FAISS?

Reach for FAISS when raw search speed over a large, mostly-static set of vectors is the priority and you're happy managing save/load yourself. Reach for Chroma when you want a real local database — automatic persistence, easy updates, and metadata baked in — with very little setup.

From Store to Retriever

A vector store knows how to search, but a RAG chain doesn't talk to it directly — it talks to a retriever. Every store converts to one with .as_retriever(), and that's where you choose how it searches.

retriever = vector_store.as_retriever(
    search_type="similarity",      # how to search
    search_kwargs={"k": 3},        # how many to return
)

# A retriever takes a plain string and returns Documents
docs = retriever.invoke("What is deep learning?")

The search_type controls the retrieval strategy:

search_typeBehaviour
similarityPlain top-k nearest neighbours (the default)
similarity_score_thresholdOnly return matches above a score — drops weak, irrelevant hits
mmrMaximal Marginal Relevance — diversifies results so the k chunks aren't near-duplicates
 
# Drop anything below a relevance threshold
retriever = vector_store.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={"k": 3, "score_threshold": 0.5},
)

# Or diversify the results with MMR
retriever = vector_store.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 3},
)

Wiring a Store Into RAG

This is the moment everything has been building toward. Take the retriever, feed its results to a prompt, and let an LLM answer — grounded in your stored documents. The cleanest way to express this is LCEL (LangChain Expression Language), piping components with |:

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough
from langchain_groq import ChatGroq

llm = ChatGroq(model="llama-3.3-70b-versatile", temperature=0,
               api_key=os.getenv("groq_api_key"))

prompt = ChatPromptTemplate.from_template(
    "Answer the question using only the context below.\n"
    "If the answer isn't in the context, say you don't know.\n\n"
    "Context: {context}\n"
    "Question: {question}"
)

# Join retrieved Documents into a single context string
def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

# The retriever pulls context; the question passes straight through
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

print(rag_chain.invoke("What is deep learning?"))

Read the chain top to bottom: the user's question goes to the retriever, which searches the vector store and returns the nearest chunks; format_docs stitches them into a context string; the prompt fuses context and question; the llm answers; and StrOutputParser hands back clean text. The vector store is the retrieval engine at the heart of it.

Managed Cloud Databases — Pinecone & Astra DB

Local stores are perfect for development. In production — millions of vectors, many users, zero ops — you reach for a managed cloud vector database. The headline feature: the LangChain interface is identical, so the RAG chain above doesn't change at all.

Pinecone

Pinecone is a fully-managed, serverless vector database. You create an index once (declaring its dimension and distance metric), then use the same add_documents / similarity_search calls.

from pinecone import Pinecone, ServerlessSpec
from langchain_pinecone import PineconeVectorStore

pc = Pinecone(api_key=os.getenv("pinecone_api_key"))

# Create the index once (dimension must match your embedding model)
if not pc.has_index("rag"):
    pc.create_index(
        name="rag",
        dimension=1024,
        metric="cosine",
        spec=ServerlessSpec(cloud="aws", region="us-east-1"),
    )

vector_store = PineconeVectorStore(index=pc.Index("rag"), embedding=embeddings)
vector_store.add_documents(documents)

# Same interface, with optional metadata filtering
results = vector_store.similarity_search("Will it be hot tomorrow?",
                                         k=2, filter={"source": "news"})

Astra DB (DataStax)

Astra DB is DataStax's serverless database, built on Apache Cassandra and tuned for low-latency AI workloads. You point the store at your endpoint and token — again, the same methods.

from langchain_astradb import AstraDBVectorStore

vector_store = AstraDBVectorStore(
    embedding=embeddings,
    collection_name="astra_vector_langchain",
    api_endpoint=os.getenv("ASTRA_DB_API_ENDPOINT"),
    token=os.getenv("ASTRA_DB_APPLICATION_TOKEN"),
)

vector_store.add_documents(documents)
results = vector_store.similarity_search("What is the weather?", k=3)
Keep Secrets Out of Code

API keys, endpoints, and tokens belong in environment variables loaded from a .env file — never hard-coded and never committed to git. Notice every example above reads them with os.getenv(...). The embedding model's dimension must also match the index (e.g. 1024 here), or inserts will be rejected.

InMemoryVectorStore — The Simplest Option

When you just need to prototype or write a quick test, InMemoryVectorStore is the lightest possible choice. Under the hood it's a dictionary plus NumPy cosine similarity — nothing is persisted, and it disappears when your program exits.

from langchain_core.vectorstores import InMemoryVectorStore

vector_store = InMemoryVectorStore(embedding=embeddings)
vector_store.add_documents(documents)

results = vector_store.similarity_search("how's the weather forecast", k=2)

# It becomes a retriever exactly like every other store
retriever = vector_store.as_retriever(search_kwargs={"k": 2})

Same five methods, zero setup. It's the fastest way to confirm a RAG flow works before committing to a real database.

Choosing a Vector Store

With the same interface everywhere, the choice comes down to operational needs — persistence, scale, and who runs it — not your application code.

If you need…Reach for
A throwaway demo or unit testsInMemoryVectorStore
Fast local search over many static vectorsFAISS
A local database with persistence & metadataChromaDB
Managed, serverless, production scalePinecone
Enterprise cloud with ultra-low latencyAstra DB

A common path: prototype with Chroma or FAISS locally, then switch to Pinecone or Astra for production by changing a single import — because the retriever, the chain, and everything downstream stay exactly the same.

Key Takeaways

Remember These Four

1. A vector store does similarity search; a vector database adds durable persistence, metadata filtering, live updates, and scale.

2. LangChain gives every backend the same interface — from_documents, add_documents, similarity_search, as_retriever — so engines are swappable with a one-line import change.

3. ChromaDB is the local-first workhorse (persists to disk); FAISS is the in-memory speed king (you save/load it); Pinecone and Astra DB are managed cloud for production.

4. .as_retriever() turns any store into the retrieval half of RAG — pick similarity, similarity_score_threshold, or mmr — and LCEL pipes it straight into the LLM.
