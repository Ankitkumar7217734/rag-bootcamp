Chapter 16: Multi-Agent RAG Systems — Specialized Agents, Supervisors & Hierarchies

When one agent with too many tools starts to struggle, split the RAG pipeline into specialized agents — then coordinate them with handoffs, a central supervisor, or a full hierarchy of nested teams.

Overview

Chapters 13–15 built increasingly capable single-agent RAG systems: an agent that decides when to retrieve, a ReAct loop that picks the right tool, and an autonomous graph that plans, reflects, and retries on its own. All of that still lives inside one LLM brain with a growing list of tools attached.

Multi-Agent RAG takes the next step. Instead of one generalist agent juggling retrieval, web search, math, writing, and charting, you create specialized agents — each with a narrow role and a focused tool set — and wire them together so they collaborate on a single query. The RAG retriever becomes one agent's job; another agent writes the blog; a supervisor decides who goes next.

This chapter walks through three coordination patterns from 8-multiagent.ipynb, each adding a layer of orchestration. They build on the same ingredients you already know: create_react_agent, FAISS retriever tools, Tavily web search, and LangGraph's Command routing.

The Core Idea

A single LLM with ten tools is a generalist — fine for a few tools, but routing gets noisy as the list grows. Multi-Agent RAG splits the pipeline into specialists: one agent retrieves, another writes, a supervisor routes work, and mid-level supervisors manage teams when the job gets too big for one coordinator. Each agent keeps a focused ReAct loop; the graph handles who talks to whom.

Why Multi-Agent?

The handwritten notes frame the motivation clearly. A single agent architecture — query in, LLM brain, tools (retriever + custom), answer out — works well for straightforward tasks. But when the task is complex, you need multiple agents each doing their own job, communicating through a shared message history.

The same notes connect multi-agent RAG back to the Autonomous RAG building blocks from Chapter 15: chain-of-thought, ReAct, self-reflection, and query planning. In a multi-agent network, the blog writer can perform self-reflection before output; the researcher runs a ReAct loop with RAG and Tavily. Specialization lets each agent apply the right technique without one prompt trying to do everything.

Problem with one agentMulti-agent fix
Too many tools — the LLM picks the wrong one or hallucinates tool names Each agent gets 1–3 tools matched to its role
Research and writing compete for context window space Researcher gathers facts; writer receives a clean handoff message
One supervisor can't route ten workers reliably Mid-level supervisors manage sub-teams; top supervisor picks the team
Math and retrieval need different system prompts Dedicated agents with role-specific instructions (research vs math)

Shared Setup

All three patterns share the same foundation: an LLM, a retriever tool built from a text file, and (where needed) Tavily web search. The reusable factory from Chapter 14 makes a return appearance — load a file, chunk it, embed into FAISS, wrap as a LangChain Tool: 

import os
from dotenv import load_dotenv
load_dotenv()

from langchain.chat_models import init_chat_model
from langchain_classic.agents import Tool
from langchain_classic.document_loaders import TextLoader
from langchain_classic.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter

os.environ["OPENAI_API_KEY"] = os.getenv("OPENAI_API_KEY")
os.environ["TAVILY_API_KEY"] = os.getenv("TAVILY_API_KEY")

llm = init_chat_model("openai:gpt-4o-mini")

def make_retriever_tool_from_text(file, name, desc):
    docs = TextLoader(file, encoding="utf-8").load()
    chunks = RecursiveCharacterTextSplitter(
        chunk_size=500, chunk_overlap=50
    ).split_documents(docs)
    vs = FAISS.from_documents(chunks, OpenAIEmbeddings())
    retriever = vs.as_retriever()

    def tool_func(query: str) -> str:
        print(f"Using tool: {name}")
        results = retriever.invoke(query)
        return "\n\n".join(doc.page_content for doc in results)

    return Tool(name=name, description=desc, func=tool_func)

internal_tool = make_retriever_tool_from_text(
    "research_notes.txt",
    "InternalResearchNotes",
    "Search internal research notes for experimental results",
)

The notebook uses sample_docs.txt for the retriever; we point at research_notes.txt here because it holds the full transformer experiment log (EfficientFormer, Longformer, Reformer, LLaMA2, TinyBERT, and more) that the agents retrieve against in the demo runs.
