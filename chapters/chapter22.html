Chapter 22: Guardrails — Building Safe, Compliant AI Agents

Twenty-one chapters of making agents smarter — better retrieval, reasoning, memory, caching. This one makes them safe: middleware that intercepts an agent's execution to redact PII, block bad inputs, pause for human approval, and validate outputs before a user ever sees them.

Guardrails in One Paragraph

Every agent so far has had one implicit assumption: whatever the user types goes to the model, and whatever the model produces goes back to the user. In production, both halves of that assumption are dangerous. Guardrails are checkpoints wrapped around the agent — before it starts (block or rewrite bad input), around its model and tool calls (redact credit cards, pause for human approval), and after it finishes (validate or replace the output). In LangChain they're implemented as middleware — a list you pass to create_agent — and they stack: each layer catches what the previous one missed.

Overview

This chapter is built from the LangChain Guardrails crash-course notebook, which works through LangChain's middleware system (docs: docs.langchain.com/oss/python/langchain/guardrails) from first principles to a production-shaped build. Two built-in middlewares do the heavy lifting — PII detection and human-in-the-loop approval — and two custom hooks (before_agent, after_agent) cover everything the built-ins don't. The capstone assembles all of it into a healthcare chatbot with four independent safety layers.

Here's the full route:

The Core Idea

Guardrails are middleware. You don't rewrite the agent to make it safe — you wrap it. Every guardrail in this chapter is an item in the middleware=[] list of create_agent(), intercepting execution at one of four points: before the agent starts, around each model call, around each tool call, or after the agent finishes. Because they're just list items, they compose — a production agent stacks a cheap deterministic filter in front of PII redaction, an approval gate, and a model-based output check, and each layer catches what the previous one missed.

What Guardrails Protect Against

“Safety” sounds abstract until you list what actually goes wrong in production. The notebook's use-case table is a checklist of real incidents waiting to happen:

Use caseExampleGuardrail in this chapter
PII leakage prevention Redact emails/credit cards before they reach the model or logs PIIMiddleware
Prompt injection blocking Detect adversarial inputs before any LLM processing before_agent filter
Harmful content filtering Block dangerous requests (“how do I hack…”) before_agent + model-based judge
Business rule enforcement Require approval for financial ops, emails, deletions HumanInTheLoopMiddleware
Output quality validation Ensure the response meets safety standards before delivery after_agent validator

Notice the pattern: two of these are about what comes in, one is about what the agent does, and two are about what goes out. That's why guardrails live at different interception points — no single checkpoint can cover all five rows.

Two Approaches — Deterministic vs Model-Based

Before any middleware, the notebook establishes the fundamental design split. Every guardrail you will ever write is one of two kinds:

Deterministic — Rules, Regex, Keywords

A deterministic guardrail is plain code: keyword lists, regex patterns, explicit checks. Ten lines, no API calls, microsecond latency: 

# Quick illustration of the two approaches

import re

# --- Deterministic approach ---
def deterministic_guardrail(text: str) -> bool:
    """Returns True if content is blocked."""
    banned_keywords = ["hack", "exploit", "malware", "bomb"]
    return any(kw in text.lower() for kw in banned_keywords)

test_inputs = [
    "How do I hack into a database?",
    "What is the capital of France?",
    "Explain how malware spreads",
]

print("=== Deterministic Guardrail Demo ===")
for inp in test_inputs:
    blocked = deterministic_guardrail(inp)
    status = "🚫 BLOCKED" if blocked else "✅ ALLOWED"
    print(f"{status}: {inp}")
=== Deterministic Guardrail Demo ===
🚫 BLOCKED: How do I hack into a database?
✅ ALLOWED: What is the capital of France?
🚫 BLOCKED: Explain how malware spreads

Model-Based — An LLM as the Judge

A model-based guardrail hands the decision to an LLM — here gpt-4o-mini at temperature=0, asked for a one-word verdict. Slower and costlier, but it understands meaning, not just characters: 

from langchain_openai import ChatOpenAI

# --- Model-based approach ---
def model_based_guardrail(text: str) -> str:
    """Uses an LLM to evaluate content safety. Returns SAFE or UNSAFE."""
    model = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    prompt = f"""Is the following user input safe to process?
Reply with only 'SAFE' or 'UNSAFE'.

Input: {text}"""
    result = model.invoke([{"role": "user", "content": prompt}])
    return result.content.strip()

print("=== Model-Based Guardrail Demo ===")
for inp in test_inputs:
    verdict = model_based_guardrail(inp)
    status = "🚫 UNSAFE" if "UNSAFE" in verdict else "✅ SAFE"
    print(f"{status}: {inp}")
=== Model-Based Guardrail Demo ===
🚫 UNSAFE: How do I hack into a database?
✅ SAFE: What is the capital of France?
✅ SAFE: Explain how malware spreads

Read the two outputs side by side — they disagree on the third input. The keyword filter blocked “Explain how malware spreads” because it contains the string malware; the LLM judged it SAFE because it's a legitimate educational question. That disagreement is the whole trade-off:

DeterministicModel-based
Mechanism Regex, keyword matching, explicit checks LLM / classifier evaluates semantics
Speed Microseconds One model call (hundreds of ms)
Cost Free Tokens per check
Predictability Fully deterministic — same input, same verdict Mostly stable at temperature=0, still a model
Weakness Misses paraphrases; false-positives on innocent matches Slower, more expensive, needs prompt care
Use for First line of defense — cheap, early Nuanced calls the rules can't make
Deterministic First, Model-Based Second

This ordering is the chapter's recurring cost rule. A keyword check costs nothing and runs before any LLM call — if it blocks the request, you never pay for the model at all. The expensive semantic judge runs only on traffic that survived the cheap filter. You'll see this exact ordering again in the five-layer production stack.

Built-in Guardrail #1 — PII Detection Middleware

The first built-in, PIIMiddleware, detects and handles Personally Identifiable Information flowing through the agent. It ships with detectors for common PII types, and you can add your own with a regex:

Built-in typeExample
emailuser@example.com
credit_card5105-1051-0510-5100
ip192.168.1.1
mac_address00:1A:2B:3C:4D:5E
urlhttps://secret-site.com

For each type you pick a strategy — what happens when the pattern is found:

StrategyResultWhen to use
redact [REDACTED_EMAIL] The value is irrelevant to the task — remove it entirely
mask ****-****-****-1234 Keep a recognizable tail (last four digits) for reference
hash a8f5f167... Need a stable identifier without exposing the value
block Raises an exception The value should never be in a prompt at all (API keys, secrets)

Wiring Three PII Rules Into an Agent

One agent, three PIIMiddleware entries — each is its own layer with its own type and strategy. Note the third: api_key isn't a built-in type, so it supplies a custom detector regex, and its strategy is block — a leaked secret should fail loudly, not get politely masked: 

from langchain.agents import create_agent
from langchain.agents.middleware import PIIMiddleware
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

# Define a simple dummy tool
@tool
def customer_lookup(query: str) -> str:
    """Look up customer information."""
    return f"Customer record found for query: {query}"

# Create agent with PII Middleware
agent = create_agent(
    model="gpt-4o",
    tools=[customer_lookup],
    middleware=[
        # Redact emails in user input before sending to model
        PIIMiddleware(
            "email",
            strategy="redact",
            apply_to_input=True,
        ),
        # Mask credit cards in user input
        PIIMiddleware(
            "credit_card",
            strategy="mask",
            apply_to_input=True,
        ),
        # Block API keys - raise error if detected
        PIIMiddleware(
            "api_key",
            detector=r"sk-[a-zA-Z0-9]{32}",
            strategy="block",
            apply_to_input=True,
        ),
    ],
)

print("Agent with PII middleware created successfully!")

The raw PII existed for exactly one moment, in the original request, and nowhere else.

Blocking — the Hard Stop

The block strategy doesn't rewrite — it raises. Feed the agent an API key:

# Test API Key Blocking
try:
    result = agent.invoke({
        "messages": [{
            "role": "user",
            "content": "Here is my key: sk-abcdefghijklmnopqrstuvwxyz123456"
        }]
    })

except Exception as e:
    print(f"🚫 Blocked as expected: {e}")
🚫 Blocked as expected: Detected 1 instance(s) of api_key in text content
Why This Matters for RAG Systems

Everything you send to a model API leaves your infrastructure, and everything in the message history lands in traces, logs, and — in this bootcamp's own Chapter 20 — caches. A semantic cache that stores raw questions will happily persist a user's credit card number forever. Redacting PII at the middleware layer, before the model call, is the single cheapest place to stop that entire class of leak.

Built-in Guardrail #2 — Human-in-the-Loop Middleware

Some actions are too consequential for an LLM to take alone: sending emails to external parties, financial transactions, deleting production data. HumanInTheLoopMiddleware pauses the agent at the moment it tries to call a sensitive tool and waits for a human decision. Two pieces make it work:

  • interrupt_on — a dict mapping tool names to whether they need approval. True pauses; False auto-approves.
  • A checkpointer — required, because “pause” really means the graph state is saved mid-run and resumed later. This is exactly Chapter 19's InMemorySaver + thread_id machinery, reused for safety instead of memory.
from langchain.agents import create_agent
from langchain.agents.middleware import HumanInTheLoopMiddleware
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.types import Command
from langchain_core.tools import tool

@tool
def search_web(query: str) -> str:
    """Search the web for information."""
    return f"Search results for: {query}"

@tool
def send_email(to: str, subject: str, body: str) -> str:
    """Send an email to a recipient."""
    return f"Email sent to {to} with subject: {subject}"

@tool
def delete_records(table: str, condition: str) -> str:
    """Delete records from the database."""
    return f"Deleted records from {table} where {condition}"

Three tools, three risk levels: reading the web is harmless, sending email is outward-facing, deleting records is destructive. The interrupt_on map encodes exactly that judgment: 

# Create agent with HITL middleware
hitl_agent = create_agent(
    model="gpt-4o",
    tools=[search_web, send_email, delete_records],
    middleware=[
        HumanInTheLoopMiddleware(
            interrupt_on={
                "send_email": True,       # Require approval
                "delete_records": True,   # Require approval
                "search_web": False,      # Auto-approve
            }
        ),
    ],
    checkpointer=InMemorySaver(),  # Required for state persistence
)

print("Human-in-the-Loop agent created!")

Step 1 — The Agent Pauses

Ask it to send an email. The model composes the tool call — and then the run stops. The returned state ends with an __interrupt__ instead of a final answer: 

# Step 1: Invoke — agent will pause before send_email
config = {"configurable": {"thread_id": "session_001"}}

result = hitl_agent.invoke(
    {"messages": [{"role": "user", "content": "Send an email to team@company.com about the Q4 results"}]},
    config=config
)

print("=== Agent paused — awaiting human approval ===")
print(result)

Step 2 — Approve and Resume

Resuming is a second invoke on the same thread_id, passing a Command instead of messages. The checkpointer restores the paused state, the decision is applied, and the tool finally runs: 

# Step 2: Human reviews and APPROVES
approved_result = hitl_agent.invoke(
    Command(resume={"decisions": [{"type": "approve"}]}),
    config=config   # Same thread_id resumes the paused session
)

print("=== Approved! Final response ===")
print(approved_result["messages"][-1].content)
=== Approved! Final response ===
I've sent the email to team@company.com about the Q4 results.

Step 3 — Or Reject, With a Reason

The reject path matters just as much — a fresh thread asks for a destructive delete_records call, and the human says no: 

# Step 3: Alternative — Human REJECTS
config2 = {"configurable": {"thread_id": "session_002"}}

hitl_agent.invoke(
    {"messages": [{"role": "user", "content": "Delete all records from the users table where active=false"}]},
    config=config2
)

rejected_result = hitl_agent.invoke(
    Command(resume={"decisions": [{"type": "reject", "reason": "Too risky, needs DBA review"}]}),
    config=config2
)

print("=== Rejected! Final response ===")
print(rejected_result["messages"][-1].content)
=== Rejected! Final response ===
It seems that you've decided not to proceed with the deletion. If you have
any questions or need further assistance, feel free to ask!

The tool never executed — the rejection (and its reason) went back to the model, which acknowledged gracefully. No deleted rows, no drama, and the whole exchange is in the thread history for audit.

Gotcha — No Checkpointer, No Pause

HumanInTheLoopMiddleware requires a checkpointer. An interrupt is a saved graph state waiting to be resumed — with nowhere to save it, the middleware can't pause. InMemorySaver is fine for development; production wants a persistent store (SQLite/Postgres checkpointer), for exactly the same reason as Chapter 19: in-RAM state dies with the process — and a pending approval that vanishes on restart is a lost transaction.

Custom Guardrail #1 — before_agent: Input Filtering

When the built-ins don't cover your rule, subclass AgentMiddleware. The first custom hook is before_agent() — it runs before any LLM processing begins, which makes it the right home for keyword filters, authentication checks, and rate limits: a blocked request costs zero model tokens.

Three pieces of the class to notice before the code:

  • @hook_config(can_jump_to=["end"]) declares that this hook is allowed to short-circuit the run.
  • Returning None means “no objection — continue normally.”
  • Returning {"messages": [...], "jump_to": "end"} injects a canned assistant reply and skips the entire agent — model, tools, everything.
from typing import Any
from langchain.agents.middleware import AgentMiddleware, AgentState, hook_config
from langgraph.runtime import Runtime
from langchain.agents import create_agent
from langchain_core.tools import tool

class ContentFilterMiddleware(AgentMiddleware):
    """
    Deterministic guardrail: Block requests containing banned keywords.
    This runs BEFORE the agent processes anything — zero LLM cost for blocked requests.
    """

    def __init__(self, banned_keywords: list[str]):
        super().__init__()
        self.banned_keywords = [kw.lower() for kw in banned_keywords]

    @hook_config(can_jump_to=["end"])
    def before_agent(self, state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
        if not state["messages"]:
            return None

        first_message = state["messages"][0]
        if first_message.type != "human":
            return None

        content = first_message.content.lower()

        for keyword in self.banned_keywords:
            if keyword in content:
                print(f"🚫 Blocked — keyword detected: '{keyword}'")
                return {
                    "messages": [{
                        "role": "assistant",
                        "content": (
                            "I cannot process requests containing inappropriate content. "
                            "Please rephrase your request."
                        )
                    }],
                    "jump_to": "end"
                }
        return None

Attach it to an agent like any other middleware:

@tool
def search_tool(query: str) -> str:
    """Search for information."""
    return f"Results for: {query}"


# Create agent with content filter
filtered_agent = create_agent(
    model="gpt-4o",
    tools=[search_tool],
    middleware=[
        ContentFilterMiddleware(
            banned_keywords=["hack", "exploit", "malware", "jailbreak", "bypass"]
        ),
    ],
)

Custom Guardrail #2 — after_agent: Output Safety

The mirror-image hook is after_agent() — it runs when the agent has produced its final response, before the user sees it. That's the place for model-based safety evaluation, compliance scanning (legal/medical/financial disclaimers), and catching anything that slipped through the input-side layers.

This middleware is the model-based half of the earlier section, packaged as a class. Two design choices worth copying: the judge is a smaller, cheaper model (gpt-4o-mini judging gpt-4o's output), and on failure it doesn't error — it replaces the unsafe content with a safe fallback, in place: 

from typing import Any
from langchain.agents.middleware import AgentMiddleware, AgentState, hook_config
from langgraph.runtime import Runtime
from langchain_core.messages import AIMessage
from langchain_openai import ChatOpenAI

class SafetyGuardrailMiddleware(AgentMiddleware):
    """
    Model-based guardrail: Use an LLM to evaluate response safety.
    Runs AFTER the agent produces a response, before it reaches the user.
    """

    def __init__(self):
        super().__init__()
        # Use a smaller, cheaper model for the safety check
        self.safety_model = ChatOpenAI(model="gpt-4o-mini", temperature=0)

    @hook_config(can_jump_to=["end"])
    def after_agent(self, state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
        if not state["messages"]:
            return None

        last_message = state["messages"][-1]
        if not isinstance(last_message, AIMessage):
            return None

        # Use a lightweight model as the safety judge
        safety_prompt = f"""Evaluate if this AI response is safe and appropriate for users.
Respond with only 'SAFE' or 'UNSAFE'.

Response to evaluate:
{last_message.content}"""

        result = self.safety_model.invoke([{"role": "user", "content": safety_prompt}])

        if "UNSAFE" in result.content.upper():
            print("⚠️  Output flagged as UNSAFE — replacing with safe fallback")
            last_message.content = (
                "I'm unable to provide that response. "
                "Please rephrase your request or contact support."
            )

        return None
@tool
def general_tool(query: str) -> str:
    """A general purpose tool."""
    return f"Tool result: {query}"


safe_agent = create_agent(
    model="gpt-4o",
    tools=[general_tool],
    middleware=[SafetyGuardrailMiddleware()],
)

# Test output safety check
result = safe_agent.invoke({
    "messages": [{"role": "user", "content": "What is the weather like today?"}]
})
print("Response:")
print(result["messages"][-1].content)
Response:
I fetched today's weather forecast for you. Please let me know if there's
anything specific you would like to know about the weather!

A benign response passes the judge untouched. If the judge had answered UNSAFE, the user would have received the fallback text instead — and never known what the raw output said. That's the point: the last thing that touches the response before the user is your validator, not the model.

This Is LLM-as-a-Judge — You've Seen It Before

Structurally, SafetyGuardrailMiddleware is the same trick as Chapter 17's document grader and Chapter 18's hallucination checker: a small, cheap LLM with a constrained one-word output sitting in judgment over a bigger model's work. CRAG graded retrieval; Adaptive RAG graded groundedness; a safety guardrail grades appropriateness. Same pattern, new checkpoint.

Layered Guardrails — Defense in Depth

No single guardrail catches everything: the keyword filter misses paraphrases, the PII regex knows nothing about harmful intent, the safety judge never sees requests the filter already blocked. The production answer is to stack them — the middleware=[] array executes in order, so each request passes through every layer: 

from langchain.agents import create_agent
from langchain.agents.middleware import PIIMiddleware, HumanInTheLoopMiddleware
from langgraph.checkpoint.memory import InMemorySaver
from langchain_core.tools import tool

@tool
def search_tool(query: str) -> str:
    """Search for information."""
    return f"Search results: {query}"

@tool
def send_email_tool(to: str, body: str) -> str:
    """Send an email."""
    return f"Email sent to {to}"

# Full layered guardrail stack
production_agent = create_agent(
    model="gpt-4o",
    tools=[search_tool, send_email_tool],
    middleware=[
        # Layer 1: Deterministic input filter (before agent)
        ContentFilterMiddleware(banned_keywords=["hack", "exploit", "malware"]),

        # Layer 2: PII redaction on input
        PIIMiddleware("credit_card", strategy="mask", apply_to_input=True),

        # Layer 3: Human approval for sensitive tools
        HumanInTheLoopMiddleware(
            interrupt_on={"send_email_tool": True, "search_tool": False}
        ),

        # Layer 4: PII redaction on output
        PIIMiddleware("email", strategy="redact", apply_to_output=True),

        # Layer 5: Model-based output safety
        SafetyGuardrailMiddleware(),
    ],
    checkpointer=InMemorySaver(),
)

print("🏭 Production-grade agent with 5-layer guardrails created!")

Read the order again — it's not arbitrary. The free keyword check runs first, so obviously bad requests cost nothing. PII is scrubbed before any model call. The approval gate wraps only the tool that acts on the outside world. And the one guardrail that costs an LLM call — the safety judge — runs last, only on responses that made it through everything else.

Real-World Build — A Healthcare Chatbot

The capstone assembles the whole toolbox for a domain where the stakes are obvious. The requirements read like a compliance document, and each maps to one guardrail:

RequirementGuardrailKind
Block off-topic or harmful requests HealthcareSafetyFilter (custom before_agent) Deterministic
Protect patient PII PIIMiddleware ×2 (email redact, card mask) Deterministic
Human sign-off before appointments are booked HumanInTheLoopMiddleware on book_appointment Human
Every answer carries a medical disclaimer MedicalOutputValidator (custom after_agent) Deterministic

The Two Custom Middlewares

The input filter is ContentFilterMiddleware specialized for the domain — and its canned refusal is domain-appropriate too, pointing users in crisis to emergency services: 

# --- Healthcare-specific content filter ---
class HealthcareSafetyFilter(AgentMiddleware):
    """Block non-medical or harmful requests in a healthcare context."""

    BLOCKED_TOPICS = ["drug synthesis", "self-harm", "suicide method", "weapon", "hack"]

    @hook_config(can_jump_to=["end"])
    def before_agent(self, state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
        if not state["messages"]:
            return None

        first_msg = state["messages"][0]
        if first_msg.type != "human":
            return None

        content = first_msg.content.lower()
        for topic in self.BLOCKED_TOPICS:
            if topic in content:
                return {
                    "messages": [{
                        "role": "assistant",
                        "content": (
                            "I'm a healthcare assistant and can only help with "
                            "medical questions, appointments, and health information. "
                            "If you're in crisis, please call 112 or your local emergency number."
                        )
                    }],
                    "jump_to": "end"
                }
        return None

The output validator shows a third kind of after_agent behavior: not judging, not replacing — appending. Every response that doesn't already carry a medical disclaimer gets one: 

# --- Medical output validator ---
class MedicalOutputValidator(AgentMiddleware):
    """Ensure all responses include appropriate medical disclaimers."""

    DISCLAIMER = "\n\n⚕️ *This is general health information, not medical advice. Please consult a qualified healthcare professional.*"

    @hook_config(can_jump_to=["end"])
    def after_agent(self, state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
        if not state["messages"]:
            return None

        last_message = state["messages"][-1]
        if not isinstance(last_message, AIMessage):
            return None

        # Add disclaimer if not already present
        if "medical advice" not in last_message.content.lower():
            last_message.content += self.DISCLAIMER

        return None

Tools and Assembly

# --- Healthcare tools ---
@tool
def search_symptoms(symptoms: str) -> str:

@tool
def book_appointment(patient_name: str, date: str, doctor: str) -> str:
    """Book a medical appointment."""
    return f"Appointment booked for {patient_name} with Dr. {doctor} on {date}"

@tool
def get_medication_info(medication: str) -> str:
    """Get information about a medication."""
    return f"General info about {medication}. Always follow your doctor's prescription."
# --- Build the healthcare chatbot ---
healthcare_bot = create_agent(
    model="gpt-4o",
    tools=[search_symptoms, book_appointment, get_medication_info],
    middleware=[
        # Guardrail 1: Block harmful/off-topic requests
        HealthcareSafetyFilter(),

        # Guardrail 2: Redact patient PII from inputs
        PIIMiddleware("email", strategy="redact", apply_to_input=True),
        PIIMiddleware("credit_card", strategy="mask", apply_to_input=True),

        # Guardrail 3: Require approval before booking appointments
        HumanInTheLoopMiddleware(
            interrupt_on={
                "book_appointment": True,
                "search_symptoms": False,
                "get_medication_info": False,
            }
        ),

        # Guardrail 4: Add medical disclaimer to all outputs
        MedicalOutputValidator(),
    ],
    checkpointer=InMemorySaver(),
    system_prompt=(
        "You are a helpful healthcare assistant. "
        "You can search for symptoms, medication information, and help book appointments. "
        "Always be empathetic and remind users to consult a doctor for diagnosis."
    )
)

print("🏥 Healthcare chatbot with full guardrail stack created!")

PII Middleware Input/Output Around model calls Data privacy, compliance
Human-in-the-Loop Tool level Before sensitive tools High-stakes decisions
Content Filter before_agent Start of invocation Blocking bad inputs early

Custom logic Any hook Anywhere Any business rule

Key Takeaways

Remember These Four

1. Guardrails = middleware. Every guardrail — built-in or custom — is an entry in the middleware=[] parameter of create_agent(), intercepting execution before the agent, around model calls, around tool calls, or after the agent. You wrap the agent; you don't rewrite it.

2. Layer them, cheapest first. Deterministic checks (keywords, PII regex) are free and run early — a request blocked in before_agent costs zero LLM tokens. Model-based judges are semantic but expensive, so they run last, on the few responses that survive everything else. Defense in depth: each layer catches what the previous one missed.

3. Human-in-the-loop needs a checkpointer. HumanInTheLoopMiddleware pauses by saving graph state (__interrupt__ with action_requests) and resumes via Command(resume={"decisions": [...]}) on the same thread_id — approve, edit, or reject. InMemorySaver for dev, a persistent store for production.

4. Custom hooks give you full control. Subclass AgentMiddleware, implement before_agent() or after_agent(), and either return None (continue) or a state update — with @hook_config(can_jump_to=["end"]) letting input filters short-circuit the whole run, and output validators free to judge, replace, or append (the healthcare bot's auto-disclaimer) before the user sees anything.
