The Graph — State, Nodes & Edges

With two retriever tools in hand, we now build the machine that uses them: a shared state, four nodes that transform it, and two conditional edges that steer the flow.

What We're Assembling

This is the heart of Agentic RAG. Everything from 13.1 was preparation; now we write the graph itself — the same one drawn in the chapter overview. We'll define the state first, then write each node and edge as an isolated function, and finally wire them together and compile. Keep one principle in mind throughout: nodes do work and return a state update; edges only choose the next node.

Step 3 — The State

Like the chatbot and the agent of Chapter 11.2 and Chapter 12, the state is simply the conversation: a growing list of messages. We declare it as a TypedDict with a single key, messages.

from typing import Annotated, Sequence
from typing_extensions import TypedDict
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages


class AgentState(TypedDict):
    # add_messages defines how updates are processed.
    # Default is to replace; add_messages says "append".
    messages: Annotated[Sequence[BaseMessage], add_messages]

The magic is in the Annotated[..., add_messages] annotation. Normally, when a node returns a value for a state key, it overwrites whatever was there. add_messages is a reducer — a function LangGraph runs to merge a node's output into the existing state instead of replacing it. For messages, that merge means append.

PieceMeaning
messagesThe one and only state key — the full conversation so far.
Sequence[BaseMessage]The type: an ordered list of messages (HumanMessage, AIMessage, ToolMessage).
add_messagesThe reducer. Every node that returns {"messages": [x]} appends x rather than clobbering the history.

This one-key design is what keeps the rest of the code so short. Because every node appends, two positions are always meaningful, no matter how long the conversation grows:

  • messages[0] — the original question. The grader and the generator both read it from here.
  • messages[-1] — the most recent output: whatever the last node produced (a tool call, retrieved docs, an answer).

Step 4 — The agent Node

The agent node is the brain. It binds the two tools to the LLM and invokes it on the current conversation. The model then either emits a tool call (it wants to retrieve) or returns a plain answer (it's done).

from langchain_groq import ChatGroq


def agent(state):
    """
    Invokes the agent model to generate a response based on the current state.
    Given the question, it will decide to retrieve using the retriever tool, or simply end.

    Args:
        state (messages): The current state

    Returns:
        dict: The updated state with the agent response appended to messages
    """
    print("---CALL AGENT---")
    messages = state["messages"]
    model = ChatGroq(model="llama-3.3-70b-versatile")
    model = model.bind_tools(tools)
    response = model.invoke(messages)
    # We return a list, because this will get added to the existing list
    return {"messages": [response]}

The crucial detail: this node does not decide what happens next. It just produces a message and appends it. When the model wants to retrieve, that message is an AIMessage with empty text and a populated tool_calls list:

AIMessage(
    content='',
    tool_calls=[{'name': 'retriever_vector_db_blog',

)

Notice the model chose the tool (retriever_vector_db_blog) and wrote the query itself ("Langgraph"). That's the whole decision. What turns that decision into action is the tools_condition edge — coming up in Step 8 — which reads this message and routes to the retrieve node.

Step 5 — The grade_documents Edge

This is the self-checking step that makes RAG "agentic." After retrieval, an LLM grades the chunks against the original question and returns a strict binary yes / no. The function's return value is a node name — "generate" or "rewrite" — which is exactly what makes it a conditional edge rather than a node.

from typing import Literal
from langchain_core.prompts import PromptTemplate
from pydantic import BaseModel, Field


def grade_documents(state) -> Literal["generate", "rewrite"]:
    """
    Determines whether the retrieved documents are relevant to the question.

    Args:
        state (messages): The current state

    Returns:
        str: A decision for whether the documents are relevant or not
    """
    print("---CHECK RELEVANCE---")

    # Data model — forces the LLM to answer with exactly 'yes' or 'no'
    class grade(BaseModel):
        """Binary score for relevance check."""
        binary_score: str = Field(description="Relevance score 'yes' or 'no'")

    model = ChatGroq(model="llama-3.3-70b-versatile")
    llm_with_tool = model.with_structured_output(grade)

    prompt = PromptTemplate(
        template="""You are a grader assessing relevance of a retrieved document to a user question. \n
        Here is the retrieved document: \n\n {context} \n\n
        Here is the user question: {question} \n
        If the document contains keyword(s) or semantic meaning related to the user question, grade it as relevant. \n
        Give a binary score 'yes' or 'no' score to indicate whether the document is relevant to the question.""",
        input_variables=["context", "question"],
    )

    chain = prompt | llm_with_tool

    messages = state["messages"]
    last_message = messages[-1]      # the retrieved documents
    question = messages[0].content   # the original question
    docs = last_message.content

    scored_result = chain.invoke({"question": question, "context": docs})
    score = scored_result.binary_score

    if score == "yes":
        print("---DECISION: DOCS RELEVANT---")
        return "generate"
    else:
        print("---DECISION: DOCS NOT RELEVANT---")
        return "rewrite"

Three things are happening here, and each is worth a closer look.

MechanismWhat it does
class grade(BaseModel)A one-field Pydantic model. It's the schema for the answer we want — a single binary_score string.
with_structured_output(grade)Forces the LLM to return an instance of that model, not free text. The output is guaranteed to have a .binary_score attribute.
messages[0] & messages[-1]The question (first message) and the retrieved docs (latest message) — pulled straight out of the growing state.
Why force a binary score?

A conditional edge has to return one of a fixed set of node names. If we let the grader reply in prose ("These documents seem somewhat related…"), we'd have to parse that text and guess. with_structured_output removes the guesswork: the LLM is constrained to fill in the grade schema, so score is reliably "yes" or "no", and the routing is a clean if. Structured output is how you turn an LLM's judgement into control flow.

Step 6 — The generate Node

If the docs passed the grade, generation is ordinary RAG: feed the question and the retrieved context into a concise answer prompt and parse the result to a string. This is the classic rlm/rag-prompt, written inline so there are no hidden dependencies.

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser


def generate(state):
    """
    Generate answer

    Args:
        state (messages): The current state

    Returns:
        dict: The updated message
    """
    print("---GENERATE---")
    messages = state["messages"]
    question = messages[0].content
    last_message = messages[-1]
    docs = last_message.content

    prompt = ChatPromptTemplate.from_messages([
        ("human",
         "You are an assistant for question-answering tasks. "
         "Use the following pieces of retrieved context to answer the question. "
         "If you don't know the answer, just say that you don't know. "
         "Use three sentences maximum and keep the answer concise.\n"
         "Question: {question}\nContext: {context}\nAnswer:")
    ])

    llm = ChatGroq(model="llama-3.3-70b-versatile")
    rag_chain = prompt | llm | StrOutputParser()

    response = rag_chain.invoke({"context": docs, "question": question})
    return {"messages": [response]}

The prompt does two protective things: it tells the model to say "I don't know" rather than invent an answer when the context is thin, and it caps the answer at three sentences so the output stays tight. Because generate flows straight to END, whatever it appends is the final answer.

Step 7 — The rewrite Node

If the docs failed the grade, we don't give up — we improve the question. The rewrite node asks the LLM to reason about the underlying intent and produce a sharper query, which is appended to the state and sent back to the agent for another attempt.

from langchain_core.messages import HumanMessage


def rewrite(state):
    """
    Transform the query to produce a better question.

    Args:
        state (messages): The current state

    Returns:
        dict: The updated state with re-phrased question
    """
    print("---TRANSFORM QUERY---")
    messages = state["messages"]
    question = messages[0].content

    msg = [HumanMessage(content=f""" \n
    Look at the input and try to reason about the underlying semantic intent / meaning. \n
    Here is the initial question:
    \n ------- \n
    {question}
    \n ------- \n
    Formulate an improved question: """)]

    model = ChatGroq(model="llama-3.3-70b-versatile")
    response = model.invoke(msg)
    return {"messages": [response]}

This is the same "question re-writing" idea you'd find in query-expansion (Chapter 8.1) — but here it's wired into a loop. The improved question becomes the newest message, the graph routes back to agent, and the agent gets a fresh shot at retrieving with a better query.

Step 8 — Assemble & Compile the Graph

Now we wire the four nodes together with the two conditional edges. Two prebuilt LangGraph pieces do the heavy lifting — ToolNode (runs whichever retriever the agent asked for) and tools_condition (the built-in "tool call → tools, else → END" router from Chapter 12).

from langgraph.graph import END, StateGraph, START
from langgraph.prebuilt import ToolNode, tools_condition

workflow = StateGraph(AgentState)

# Nodes
workflow.add_node("agent", agent)
retrieve = ToolNode([retriever_tool, retriever_tool_langchain])
workflow.add_node("retrieve", retrieve)
workflow.add_node("rewrite", rewrite)
workflow.add_node("generate", generate)

# Start at the agent
workflow.add_edge(START, "agent")

# Decide whether to retrieve (tool call) or finish
workflow.add_conditional_edges(
    "agent",
    tools_condition,
    {
        "tools": "retrieve",   # the agent asked for a retriever
        END: END,              # the agent answered directly
    },
)

# After retrieving, grade the docs: generate or rewrite
workflow.add_conditional_edges("retrieve", grade_documents)

workflow.add_edge("generate", END)
workflow.add_edge("rewrite", "agent")   # the loop back-edge

# Compile
graph = workflow.compile()

Walk the edges one at a time — this is the entire control flow of the system:

EdgeTypeMeaning
START → agentPlainEvery run begins at the agent.
agent → {retrieve, END}Conditional (tools_condition)Tool call → retrieve; plain answer → END.
retrieve → {generate, rewrite}Conditional (grade_documents)Our grader's return value picks the next node.
generate → ENDPlainThe answer is ready; stop.
rewrite → agentPlainThe loop back-edge — a failed retrieval gets another attempt.

Two edges carry all the intelligence. add_conditional_edges("retrieve", grade_documents) hands routing to our grader — its "generate" / "rewrite" return value is the next node, no mapping dict needed. And workflow.add_edge("rewrite", "agent") is the back-edge that closes the loop: a failed retrieval doesn't end the run, it sends an improved question back to the top. Remove that one line and you'd have a plain pipeline that gives up on the first bad fetch.

Nodes vs edges — the rule of thumb

If a function changes the state (adds a message, retrieves docs), it's a node. If it only looks at the state and picks a direction, it's a conditional edge. agent, retrieve, generate, and rewrite are nodes; tools_condition and grade_documents are edges. Same Python functions — different job in the graph.

Read the two loops: the right-side amber path (rewrite → agent) is the self-correction loop — a bad retrieval gets another attempt.
