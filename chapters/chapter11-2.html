Building a Chatbot with LangGraph

A graph whose state is a list of messages — that one change turns the toy graph from the parent chapter into a real conversational chatbot.

From Strings to Messages

So far our state held a plain string. A chatbot needs something richer: a running list of the conversation. In LangGraph, the state for a chat application is a list of messages, and each new turn appends to it. The state schema looks like this:

from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

# Reducers
from typing import Annotated
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]

That Annotated[list, add_messages] is the key idea of this whole chapter — we'll unpack it in a moment. First, the model.

Plugging in a Chat Model

The brain of the chatbot is a chat model. LangGraph doesn't care which provider you use — here we load environment keys and show two interchangeable options: OpenAI's gpt-4o and a Groq-hosted model.

import os
from dotenv import load_dotenv
load_dotenv()

os.environ["OPENAI_API_KEY"] = os.getenv("OPENAI_API_KEY") or ""
os.environ["GROQ_API_KEY"] = os.getenv("GROQ_API_KEY") or ""
from langchain_openai import ChatOpenAI
llm = ChatOpenAI(model="gpt-4o", temperature=0.5)
llm.invoke("help")
AIMessage(content='Of course! How can I assist you today?', ...)

Swapping in Groq is a one-line change — the rest of the graph stays the same:

from langchain_groq import ChatGroq
llm_groq = ChatGroq(model="openai/gpt-oss-120b", temperature=0.5, max_tokens=1000)
llm_groq.invoke("hello")
AIMessage(content='Hello! How can I help you today?', ...)
Why this matters

Because the model is just a value you pass into a node, LangGraph graphs are provider-agnostic. You can develop against a fast, cheap model and switch to a stronger one for production by changing a single line — the nodes, edges, and state never change.

The Reducer — Why add_messages Is Essential

Here's a subtle but critical problem. Each node returns a new value for the messages key. By default, that new value overwrites the old one. But in a conversation we don't want to replace the history every turn — we want to append to it.

The fix is a reducer. A reducer is a function that tells LangGraph how to combine the existing state value with a node's update, instead of blindly overwriting it. The pre-built reducer for messages is add_messages: it appends new messages to the existing list. You attach it by annotating the key:

from typing import Annotated
from langgraph.graph.message import add_messages
from langchain_core.messages import AnyMessage

class State(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]

You can see the reducer in isolation. Given a starting list and a new message, add_messages returns the combined list rather than the new message alone:

from langchain_core.messages import HumanMessage, AIMessage

initial_state = [AIMessage(content="Please help me with this question:", role="LLMModel")]
initial_state.append(HumanMessage(content="Attention is all you need", role="Ankit"))

ai_message = AIMessage(content="Attention is all you need", role="LLMModel")

# add_messages appends instead of overwriting
add_messages(initial_state, ai_message)
[AIMessage(content='Please help me with this question:', ...),
 HumanMessage(content='Attention is all you need', ...),
 AIMessage(content='Attention is all you need', ...)]

All three messages survive — and each is automatically assigned a unique id. That's exactly the behaviour a chatbot needs.

The Chatbot Node

With messages-as-state and the reducer in place, the chatbot node itself is almost trivial: take the running list of messages, send it to the model, and return the model's reply. The reducer takes care of appending it.

def superbot(state: State):
    return {"messages": [llm_groq.invoke(state["messages"])]}

Wire it into a graph — a single node between START and END:

graph = StateGraph(State)

# Node
graph.add_node("superbot", superbot)
# Edges
graph.add_edge(START, "superbot")
graph.add_edge("superbot", END)

graph_builder = graph.compile()

# Display
from IPython.display import Image, display
display(Image(graph_builder.get_graph().draw_mermaid_png()))

Invoking the Chatbot

Call .invoke() with an opening message. LangGraph wraps your string into a HumanMessage, runs the node, and appends the AI's reply:

graph_builder.invoke({"messages": "Hi, My name Ankit and I like Cricket"})
{'messages': [
  HumanMessage(content='Hi, My name Ankit and I like Cricket', ...),
  AIMessage(content="Hello Ankit! 👋 Great to meet you. Cricket is such an exciting
            sport—do you have a favorite team or player? ...", ...)
]}

Notice the returned state holds both messages — the human turn and the AI turn — because the reducer appended rather than overwrote.

Streaming the Response

Calling .invoke() waits for the whole run to finish before returning. For a chatbot you usually want to stream — show output as it's produced. LangGraph's .stream() yields an event per node as the graph executes:

for event in graph_builder.stream({"messages": "Hello my"}):
    print(event)
{'superbot': {'messages': [AIMessage(content='Hello! How can I assist you today?', ...)]}}

Each event is keyed by the node name (superbot) and carries that node's update to the state. For a one-node graph there's a single event; for a larger graph you'd see one event per node as control flows through it — which makes .stream() invaluable for both live UIs and debugging.

Remember These Four

1. A chatbot's state is a list of messages (HumanMessage, AIMessage, SystemMessage, ToolMessage). 2. The add_messages reducer makes node updates append to that list instead of overwriting it — without it, every turn would erase the history. 3. The chat model is just a value passed into a node, so graphs are provider-agnostic (OpenAI, Groq, …). 4. Use .invoke() for a single result, .stream() to emit one event per node as the graph runs.
