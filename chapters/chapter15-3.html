Query Planning & Decomposition

Split a complex query into explicit sub-questions, retrieve for each one, then merge the context into a single consolidated answer.

Overview

Some questions aren't hard because they're deep — they're hard because they span multiple topics in one sentence. “Explain how agent loops work and what are the challenges in diffusion video generation?” touches two completely different subjects. A single retrieval against the raw question might pull chunks about agents but miss the diffusion video content entirely (or vice versa).

Query Planning and Decomposition handles this by splitting the question into 2–3 explicit sub-questions first, retrieving for each one separately, pooling all the documents, and then answering the original question once from the combined context. It's like reverse-engineering a compound question into parts you can search for individually.

CoT vs. Query Planning — what's the difference?

Both techniques decompose a question, which is why they're easy to confuse. The difference is intent: CoT produces a reasoning scratchpad; query planning produces explicit search queries.

AspectChain-of-Thought (15.1)Query Planning (this section)
Purpose Let the LLM reason step-by-step before searching Break a complex query into structured sub-queries
Output of planner Reasoning steps (“understand baseline metrics”) Sub-questions (“What are agent loops?”)
Best for Deep, single-topic questions that need structured thinking Multi-topic questions spanning different domains in one store
Flow Think → Retrieve → Answer Plan all → Retrieve all → Answer once

The diagram shows the fan-out clearly: one complex question becomes three sub-questions, each gets its own retrieval pass, and all results merge before the responder writes a single answer. In code, the fan-out happens inside one node (retrieve_for_each) rather than as three separate graph nodes — simpler to build, same effect.

Step 1 — Load from the Web & Embed

Instead of a local text file, we scrape two blog posts with WebBaseLoader. Both posts land in the same FAISS index, which is exactly the scenario decomposition is designed for: one vector store, two unrelated topics, one compound question.

Imports and LLM

import os
from typing import List
from pydantic import BaseModel
from langchain_openai import OpenAIEmbeddings
from langchain_classic.schema import Document
from langchain_classic.text_splitter import RecursiveCharacterTextSplitter
from langchain_classic.vectorstores import FAISS
from langchain_community.document_loaders import TextLoader, WebBaseLoader
from langchain.chat_models import init_chat_model
from langgraph.graph import StateGraph, END

llm = init_chat_model("openai:gpt-4o")

Scrape, chunk, and index

Loop over the URLs and extend a single docs list. After splitting and embedding, chunks from both posts live side by side in the same vector store. The retriever has no idea which post a chunk came from — it just returns the nearest vectors to whatever query you pass. 

urls = [
    "https://lilianweng.github.io/posts/2023-06-23-agent/",
    "https://lilianweng.github.io/posts/2024-04-12-diffusion-video/"
]
docs = []
for url in urls:
    docs.extend(WebBaseLoader(url).load())

splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)
embedding = OpenAIEmbeddings()
vectorstore = FAISS.from_documents(chunks, embedding)
retriever = vectorstore.as_retriever()

You may see a USER_AGENT environment variable not set warning when scraping. That's normal and doesn't block the pipeline.

Step 2 — State & Nodes

The state

Same four fields as CoT, but the planner output is called sub_questions instead of sub_steps — a small naming choice that signals these are search queries, not reasoning steps. 

class RAGState(BaseModel):
    question: str
    sub_questions: List[str] = []
    retrieved_docs: List[Document] = []
    answer: str = ""

Node 1 — plan_query

Ask the LLM to break the compound question into 2–3 standalone sub-questions. Each sub-question should be specific enough to retrieve against on its own. The prompt ends with Sub-questions: to nudge the model into list format. 

def plan_query(state: RAGState) -> RAGState:
    prompt = f"""
Break the following complex question into 2-3 sub-questions:

Question: {state.question}

Sub-questions:
"""
    result = llm.invoke(prompt)
    sub_questions = [line.strip("- ").strip() for line in result.content.strip().split("\n") if line.strip()]
    return RAGState(question=state.question, sub_questions=sub_questions)

Node 2 — retrieve_for_each

For each sub-question, call the retriever and collect all returned documents into one list. Because both blog posts share the same index, a sub-question about “agent loops” pulls from the agents post, and one about “diffusion video challenges” pulls from the video post — automatically, with no routing logic. 

def retrieve_for_each(state: RAGState) -> RAGState:
    all_docs = []
    for sub in state.sub_questions:
        docs = retriever.invoke(sub)
        all_docs.extend(docs)
    return RAGState(question=state.question, sub_questions=state.sub_questions, retrieved_docs=all_docs)

Node 3 — generate_final_answer

Pool all documents, pass them to the LLM along with the original compound question (not the sub-questions). The model's job is to weave both topics into one coherent answer. 

def generate_final_answer(state: RAGState) -> RAGState:
    context = "\n\n".join([doc.page_content for doc in state.retrieved_docs])
    prompt = f"""
Use the context below to answer the question.

Context:
{context}

Question: {state.question}
"""
    answer = llm.invoke(prompt).content
    return RAGState(
        question=state.question,
        sub_questions=state.sub_questions,
        retrieved_docs=state.retrieved_docs,
        answer=answer
    )

Step 3 — Build & Run

Same straight-chain wiring as 15.1: planner → retriever → responder → END. Then invoke with a deliberately multi-topic question to see decomposition in action. 

builder = StateGraph(RAGState)
builder.add_node("planner", plan_query)
builder.add_node("retriever", retrieve_for_each)
builder.add_node("responder", generate_final_answer)

builder.set_entry_point("planner")
builder.add_edge("planner", "retriever")
builder.add_edge("retriever", "responder")
builder.add_edge("responder", END)

graph = builder.compile()

Run the graph and print the sub-questions before the final answer. Checking the sub-questions first tells you whether the planner understood the compound nature of the query. 

user_query = "Explain how agent loops work and what are the challenges in diffusion video generation?"
initial_state = RAGState(question=user_query)
final_state = graph.invoke(initial_state)

print("\n🔍 Sub-questions:")
for q in final_state["sub_questions"]:
    print("-", q)
print("\n✅ Final Answer:\n", final_state["answer"])
Output — Two Topics, One Answer

Sub-questions:

1. What are agent loops, and how do they function?
2. What specific challenges arise in generating videos with diffusion-based methods?
3. How do agent loops interact with the challenges in diffusion video generation?

Final Answer: Agent loops are think–act–observe cycles that let an LLM-based agent plan, use memory, and call tools. Diffusion video generation is harder than image generation because it demands temporal consistency across frames and large amounts of high-quality video–text data. Because the sub-questions retrieved from the two blog posts independently, the final answer covers both topics accurately — something a single retrieval against the raw question would have struggled to do.

Compare this to what a single retrieval would return: the embedding of the full compound question sits somewhere between “agents” and “diffusion video” in vector space, often missing chunks from one topic entirely. Decomposition forces coverage of both sides.

Key Takeaways

Remember These Three

1. Decomposition shines when one query spans multiple topics or knowledge domains in the same vector store.

2. Plan all sub-questions first, retrieve all, answer once — different from interleaved CoT reasoning.

3. The responder always answers the original question, not the sub-questions individually.
